# OPENTHREAD-002: Strengthen all OPEN THREAD field guidance and correct the answer_if_known prompt-facing classification

**Status**: COMPLETED
**Priority**: MEDIUM
**Effort**: Medium
**Engine Changes**: Yes — adds/updates `specificGuidance` entries in `@loom/core` field guidance for OPEN THREAD; sets `answer_if_known` to `promptFacing: "never"`. No schema, validation, compiler, or renderer changes.
**Deps**: None (independent of OPENTHREAD-001; the `answer_if_known` guidance is best experienced after its control is fixed, but neither blocks the other)

## Problem

Every OPEN THREAD field except `possible_pressure_now` falls through to the generated default guidance in `recordEntry` (`packages/core/src/records/field-guidance-records.ts:196-217`): e.g. `summary` shows only "summary for the OPEN THREAD record." with no examples, no contrast with siblings, and no per-enum-value help. This produces two concrete confusions reported while authoring a production OPEN THREAD:

1. **`summary` is undifferentiated.** An author cannot tell whether `summary` should recap everything that has happened, describe what the thread is about, or hint at consequences. The docs + compiler settle this: `summary` is concise present-tense prose describing **what the unresolved thread is**; it compiles into `{active_open_threads}` right after the title (`packages/core/src/compiler/sections/pressure.ts:99-107`). It is **not** a recap of past events (prose-derived summaries are barred from prompt context — `docs/ACTIVE-DOCS.md:125`), **not** a consequence hint (that is `possible_pressure_now`), and **not** the answer (that is `answer_if_known`). The three prose fields partition cleanly and the guidance must say so.

2. **`answer_if_known` mislabels its prompt-facing status.** Its guidance currently reports `promptFacing: "conditional"` (the `recordEntry` default, because the field is not in `operationalFields`), so the help popup tells authors it is "Sent when the owning surface is active…" (`packages/web/src/field-help/FieldHelp.tsx:122-137`). This is false: the compiler never renders `answer_if_known` — `pressure.ts:99-107` emits only `title`, `summary`, `urgency`, and `possible_pressure_now`, and a repo-wide grep finds no compiler resolver that references the field. Per §18 (OPEN THREAD records "are not commands to close anything… color the current local unit without forcing resolution", `docs/FOUNDATIONS.md:607`), feeding a known answer to the prose writer would pressure premature closure/reveal. The field is **authoring metadata**: it must be classified `promptFacing: "never"`, making its absence from the prompt correct-by-design and the popup honest.

Beyond these two, `type`, `status`, `audience_visibility`, `urgency`, and `current_relevance` have no per-enum-value guidance (the popup "Values" section renders empty — `FieldHelp.tsx:162-184`), so authors cannot distinguish e.g. `tension` from `mystery`, do not know only `status: active` compiles (`isActiveStatus`, `pressure.ts:228-230`), and do not know `audience_visibility`/`current_relevance` are operational/non-prompt-facing.

## What to Change

### 1. `OPEN THREAD.summary` — define what the field is and contrast its siblings

Add a `specificGuidance` override: `short` = what the unresolved thread is, in one or two sentences of present-tense prose; advice that it describes the open question/promise/risk itself, not a recap of past events and not its consequences; `examples` (a concrete one-line thread) and `antiExamples` (a "everything that has happened" recap; a consequence sentence that belongs in `possible_pressure_now`); `relatedFields: ["OPEN THREAD.possible_pressure_now", "OPEN THREAD.answer_if_known"]`.

### 2. `OPEN THREAD.answer_if_known` — reclassify as authoring metadata and explain it

Add a `specificGuidance` override that sets `promptFacing: "never"`, `promptDestinations: []`, a `validationRole` (mirror the proven "it is not sent to the prose prompt" phrasing so the doctrine test passes), a `short` explaining it privately records the answer if already known (or `none` while still open), `examples` (a known-answer line; `none — the answer is not yet decided`), and an advice line that it cannot pressure the thread to resolve because it never reaches the writer. Do **not** add the field to `operationalFields`.

### 3. Remaining OPEN THREAD fields — sharpen `short` text and add `enumValues` guidance

- `type`: one-line distinctions for `question | promise | unresolved_setup | tension | mystery | risk`.
- `status`: per-value guidance for `active | answered | resolved | escalated | abandoned | superseded`, noting only `active` threads compile into the prompt.
- `title`: `short` noting it is the display label and the first compiled line of the thread.
- `audience_visibility`: per-value guidance for `hidden | implied | explicit | ambiguous`; note it is operational (already `never` via `operationalFields`).
- `urgency`: per-value guidance for `low | medium | high | critical`; note it compiles as `urgency: <value>`.
- `current_relevance`: per-value guidance for `none | low | medium | high | critical`; note it is operational and informs selection salience, not prompt text.

### 4. (Optional, minor) doc alignment

In `docs/story-record-schema.md` §8.7, add a one-line note that `answer_if_known` is authoring-only and not sent to the prose prompt, keeping the schema doc consistent with the corrected guidance. The guidance system remains the primary authority surface.
