### F001 — Imported optional list could not return to zero items

**Observed fact.** In Mara's imported Cast draft, the author removed every proposed sample utterance because none was safe to preserve. One empty list item remained visible. Saving highlighted two invalid fields from that single residual-item condition; pressing Remove again did nothing, and the same save failure reproduced. The author could proceed only by adding one bounded noncanonical sample and marking it never-copy.

**Interpretation and impact.** An optional field became effectively mandatory after import. That forced the author to preserve content they had explicitly rejected, weakened trust in import review, and added two failed saves plus a workaround to an already correction-heavy path.

**Expected versus actual.** The visible editor and schema permit the optional sample list to be absent or empty. Removing its final imported item should therefore leave a lawful zero-item state. Instead, the UI retained a blank item that failed validation and could not be removed.

**Visible reproduction.** Create a Cast draft containing sample utterances, import it, remove each sample until none should remain, save, then try Remove and save again on the residual blank item. The validation failure repeats. Existing editor tests exercise removal from two samples to one, but not removal of the sole imported item to zero, which narrows rather than proves the implementation cause.

**Evidence and workaround.** Privacy-safe screenshots retain the failed save and residual empty item. The workaround required inventing one noncanonical sample, setting the available never-copy protection, and saving again. Likely layer: UI/editor state, source-confirmed only to the extent that zero is lawful and the current test seam lacks the final-item case.

**Desired author-visible outcome.** Removing the last optional item should visibly produce an empty, valid list and save on the next attempt. Acceptance guidance should include an imported-many-to-zero editor regression without weakening validation. This was reproduced on one Cast record in one Chrome run; other optional list controls were not exhaustively tested.

### F002 — Assistance contract made optional completion disproportionately expensive

**Observed fact.** Each cold response populated 79 field instances across all 51 Cast record properties. Mara required 45 truth/validation corrections after 8 uncertainties and 33 invented paths; Elias required 59 corrections after 11 uncertainties and 50 invented paths. The template instructed the executor to fill every field and propose dossier-fitting material when the premise was silent.

**Interpretation and impact.** The assistance path front-loaded speculative dossier work even when the immediate narrative role was offstage compression. The author could not treat the imported result as a small starting core; every non-empty proposal became a truth decision. Elias initially contributed only one labeled offstage leaf, so most immediate-run value was deferred until promotion.

**Expected versus actual.** A lawful manual Cast can be created with its ENTITY link and 25 required leaf fields; three required list properties may be empty and optional extended properties may remain absent. Assistance could reasonably preserve that truth boundary while offering optional enrichment. Instead, its output contract required non-empty proposals for all 51 properties. This is a structural source-and-visible comparison, not a separately timed manual control.

**Visible reproduction.** Start Cast creation from the clipboard assistance route with a premise that intentionally leaves many dossier details unknown, copy the static prompt, import a conforming response, and review the populated editor. The import summaries and subsequent correction ledger expose the breadth of invented proposals.

**Evidence and workaround.** Both privacy-safe import summaries show 79 filled instances and zero skipped. The only workaround was field-by-field curation: 129 successful writes/selections for Mara and 143 for Elias, with 45 and 59 substantive corrections. Likely layer: prompt contract, with the editor/schema contrast source-confirmed.

**Desired author-visible outcome.** The assisted route should make the required truth core complete while allowing genuinely unknown or optional dossier detail to remain visibly undecided, without weakening required-field validation. A follow-up should measure whether a bounded output contract reduces correction decisions while preserving later promotion usefulness. The evidence covers two dossiers in one story and does not estimate general model reliability or human completion time.

### F003 — Cast identity became ambiguous in compiled prompt

**Observed fact.** The literal string `Elias Venn` was absent from the initial offstage packet and from Elias's promoted active dossier. Offstage rendering included an unlabeled one-line summary plus one labeled voice leaf; active rendering used that one-line summary as the dossier heading. Because the authored `identity.one_line` intentionally described the person without repeating his name, neither form preserved the record's literal identity.

**Interpretation and impact.** The prompt contained the right dossier content but did not anchor it unambiguously to the name used in the brief and story. That can make attribution harder for an executor, particularly with multiple compressed offstage figures or descriptive one-line identities.

**Expected versus actual.** A Cast record selected as Elias should remain visibly identifiable as Elias after compilation, independently of whether a descriptive identity field happens to repeat the name. Actual output relied on `identity.one_line` as the display label and omitted the record label from compressed relevance.

**Visible reproduction.** Create a Cast record whose one-line identity does not repeat the entity name, place it offstage, refresh and search the visible preview; then promote the same record, refresh again, and inspect its active heading. The literal name is absent in both states. Post-journey source inspection confirmed that active headings use the derived display label while compressed offstage output emits only the one-line identity and voice.

**Evidence and workaround.** Exact packet fingerprints and field census are recorded, but full packets are intentionally not retained. The available workaround is to duplicate the literal name into `identity.one_line`, which consumes author effort and distorts a field intended as a concise description. Likely layer: prompt compiler/display-label contract.

**Desired author-visible outcome.** Both compressed and active Cast prompt forms should retain an explicit, stable identity label while preserving the descriptive one-line field separately. Acceptance guidance should cover a one-line identity that omits the person's name in both bands. The cold prose inferred Elias correctly in this one run, so the observed ambiguity is a prompt-integrity defect, not evidence of a downstream prose failure rate.
