# Deliverable Execution

Load this reference only when the requested outcome includes a document, issue, code change, implementation, publication, or other persistent mutation. The read-only guardrail in `SKILL.md` always wins.

## Confirm the deliverable

Preparatory wording may imply a written determination only when it clearly asks for that artifact and no read-only instruction applies. Otherwise close on the conversational recap.

Before a downstream deliverable, show the recap and get confirmation when a user-ratified choice shaped the scope. A later `proceed`, `implement this`, or equivalent can satisfy that checkpoint. If the work was explicitly delegated up front and only explored facts shaped it, proceed within the stated scope without inventing another decision.

For an audit-plus-edit hybrid, a terminal confirmation that already enumerates the exact edit may satisfy the checkpoint; ordinary acceptance of a mid-interview branch does not.

## Compose supporting skills

Before mutation, inventory any supporting or downstream skill's mandatory writes, exact final block, key order, or final-position constraint.

- Hold supporting writes while a read-only instruction or unresolved mutation decision applies.
- Give every active closeout constraint one output home; do not paraphrase an exact block.
- If same-priority constraints cannot coexist, stop before mutation and surface the conflict.
- A supporting skill that owes no update produces a result, not a fabricated decision.

## Execute

Perform only the confirmed deliverable. Mechanically required consistency edits are in scope when they trace directly to that deliverable; enumerate them at closeout.

For repository writes:

1. capture branch, HEAD, and full short status immediately before the first write;
2. preserve unrelated work;
3. run the repository's required checks in proportion to the change; and
4. refresh the same state before the final response.

For external writes, use [operational-execution.md](operational-execution.md) and read back every affected resource. Report checks as passed, failed, or unavailable; never claim unrun gates passed.

In an explicitly autonomous or unattended run, provisional choices permit only the deliverable already requested. They do not authorize extra artifacts or mutations.

## Closeout

Rebuild the applicable recap from final evidence after any mutation; an earlier approval message is not closeout. State what changed, required consistency edits, verification, remaining uncertainty, and exact readback.
