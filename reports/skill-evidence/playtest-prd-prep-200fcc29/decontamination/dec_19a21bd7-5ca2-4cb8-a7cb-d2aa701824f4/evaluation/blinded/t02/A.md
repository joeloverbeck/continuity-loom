### Ticket Packet: Ideate Assigned-Slot Completeness Guard

Sources: F013
Type and readiness: bug plus ready-for-agent - repeated evidence and the current parser contradict the already-settled output contract; both labels are verified on current closed issues
Problem: An author can request and inspect five grounded Ideate slots, receive only four valid blocks, and see the response accepted as a complete slate because parsing validates blocks individually but never compares them with the compiled assignment.
Product rule: A successful Ideate response contains exactly one block for every slot assigned by the inspected and server-rebuilt request, with matching slot number and operator. A missing, duplicate, unassigned, or operator-mismatched block is malformed and remains quarantined raw scratch; there is no partial salvage or automatic retry.
Affected surfaces: existing Ideate response parser, server analyze route, quarantined result presentation, and their focused tests; no project schema, compiler source profile, or provider transport change
Scope: carry the exact deterministic assigned-slot contract into response validation; detect incomplete or mismatched sets; reuse the existing malformed-result recovery; keep the inspected prompt, fingerprint, citations, and single explicit send boundary unchanged
Acceptance:

- A response with exactly one matching block for every compiled slot is accepted in both ideas and questions mode.
- A four-of-five response, duplicate slot, unexpected slot, or operator mismatch is returned through the existing malformed quarantine rather than displayed as a complete slate.
- Validation uses the server-rebuilt current assignment associated with the matching fingerprint and never trusts client-supplied slot metadata.
- Failure makes no second provider call, writes no project data, and leaves retry as an explicit author action.
- Existing deterministic slate shrinkage remains valid because validation compares against assigned slots, not the requested count alone.

Preserved strengths: F021
Testing seam: existing /api/ideate route as the highest behavior seam, backed by parser fixtures and Ideate result-view component coverage
Out of scope: changing slot assignment, count range, dormant-slot behavior, Author focus, prompt bytes, provider settings, hidden retries, partial-slate salvage, keeper persistence, records, or Generation Brief data
Browser-visible guidance checklist mapping:

- \`entry point and availability\`: the existing Ideate send/result path remains the entry point; exact-slot validation runs only after the explicit provider response.
- \`user-visible states, actions, and outcomes\`: the issue body must distinguish an accepted exact slate from an incomplete response shown in the existing malformed scratch state.
- \`validation, warning, error, and recovery behavior\`: missing, duplicate, unexpected, and operator-mismatched slots are quarantined with the existing explicit retry recovery and no silent salvage.
- \`prompt preview contents and freshness\`: validation uses the assignment rebuilt from the same current request whose fingerprint matched; prompt bytes and inspection remain unchanged.
- \`user-initiated external LLM boundary\`: one explicit send produces at most one provider call; validation never retries automatically.
- \`canon and prose boundary visibility\`: accepted and malformed Ideate output both remain non-canonical scratch with no insert, apply, or use-as-prose path.
- \`persistence, migration, export, and provenance\`: N/A - response-set validation is ephemeral and adds no stored or exported data.
