# Triage

Move tracker issues through categorisation, verification, maintainer decision, and a durable handoff. If the repository's issue-tracker authority treats external pull requests as a request surface, include them: a PR is an issue with attached code. Resolve a bare number as that authority directs.

## State and safety contract

Category roles are `bug` and `enhancement`. State roles are `needs-triage`, `needs-info`, `ready-for-agent`, `ready-for-human`, and `wontfix`. A triaged item has exactly one category and one state. Stop and ask the maintainer before resolving conflicting roles.

The usual transitions are: unlabeled to `needs-triage`; `needs-triage` to any terminal/next state; and `needs-info` back to `needs-triage` after the reporter replies. A maintainer may override this sequence.

Before recommending or applying labels, read the repository's issue-tracker and label-mapping authorities (`docs/agents/issue-tracker.md` and `docs/agents/triage-labels.md` here). They own actual label strings, external-PR rules, and per-label preconditions. If the project has no mapping/configuration, stop and route to `/setup-matt-pocock-skills` rather than inventing it.

Every tracker issue or comment created by triage starts with:

```markdown
> *This was generated by AI during triage.*
```

Recommend before mutating and wait for maintainer direction. A direct imperative to make a transition, or a verified standing conditional transition recorded in the item, is direction; follow the override rules below. After every mutation, read the exact item back and verify the intended result before claiming success.

Load [AGENT-BRIEF.md](AGENT-BRIEF.md) only when drafting a `ready-for-agent` or `ready-for-human` handoff. Load [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) only when checking a plausible prior rejection or recording a rejected enhancement.

## Queue views

For "show me what needs attention," query and present oldest first:

1. Unlabeled items.
2. `needs-triage` items.
3. `needs-info` items with reporter activity after the last triage note.

Include in-scope external PRs, tagged `[PR]`; tag issues `[issue]`. The external-author filter applies to discovery, not to an explicitly named PR. If the tracker cannot expose the configured author metadata, inspect returned PRs individually or mark PR discovery as limited—never silently omit them. On items already reported, flag missing or multiple category/state roles, recommend exact fixes, and wait before applying them. Show counts and one-line summaries, then let the maintainer choose.

For "what next," group open items by `ready-for-agent`, `ready-for-human`, `needs-triage`, `needs-info`, then unlabeled. Account for explicit dependencies, blocker comments, and recent maintainer decisions. Prefer unblocked agent work; prefer human decisions when they unblock several items; keep grooming and label hygiene distinct from the recommended work item.

When a blocker closes or an issue family matters, inspect relationships in both directions. Start from compact open-item metadata and exact-read plausible matches. Confirm literal gate identifiers in context; a search hit is not proof. If comment search requires a broader listing, bound and record the pages before filtering locally, and report limits. For a child, report the parent's remaining open implementation children. For a parent, distinguish implementation children from predecessors/repair gates, evidence/replay work, and unrelated references; only implementation children govern closeability, and say when none were published.

## Triage one issue or PR

### 1. Gather current context

Read the full body, comments, labels, author, dates, and—for a PR—the diff. Parse earlier triage notes so resolved questions stay resolved. In a worktree, start with `git status --short`, separate relevant work from unrelated dirt, and preserve unrelated paths. Verify named prerequisites against the implementation starting ref; local presence is not publication or reachability.

Use the domain glossary and ADRs for the affected area. Check:

- **Redundancy:** search by domain concept for existing behavior. If a code request is already implemented, propose `wontfix` and point to the implementation. For an evidence, replay, closeout, or docs artifact, instead verify whether it was produced against the current tree.
- **Prior rejection:** when conceptually plausible, consult `OUT-OF-SCOPE.md` and the repository's `.out-of-scope/` entries. Absence is not an error.
- **Related ownership:** use compact open-item discovery, then exact-read plausible owners. Search all states/bodies only when compact discovery is inconclusive; exact-read a named accepted contract directly.

### 2. Verify the claim and gates

- For a bug, reproduce the reporter's steps.
- For a PR, confirm the diff and run relevant checks.
- For an enhancement or PRD, verify completed dependencies by state reason, named implementation prerequisites at the starting ref, and the integration seam in the current tree.

Report verification as confirmed with evidence, failed, or insufficient detail. Classify named artifacts: implementation prerequisites must be durable at the starting ref; summarized provenance does not block readiness when the tracker body contains the implementation-critical conclusions and current repository authority supports them.

When current HEAD contradicts acceptance criteria from a contract closed as completed, treat it as a bug/conformance repair, not a duplicate enhancement or automatic reopen. Preserve the completed item as history, recommend a new repair owner by default, and wait before reopening or publishing. Dependent work remains `needs-triage` until the repair closes as completed.

Earlier proof is not current proof after later commits touch its execution surface or explicitly defer the proof to this item. Re-run it at HEAD. Use throwaway data for stateful verification and stop processes you start.

### 3. Recommend, then pause

Give the category, state, reasoning, codebase/authority summary, redundancy result, and verification result. Wait for direction unless a standing conditional override is now proven.

If the request still contains product or design choices, route to `/grilling` and `/domain-modeling`; do not manufacture an implementation brief around unresolved decisions.

### 4. Apply the approved outcome

Re-read label preconditions immediately before mutation and record compliance in the outcome comment.

- `ready-for-agent`: post a brief from `AGENT-BRIEF.md`. If readiness relies on a commit, verify remote reachability and name the required starting branch/ref when it is not generally reachable.
- `ready-for-human`: use the same brief shape and state why judgment, access, design, or manual verification prevents delegation.
- `needs-info`: post what is established and specific actionable questions for the reporter. Never ask only for "more info."
- Resolved decision-only issue: post the chosen decision and follow-up, then close as resolved. Do not call an accepted decision `wontfix`.
- `wontfix`: for already-built work, point to the implementation and do not write `.out-of-scope/`; for a rejected bug, explain and close; for a rejected enhancement, follow `OUT-OF-SCOPE.md`, link the record, and close.
- `needs-triage`: apply the state; when a readiness gate blocks an otherwise specified item, record passed checks, the exact unmet gate/evidence, and the re-evaluation trigger. Use `needs-info` instead when the reporter must answer.

### 5. Verify the outcome

Read the exact issue or PR after each mutation. Confirm open/closed state and state reason, exactly one category and state role, and the exact outcome comment or brief. Identify a new comment by the mutation's returned URL/ID, fetch that exact record, and compare its body byte-for-byte with the staged source; presentation output that changes newlines is not proof. Repair a safe in-scope mismatch or report the transition incomplete.

## Overrides and resume

A direct request such as "move #42 to ready-for-agent" skips re-grilling. Briefly state the exact label/comment/close operations, check current item state and label preconditions, then act. A `ready-for-agent` transition still gets a brief: ask whether to write it when the maintainer is present and did not already approve one; write it when operating autonomously.

A conditional transition recorded in the issue body or a maintainer comment becomes a standing override when its gate is verified closed as completed and every named prerequisite holds. The body and comments carry equal authority. Do not seek another approval. For `ready-for-agent`, post one disclaimer-prefixed comment combining gate evidence, ref caveats, and the full brief; verify that body exactly before applying the state label.

When resuming, read prior triage notes, incorporate reporter answers, and present the updated state without repeating resolved questions.
