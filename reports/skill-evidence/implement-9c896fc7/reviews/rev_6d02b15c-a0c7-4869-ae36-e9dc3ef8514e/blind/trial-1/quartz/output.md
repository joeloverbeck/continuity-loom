# Ordered execution plan

1. **Open the implementation-evidence sinks before relying on any proof.** Create the durable verification-command ledger before the first command whose result will support a pass/fail or closeout claim. Create or update the durable TDD/implementation ledger before the first red. It records the TDD preflight, both TDD seams, their red/green evidence, acceptance atoms, proof surfaces, sequences, authority disposition, and the full fielded `TDD evidence gate passed: durable sink ...` line. Record the authorized expectation rewrite as a compact TDD row whose seam is exactly `existing contract-change expectation` and whose failure cell begins `existing contract-change expectation in <test file> because ...`; if that row is additional to the two ordinary seams, preserve it as an additional row rather than collapsing it. Browser proof, browser console state, proof-server preflight, backend currentness, and browser freshness are explicitly N/A with the reason that no browser/manual evidence applies.

2. **Create the pre-review acceptance artifacts, but not the final closeout scaffold.** From the saved exact issue JSON, create the one-issue deterministic acceptance manifest and its working acceptance-audit scaffold. Fill one row per acceptance or Principles/ADR check, with exact criterion text, literal `satisfied`/`blocked`/`not done` status, and `atoms:`, `proof surfaces:`, and `sequence:` in every satisfied row. Run the acceptance-exactness challenge and audit-only review-entry validation. This completed working audit is the durable pre-close review-entry artifact; it is not yet the publishable closeout body. Update it if implementation evidence changes. Reconcile artifact disposition and ownership/placement, expose the pre-stage and scoped-staging gates, and commit only implementation-owned files. Any tracked implementation report committed here remains SHA-independent: it may identify the fixed point and subject, but it must not claim its own final SHA, reviewed HEAD, or terminal review result.

3. **Run normal review against a fixed point and preserve both findings.** Anchor review at the original implementation fixed point so the eventual frame covers the implementation commit and the follow-up fix commit, not merely `HEAD~1` after the follow-up. Record the initial Standards and Spec results and give the behavior defect and evidence-only defect immutable `P1-<axis>-<ordinal>` IDs on the axes where review found them.

4. **Fix the behavior finding with the single RF row.** Before changing behavior, obtain a red that fails for that exact defect, then green it. Add `RF-1` to the keyed TDD review-fix map, mapped to the affected issue/seam, with durability, browser-freshness N/A, backend-currentness N/A, and same-sink identity-refresh dispositions. If the first red fails for the wrong reason, retain it as `partial red - wrong reason: ...` and obtain the intended red before patching.

5. **Fix the evidence-only finding without inventing a second RF row.** Repair the evidence artifact and record the finding as `repairClass: evidence-only`, with the canonical red-first-skip disposition because the repair did not change behavior. Keep it as the second review-finding ledger row, but do not turn it into another TDD review-fix row. Update the working audit and TDD durable sink only where the evidence repair actually changes them.

6. **Create the follow-up commit, which creates the final SHA.** Recheck scope and stage only implementation-owned behavior, tests, and evidence files for the review fixes. Create the follow-up commit; only now is the final SHA known. Keep the final review frame anchored at the original implementation fixed point through this final `HEAD`.

7. **Refresh all final-tree evidence after the follow-up commit.** Rerun every required focused and canonical verification gate on the final SHA and update the verification-command ledger with exact command, observed counts/result, run count, and represented SHA. Publish only final-SHA rows. Complete the browser/manual mini-gate as N/A: files touched since browser/manual smoke are N/A because no smoke was used, backend process currentness is `N/A because no browser/manual evidence was used`, console state is N/A, and smoke freshness is N/A for the same reason. Re-run final Standards and Spec review over the complete fixed-point-to-final-HEAD range and record the terminal normal-review result as findings fixed in the final SHA, with two finding-ledger rows, one RF row, no residual findings, final source inventories, verification reruns, and follow-up-commit handling.

8. **Finalize evidence identities and the structured evidence source before any closeout-body build.** Update the durable TDD/review evidence with all current, historical-red, and superseded identity categories, plus a superseded-token sweep over the exact evidence set. Historical red identities must retain the real red-only identities rather than being mislabeled as active or stale. Then create or refresh one structured evidence JSON containing the manifest-exact audit rows, every TDD seam row, the required existing-contract-change row, the single `RF-1` review-fix record, and both terminal review-finding records. The counts must reconcile to two review findings and one RF row. This JSON is complete before the first `build-closeout-body` use so the builder, rather than hand-copying, derives the compact TDD rows, RF row, review ledger, counts, severities, and repeated summaries.

9. **Freeze the closeout shape, then build the publishable scaffold for the first time.** Freeze the branches only after terminal review, the follow-up commit/final SHA, final verification, browser N/A freshness, remote-reachability decision, and identity refresh are known: one issue, normal review, immediate fix, TDD present, browser N/A, Principles conditional on the issue, no child-family fields, and local-only wording only if required and authorized. Now—and not before review—perform the first closeout-scaffold build in an uncommitted temporary body using the acceptance manifest and structured evidence JSON, normal-review immediate-fix mode, and the TDD parent-rollup compact format. Run the size plan with required headroom as part of this first build; stop and split before filling if it reports low headroom or exceeds the tracker limit. Publishable sink fields use a stable issue reference, never the temporary local path.

10. **Fill only the non-derived final fields and inspect the exact body.** Add the final SHA, reachability or full `Local-only SHA:` sentence, final-SHA verification ledger rows, terminal `Review:` line and review frame, TDD fielded closeout evidence, browser/console/freshness N/A fields, evidence-identity block and sweep, Principles/ADR disposition, acceptance audit, and closeout preflight. Do not hand-edit builder-derived compact TDD, RF, finding-count, severity, or review-summary sections; correct the structured JSON and rebuild if those are wrong. Run bounded size, placeholder, token, table/status, and visual inspections against the exact body.

11. **Validate the exact final body in layers.** Validate the compact TDD evidence against the acceptance manifest and final SHA. Validate normal code-review evidence with immediate-fix and TDD parent-rollup mode, the acceptance manifest, closing mode, and the final SHA; browser, parent-PRD, and child-family flags remain absent. Validate implement closeout with closing mode, expected final SHA, and the acceptance manifest, adding Principles or local-only flags only when applicable. Repair the source artifact and regenerate if a validator finds a mismatch; after any body, SHA, evidence, or commit change, re-run the relevant final review/freshness/identity refresh and validators.

12. **Immediately before the first tracker mutation, re-read the closeout gates and run the mutation-ready gate.** Recheck the exact body, final SHA/reachability, manifest coverage, literal statuses, TDD/review evidence, N/A browser fields, identity sweep, and stable sink wording. Run the implement closing validator one last time with expected final SHA, acceptance manifest, `--emit-preflight`, and `--mutation-ready`, plus only applicable conditional flags. Copy its emitted `Closeout preflight:` block, `Closeout gate passed: audit sink ...` line, `Post-comment verification next:` reminder, mutation-ready confirmation, and machine-derived residual summary verbatim into the conversation or durable audit sink; do not hand-transcribe them.

13. **Make the first mutation-ready tracker comment and verify it.** Post the exact inspected body to the single issue using the body-file path, capture the returned comment URL, and immediately exact-read that stored comment against the local body before any close command. This comment is the first tracker mutation and becomes the durable external sink for the final SHA and terminal review evidence. If exact readback fails or is ambiguous, preserve the URL and mutation output, do not replay the comment, and retry only the read-only verification. Stop here; issue closure is a later mutation and remains blocked until exact stored-body verification passes.
