# Ordered execution plan

No commands are performed by this plan. The closeout uses two evidence sinks: a SHA-independent implementation evidence sink that may be committed, and an external publishable closeout body that is not committed and can therefore name its own final SHA and terminal review result.

## Artifact lifecycle

| Artifact | Create | Update/finalize |
|---|---|---|
| Verification-command ledger | Before the first test or verification command whose result will support a pass/fail or closeout claim | After every evidence-bearing invocation; after the review-fix commit, rerun the required commands and retain publishable rows representing only the final SHA |
| SHA-independent implementation/TDD evidence sink | Before review and before staging the implementation commit | Before the implementation commit, complete the working acceptance audit and both TDD seam rows; before the review-fix commit, add the review-fix evidence that belongs in tracked evidence, but never add that commit's final SHA, reviewed-HEAD SHA, or terminal `Review:` result afterward |
| Acceptance manifest and working acceptance audit | The single acceptance manifest already exists; generate/fill its deterministic audit scaffold before review | Run the pre-review exactness challenge and audit-only review-entry validation before the implementation commit; after fixes, refresh any affected audit evidence and keep every closing row literally `satisfied` |
| Normal-review evidence record | Create when the initial Standards and Spec passes complete | Add the two immutable finding rows, the repair evidence, final Standards and Spec passes, the final fixed-point-to-HEAD frame, and terminal reviewer/cleanup status after the review-fix commit |
| Structured closeout evidence JSON | Create after the initial review has identified both findings | Update after the behavior red/green, the evidence-only repair, the review-fix commit, final review, final verification, and identity refresh. It is complete before the first closeout-scaffold build |
| Uncommitted publishable closeout body | **First build only after the review-fix commit creates the final SHA and final review/verification evidence is available** | Fill and validate it against the final SHA. Any later SHA, evidence, identity, or body change requires a refresh, reinspection, and revalidation before mutation |
| Tracker closeout comment | Create as the first tracker mutation only after mutation-ready validation | Capture its URL and exact-read the stored UTF-8 body before any issue-close command; do not close the issue within this plan |

## Execution order

1. **Establish the implementation evidence sinks.** Start the durable verification-command ledger before the first evidence-bearing test command. Record each exact invocation, observed result/counts, run count, and the working-tree identity it represents. Establish the SHA-independent implementation/TDD sink early enough that its TDD preflight and compact rows demonstrably precede their red commands.

2. **Record the two TDD seams before review.** Carry exactly the two actual seam rows into the implementation sink, including the known expectation-rewrite row. That row uses `Seam` = `existing contract-change expectation`, and its red/failure cell begins `existing contract-change expectation in <test file> because ...`, then names the issue-authorized contract change and failing command. Complete the TDD closeout preflight, pre-red evidence reference, authority disposition, acceptance atom and sequence maps, evidence identities (including historical red identities), and the full fielded `TDD evidence gate passed: durable sink ...` line. The contract-change row is not treated as an undocumented test edit; its acceptance cell identifies the exact issue criterion that authorized changing the expectation.

3. **Complete review-entry acceptance evidence.** Fill the manifest-derived working audit with one exact row per acceptance criterion and Principles/ADR check. Every row intended for closeout has the exact columns `Acceptance criterion or conformance check` and `Status`, literal status `satisfied`, and evidence containing independent `atoms:`, `proof surfaces:`, and `sequence:` fields. Perform the acceptance-exactness challenge, then apply audit-only review-entry validation against the supplied manifest. Unsupported rows remain `blocked` or `not done` and would stop this plan before review.

4. **Stage and commit the implementation evidence fixed point.** Make the implementation pre-stage gate visible, including the audit sink, no blocked/not-done rows, artifact disposition, ownership decisions, and unrelated dirt. Stage only implementation-owned files, inspect the staged list, make the implementation commit gate visible, and create the implementation commit. The committed evidence may identify the review fixed-point input and subject, but it does not claim its own final SHA or terminal review result.

5. **Run the initial normal code review against a fixed point.** Anchor review at the original implementation fixed point and the implementation commit. Preserve the initial Standards and Spec outputs, source inventories, reviewer identities, and terminal/cleanup evidence in the normal-review record. Assign both findings immutable IDs using their actual axes: one row for the behavior defect and one for the evidence-only defect. The initial finding count is exactly two.

6. **Repair the behavior finding with the single RF row.** Add `RF-1` for the behavior defect. Its first intended-red evidence must fail for that exact reviewed behavior; a wrong-reason failure is recorded as partial red and followed by the smallest correct intended red before changing behavior. Apply the repair, rerun green evidence, and map `RF-1` to the affected one of the two existing TDD seam rows with durable regression evidence. There is exactly one TDD review-fix-map row.

7. **Repair the evidence-only finding without inventing another RF row.** Correct the evidence sink/body/report defect and record the finding's repair class as `evidence-only`. Its finding-ledger TDD disposition states the canonical red-first skip reason because the fix changed evidence rather than behavior. It does not add a third TDD seam or a second RF row. Refresh any tracked SHA-independent implementation evidence now, before committing the fixes.

8. **Create the follow-up review-fix commit.** Recheck worktree scope, stage only implementation-owned behavior, test, and evidence changes, and commit both repairs together. This follow-up commit is the point at which the final SHA first exists. Keep the ultimate review frame anchored at the original fixed point through this final `HEAD`; do not narrow it to `HEAD~1`, which would review only the repair commit.

9. **Re-review the final tree and finalize normal-review evidence.** Run final Standards and Spec passes over the original fixed point through final `HEAD`, using final source inventories that include all pre-dispatch authorities and any authority introduced during repair. Record both findings as `fixed`, final per-axis outcomes, final reviewer identities/statuses, the reviewed-HEAD SHA, and `Review: code-review against <fixed point>; outcome findings fixed in SHA <final SHA>; verification rerun <commands>.` The review finding ledger has two rows; only the behavior row points to `RF-1`. Residual findings are `none`.

10. **Refresh final-tree evidence.** Rerun the required focused and canonical verification on the final SHA and update the verification ledger from actual output; publish only final-SHA rows. Record browser evidence, console state, backend currentness, and freshness as N/A with the authoritative reason that browser proof does not apply. Complete the post-review evidence-identity refresh with every current, historical-red, and superseded category, and run the superseded-token sweep. Determine remote reachability; if the SHA is local-only and policy permits local-only closeout, include the full `Local-only SHA:` explanatory sentence, otherwise push as required before tracker mutation.

11. **Finalize the structured evidence input before any closeout scaffold exists.** The JSON contains:

   - exact `auditRows` for every manifest check;
   - the two `tddRows`, including the validator-exact existing-contract-change row;
   - exactly one `tddReviewFixes` entry, `RF-1`, for the behavior defect;
   - exactly two `reviewFindings` entries, with immutable IDs, actual axes/severities, repair classes `behavior` and `evidence-only`, final status `fixed`, and only the behavior row linked to `RF-1`.

   This placement is mandatory: the existing-test row is present in the structured JSON **before the first `build-closeout-body` run**, so no compact TDD row or derived count is hand-copied later.

12. **Build the closeout scaffold for the first time.** Run the scaffold builder from the single acceptance manifest and the completed structured evidence, selecting normal review with `--immediate-fix`, `--tdd-parent-rollup`, and the size-plan/headroom gate required by the body mode. The builder, not manual transcription, derives the two compact TDD rows, one RF row, two finding rows, finding counts, issue/seam accounting, and repeated review summaries. If the size plan is not `ok`, split before filling; do not compress away required audit atoms or surfaces.

13. **Fill the uncommitted publishable body.** Add the final SHA and reachability, final verification ledger, full fielded TDD evidence, normal immediate-fix review frame and source inventories, two-row finding ledger, one-row RF map, Principles/ADR disposition, browser/console/backend/freshness N/A fields, evidence-identity inventory and sweep, exact manifest audit, and closeout preflight scratchpad. Publishable sink fields use a stable issue reference, never the local staging path. The body remains outside the implementation commit so its final-SHA and terminal-review claims are not self-invalidating.

14. **Inspect and validate the exact body.** Check size, bounded excerpts, unresolved placeholder-like tokens, exact audit columns/statuses, required labels, and the grouped criteria visually without truncated output. Run the applicable TDD closeout validator, the **normal** code-review validator with immediate-fix and TDD/manifest flags, and the implement closing validator against the expected final SHA. Do not use fallback review labels or fallback validation. A successful diagnostic closing validation is not yet mutation-ready.

15. **Perform the last pre-mutation refresh.** Immediately before mutation, re-read the closeout-gate rules and ensure that no final SHA, review evidence, identity, or body field changed after validation. If anything changed, refresh the external body, review frame if needed, verification/freshness/identity evidence, token sweep, inspections, and all applicable validators.

16. **Make the body mutation-ready.** As the last validation before tracker mutation, run the implement closing validator on the exact inspected body with the expected final SHA, acceptance manifest, and `--emit-preflight --mutation-ready` plus only applicable condition flags. Copy its output verbatim into the conversation/durable audit sink, including `Mutation-ready closeout validation passed:`, the full `Closeout preflight:` block, exact `Closeout gate passed: audit sink ...` line, and `Post-comment verification next:` reminder. Also make the exact `Closeout body check passed: ...` line visible after the final untruncated inspection. No tracker command may precede this point.

17. **Post the first mutation-ready tracker comment and exact-read it.** Post the validated body to the one issue using the body-file form as the first tracker mutation, capture the returned comment URL, and immediately verify that the stored comment is an exact UTF-8 match for the inspected body. A mismatch or unavailable readback keeps closeout blocked and must not trigger a duplicate post. Stop here: the issue-close command is still prohibited until this exact stored-body verification passes and the later close-command gate is rechecked.
