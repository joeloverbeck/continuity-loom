# T07 Operator Packet — Clipboard Cast drafting and promotion

## Simulation disposition

This packet specifies a future, real-browser `new_story` playtest. This paired-trial execution
does not launch the app, create or alter a story project, open product source or documentation, or
make any provider request. The run described below is not complete until a future operator either
visibly accepts exactly one segment and finishes post-acceptance continuity work, or exhausts the
blocker policy and publishes a truthful blocked report.

## Run charter

- Working title: **The Signal at Long Barrow**.
- Mode: `new_story`; there is no prior report or inherited project.
- Author stance: instructed but source-and-doc blind. Learn the product only through the visible
  1440x900 browser UI, field help, prompt inspectors, and visible diagnostics.
- Story location: a storm-isolated observatory. Do not leave it during this segment.
- POV and physical state: Mara Chen is the local focal character and is physically alone.
- Signal truth: the signal uses a private calibration sequence associated with Mara's estranged
  former mentor, Elias Venn.
- Locked unknowns: sender, sender location, motive, Elias's physical presence, and Elias's safety
  all remain unknown. Association with the sequence is not proof that Elias sent it.
- Local segment: Mara detects and verifies enough of the calibration pattern to recognize its
  association and experiences the immediate pressure that recognition creates.
- Mandatory stop: end before Mara decides whether to reply or how she would reply.
- Cast task: draft both Mara and Elias records from their exact visible static Cast prompts, use
  the visible local clipboard/import path, review and correct the imported drafts, and explicitly
  save the corrected records. No imported text is authority before that review and save.
- Participation task: begin with Mara in the active/onstage POV function and Elias as compressed
  offstage relevance. Later promote Elias to the visible active participation band solely for a
  prompt-participation comparison. The promotion must not edit either record, establish canon, or
  add/remove working-set membership.
- Generation Brief task: deliberately populate the prompt-facing setting, knowledge, continuity,
  and stopping-point fields, plus only the universal minimum needed to make the moment legible.
- Acceptance limit: at most one segment, and only after `none` or `light` local intervention. A
  candidate that needs `substantial` repair or a `rewrite` is not acceptable for T07.
- Non-goals: do not solve the signal, establish that Elias is alive or present, decide Mara's
  response, manufacture extra cast or records, or use accepted prose as automatic continuity.

The quantitative journey ledger remains `inactive`: this task asks for a behavioral boundary, not
action, field, setup-cost, or promotion-cost counts. All four bounded method pilots are
`awaiting-disposition`, so this run performs no Cold First-View Witness, Paired-Draw Check,
Independent Claim Challenge, or method-register natural-run row. Their report counters remain
zero and Coverage Limitations records the inactive states.

## Visible author journey

### 1. First screen and project creation

Because the first-view pilot is not active, the main operator directly inspects the unopened app
screen before typing. Record what the screen appears to be for, the first action it suggests, the
expected outcome, and unclear terminology. Then use the visible Create form with the exact parent,
folder, title, and optional description emitted by preparation. Confirm the visible title, path,
and project status before proceeding.

### 2. Configure only the local story contract

Through Story Configuration, choose a close Mara-centered prose mode, a restrained suspense tone,
and a content envelope suitable for isolation and relational unease. Keep the story contract
local: the segment concerns perception and recognition, not an answer to the mystery. Record any
unclear or disproportionate configuration work. Do not compensate for unclear UI by consulting
docs or source.

### 3. Draft Mara through the local Cast clipboard path

1. Open the visible Cast record creation surface for Mara without pre-populating authoritative
   record prose.
2. Open the visible static drafting prompt. Before extraction, record the prompt kind, displayed
   scope/counts/version metadata, and its visible fingerprint if supplied. Exercise the rendered
   **copy prompt** clipboard control, and, while the exact prompt remains visibly open, use
   `browser-act.mjs text-file` to preserve those exact visible bytes in the run's `/tmp/exchange`
   directory without printing them.
3. Fingerprint the extracted file with lowercase SHA-256. Send only that file to one genuinely
   fresh cold context, with no parent turns, story brief, expected answer, defect hypothesis,
   repository context, or earlier response. Require the raw response to be saved verbatim to a
   new `/tmp/exchange` file and return only `Response saved.`
4. Assess the untouched response against the visible Cast output contract before importing it.
   Record unsupported invention, omissions, structural compliance, and expected correction work.
5. Use only the visible local paste/import control to place that saved response into the Cast
   import-review surface. Do not inject a provider result, call an API, or mutate the DOM.
6. Before save, visibly review every imported field. Correct the draft to the sealed facts: Mara
   is alone at the observatory; she recognizes the calibration sequence's association with Elias;
   estrangement is established; the sender, location, motive, Elias's presence, and Elias's safety
   are not. Remove invented biography or mystery answers that the local segment does not need.
7. Explicitly save the corrected record and confirm its visible saved state. Only this corrected,
   saved version can serve as record authority.

### 4. Draft Elias through the same boundary

Repeat the exact-prompt, fresh-context, untouched-response assessment, visible paste/import,
field-by-field review, correction, explicit save, and visible confirmation sequence in a new Cast
record for Elias. Use a new prompt file, fingerprint, response file, and fresh cold context.

The corrected record may establish only that Elias is Mara's estranged former mentor and that the
private calibration sequence is associated with him. It must not establish that he authored the
signal, is at the observatory, is alive, is safe, has a present motive, or expects a reply. Preserve
these unknowns explicitly enough that later prompt use cannot casually collapse them.

After both saves, record the visible record count and a privacy-safe summary of the authority now
present. Do not preserve either full imported draft or full record payload in repository evidence.

### 5. Working set, cast functions, notes, and brief

1. Add the two saved records to the Active Working Set through visible controls. Record the selected
   membership before changing participation bands.
2. Assign Mara the visible POV/active/onstage function appropriate to a physically solitary local
   scene. Assign Elias the offstage-pressure/relevance function with compressed relevance. Verify
   that he is not shown as physically present or as a speaker.
3. Use Private Notes only if a real scratch need arises, such as tracking the unanswered sender
   hypothesis. Keep notes inert and visibly distinct; re-author any decision needed for generation
   into a canonical record or the Generation Brief.
4. Populate a natural Generation Brief and predeclare each field's expected observable influence:
   - manual directive: Mara notices, checks, and recognizes the sequence without choosing a reply;
   - setting: the storm-isolated observatory, with Mara physically alone;
   - knowledge/reveal boundary: association with Elias is known, identity and all locked unknowns
     remain unresolved;
   - continuity/current state: the signal is present and the relationship is estranged, but Elias
     is not established onstage;
   - stopping point: stop before Mara decides whether or how to reply;
   - any universal POV or prose-mode minimum required by the UI.
5. Leave other optional fields blank unless a sincere author need arises. Save the draft, inspect
   readiness, and resolve blockers only through the canonical visible UI.

### 6. Capture the offstage prompt state

With Elias still in compressed offstage participation:

1. Open the visible prose prompt inspector; do not send it.
2. Record displayed fingerprint, template/compiler versions, scope, record/field counts, readiness,
   and source disclosure.
3. Use the inspector's visible search control to verify the presence and placement of Mara, Elias,

4. Extract the complete visible prompt with `text-file` to `/tmp/exchange`, fingerprint the exact

5. Capture a tight privacy-safe view of the visible cast-band/metadata state if it will be needed

### 7. Promote Elias without authority leakage

1. Before the action, record a boundary snapshot: both record identities and visible saved status,
   Active Working Set membership/count, Elias's current offstage band, and current prompt
   fingerprint.
2. Use the dedicated visible cast-participation control to promote Elias from compressed offstage
   relevance to the active band. Do not open an edit/save path for either record during this

3. Immediately verify through visible UI that:
   - Elias's participation band alone changed;
   - Mara and Elias remain the same two selected working-set members;
   - the saved record summaries remain unchanged;
   - no new physical-presence, speech, sender, motive, location, or safety assertion appeared.
4. If record content, canon, or membership changed silently, stop and record the visible state as a

5. Reopen/recompile the prose prompt through the visible UI. Capture and fingerprint the exact

## Cold prompt evaluations

### Isolation and accounting

- The two static Cast drafting prompts each receive one fresh, single-prompt cold execution. A
  retry is permitted only if the response is meaningfully poor or malformed, uses the same exact
  prompt bytes, and is performed in another fresh context without feedback from the first.
- Only the final post-promotion prose prompt is dispatched for candidate prose. The pre-promotion
  prompt is a visible participation comparison, not a candidate attempt and not a targeted
  counterfactual.
- Every fresh context receives exactly the baseline exchange instruction and, when file custody is
  needed, the baseline delivery instruction. No story explanation or desired correction is added.
- Record exact prompt fingerprints, timestamps, executor host family, exact model only if exposed
  (otherwise `unknown`), exposure boolean, raw verdict, adoption, and burden.
- Count actual unchanged-prompt attempts faithfully in report frontmatter. The Cast drafting and
  any naturally invoked post-acceptance assistance responses count as cold assistance attempts;
  the final prose response and unchanged-prompt retry count as cold prose attempts. There is no
  paired draw and no diagnostic counterfactual.

### Final prose prompt and field-influence gate

Before dispatch, complete the visible-prompt evidence column for every deliberately populated
prompt-facing brief field and the cast-participation comparison. Then send the exact
post-promotion prompt to one fresh context and preserve the raw response under `/tmp/exchange`.

Judge the untouched response for:

- the observatory-only local scope and Mara-centered POV;
- Mara's physical solitude despite Elias's promoted prompt participation;
- association without claiming Elias sent the signal;
- preservation of every sender/location/motive/presence/safety unknown;
- no reply decision and no planning of how to reply;
- intended voice, continuity, stop-point discipline, unsupported durable change, and prose quality.

For each Generation Brief field, record response evidence and one verdict from `used`, `partly
used`, `not compiled`, `not observably used`, `contradicted`, or `not assessable`. Separate prompt
contract mismatch from model-output failure. If the first prose response is meaningfully poor,
take at most one fresh retry on unchanged prompt bytes and assess it independently without telling
the second context what failed. Choose the response an author would sincerely rather edit.

Ideate is skipped unless the author genuinely becomes stuck; this premise already supplies a local
directive and stopping point. Record Hygiene is skipped unless the reviewed records actually
overlap, restate one another, or become stale. Segment Reconciliation is naturally eligible after
acceptance if the segment creates a durable event or state worth reviewing; if invoked, inspect and
extract its exact visible prompt, use one fresh cold context, assess the raw response, and manually
re-author only adopted changes through canonical editors. Never inject cold output into a provider
result card, and never click an OpenRouter control. Explain every invocation or skip in the report.

## Acceptance and continuity gate

1. Preserve the chosen raw prose assessment before import. Open **Write or paste candidate** and
   use `fill-file` to enter the selected temporary response through the visible editor.
2. Edit only as an ordinary author. `none` or `light` means copyediting and a few local corrections
   while retaining the response's segment. If the candidate needs multiple continuity, POV,
   physical-presence, mystery-answer, or stopping-point repairs, classify it `substantial`; if most
   useful prose must be replaced, classify it `rewrite`.
3. Accept only if the final candidate is coherent, remains at the observatory, keeps Mara alone,
   preserves the association-versus-identity boundary and all locked unknowns, and stops before a
   reply decision, with intervention no heavier than `light`.
4. If neither of the two allowed unchanged-prompt prose responses meets that gate, stop with
   `prose-quality-blocker`. Do not manufacture completion by silently rewriting the segment.
5. Accept exactly once through the visible control. Visit Accepted Segments and visibly confirm
   sequence `1`. If acceptance cannot be proven in the archive, do not claim completion.
6. Inspect the durable-change reminder. Treat the accepted segment only as bounded evidence. If it
   establishes a durable receipt/detection/reaction state worth carrying forward, use the natural
   Segment Reconciliation cold path described above and manually update the appropriate records or
   Generation Brief fields. Do not infer any mystery answer from the prose.
7. Leave the project ready for continuation with the unresolved response point intact. Do not
   begin a second prose generation.

Immediate terminal blockers include a provider-send attempt; unavailable guarded browser or cold
context after bounded recovery; inability to create the project, inspect an exact prompt, import and
review Cast drafts, save corrected records, preserve the promotion authority seam, enter a local
candidate, accept it, or verify sequence `1`; unsafe or ambiguous project state; and failure to
obtain a candidate within T07's `light`-intervention ceiling.

For a probable visible blocker, first record the last successful action, intended action, expected
result, and visible state; capture privacy-safe evidence; inspect only the already-running
diagnostic streams and a narrowly relevant tree/box/DOM fragment; then allow one visible recovery,
one reload if persistence should be safe, and one fresh guarded browser against the same app and
project. Never restart the app, create a replacement project, bypass readiness, or repeat until the
state looks favorable.
