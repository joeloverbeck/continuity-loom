# T01 — New Sensitive Opening: Operator Packet

## Run charter

- Mode: `new_story`.
- Work as a source-and-doc-blind first-time author through the visible browser UI only.
- Create a unique project under the prepared `/tmp/continuity-loom-playtest-projects/` path.
- Accept exactly one local prose segment, then perform deliberate post-acceptance continuity work. If the journey cannot proceed after bounded recovery, produce an honest blocked report.
- Do not use OpenRouter, direct APIs, project files, hidden state, DOM mutation, source, tests, or product documentation during the author journey.
- Leave the separate `/tmp` story project intact for continuation.

Sealed story intent:

- POV: Hina Mori, sixteen, Japanese, from Yokohama.
- Opening state: Hina is conscious on wet shrine stone in Veyr.
- Onstage: Hina and injured shrine warden Calen.
- Physical action: Calen is bracing a failing barred door.
- Offstage pressure: something unidentified is acting against the door; it must not be identified in this segment.
- Object geometry: a dormant ceremonial sword remains mounted within two steps of Hina.
- Required ending: Hina deliberately agrees to help and grips the still-mounted sword.
- Hard stop: before sword activation, breach, combat, threat identification, prophecy, or transfer explanation.
- Mental-health boundary: Hina’s intrusive thoughts remain ego-dystonic and non-causal.
- Exclusions: magical mind-reading, violence stereotyping, cure-by-isekai, sexualization, destiny shortcuts, diagnostic exposition replacing characterization, or an OCD-derived supernatural mechanism.
- What remains undecided: the sword’s behavior, the outside threat’s identity, the transfer’s cause, and what follows Hina’s grip.

The quantitative ledger remains inactive because T01 does not request action or field counts. All four bounded pilots are `awaiting-disposition`; record zero counters and do not run or reconstruct them.

## Cold prompt evaluations

For prose and every naturally invoked assistance surface:

1. State the author need and expected result before opening the prompt inspector.
2. Record visible prompt kind, fingerprint, versions when shown, scope, record/field counts, readiness, and source disclosure.
3. With the exact prompt visibly open, extract it through `text-file` into the run’s `/tmp` exchange directory. Never print it or obtain it from an API or project file.
4. Send only that exact prompt file to a genuinely fresh, non-forked cold context with the baseline instruction to perform it as the sole task. When file custody is needed, require a verbatim response file and only a “Response saved” confirmation.
5. Give the cold evaluator no story synopsis, expected answer, suspected defect, repository context, prior response, or operator conversation.
6. Assess the untouched response before editing:

   - contract and structural compliance;
   - factual and continuity grounding;
   - unsupported invention;
   - immediate usefulness;
   - omissions, overreach, contradictions, and false positives;
   - adoption, rejection, reinterpretation, and repair burden;
   - likely layer;
   - verdict and confidence.

7. For prose, allow one unchanged-prompt retry in a new cold context only if the first response is meaningfully poor or malformed. Do not disclose the first response’s defects to the retry.
8. For each populated Generation Brief field, search the visible prompt inspector for its distinctive saved value and label. Record `present`, `absent`, or `unclear`, placement/prominence, response evidence, and one of the prescribed influence verdicts.
9. A single targeted counterfactual is permitted only if a high-impact field is visibly compiled but appears ignored and the comparison could change a product decision. Its response is diagnostic only and can never enter the app.
10. Evaluate assistance specifically:

   - Ideate: grounded, distinct, locally usable non-prose options/questions versus generic restatement or branching.
   - Record Hygiene: real change proposals versus low-value reassurance, false merges, missed overlap, or unsafe authority collapse.
   - Segment Reconciliation: explicit supported deltas, valid targets, paraphrased evidence, and no invented canon.

11. Manually re-author only independently accepted assistance into canonical visible editors. Never inject a raw assistance response or treat it as authority.
