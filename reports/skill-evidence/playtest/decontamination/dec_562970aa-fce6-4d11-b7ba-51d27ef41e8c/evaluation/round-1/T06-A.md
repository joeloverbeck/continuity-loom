# T06 Candidate Operator Packet — Offstage Pressure Only

## Trial scope and terminal condition

This packet specifies a future, real-browser `new_story` playtest; it does not
execute that playtest. The operator must remain source-and-doc blind during the
author journey, use only visible UI controls, and make no API, filesystem,
database, hidden-state, DOM-mutation, runtime-evaluation, or provider request.

The run succeeds only after exactly one local prose segment is explicitly
accepted and its new sequence is verified in **Accepted Segments**, followed by
deliberate author-side continuity review. Otherwise it ends in a truthful
blocked report after the bounded recovery policy. Never manufacture acceptance
or continue into a second segment.

## Sealed pre-use run charter

Record this charter in the run scratchpad before opening the app.

| Charter dimension | Sealed instruction |
| --- | --- |
| Story intent | Nera Vale maintains an unmanned tidal-ferry prototype and must choose safety and evidence preservation over a stamped launch order that threatens her certification. |
| Local segment | In close third from Nera's viewpoint, she discovers and verifies a hairline drive-coupling fracture before the noon launch test; refuses the unsafe launch; photographs and tags the defect; locks out the drive; and preserves the evidence. |
| Exact stop | Stop after the evidence is secured and the drive is locked out, before Iven Marr appears or any next-segment confrontation begins. |
| POV and presence | Nera is the sole viewpoint and sole onstage person. Iven is pressure-only and offstage. There is no silent or minor onstage participant. |
| Iven boundary | His only influence is the already-existing paper launch order. Do not supply live speech, a current location or action, thoughts, intentions, arrival, response, or implied offstage communication. |
| Knowledge and reveal | At the opening, Nera has not yet completed the segment's discovery and verification. The prose may establish the fracture through her local inspection, but it must not grant her knowledge outside that process or reveal new information about Iven. |
| Physical continuity | The ferry is unmanned; the drive has a hairline coupling fracture; the launch test is due before noon; the order already exists on paper. Required end state: defect photographed and tagged, drive locked out, evidence preserved, launch refused. |
| Forbidden content | No third person, crowd, call, radio voice, flashback speaker, injury, passenger peril, romance, or violence. Do not turn the test into an occupied or already-launched emergency. |
| Voice and prose | Close-third Nera, physically legible maintenance work, local scope, and controlled pressure. Do not compensate for the one-person scene by inventing dialogue or biography. |
| Undecided/reserved | Exact confrontation content, Iven's arrival, and every new fact about Iven remain for a later segment. Local sensory and maintenance detail may be chosen only when it does not create new continuity authority or violate the bounds above. |
| Expected records | One durable active dossier for Nera; an ENTITY-first, minimal representation of Iven; and only the smallest visible records needed for the prototype, existing paper order, fracture/opening knowledge state, and local physical continuity. Avoid duplicate authority. |
| Working Set | Nera is selected in the visible POV/onstage function. Select Iven only in an explicitly offstage-pressure function if the UI requires that selection. Include only local object/fact/state records required for the segment. |
| Generation Brief | Express the local directive, opening state, POV/knowledge boundary, participation boundary, forbidden-presence/content list, and exact stop. Use optional fields only when their visible purpose is useful. |
| Private Notes | No sincere inert scratch need is present at setup, so skip. If a real temporary thought arises, keep it inert; independently author any adopted decision into a canonical record or brief field. |
| Assistance expectation | Ideate is not initially needed because the action and stop are already settled. Record Hygiene is not initially needed because the two people have intentionally distinct representations. Segment Reconciliation is expected after acceptance because the segment deliberately changes the fracture/evidence/lockout state. |
| First-time-author model | Expect visible configuration, records, a selected working context, a brief, readiness feedback, an inspectable local prompt, candidate entry, explicit acceptance, an accepted archive, and a reminder to review durable changes. Treat the UI's own labels/help as authority for how to reach them. |
| Explicit non-goals | Do not test every optional field, generate Iven material, add biography/appearance/relationships/voice guidance for Iven, create a private note for coverage, invoke assistance for coverage, or explore a second segment. |

Do not activate a quantitative observation ledger; T06 asks for no action,
field, or promotion counts.

## Guarded setup

   working title for this story. Use every unique path from its receipt exactly,
   including the proposed `/tmp` project path, report stem, evidence directory,
   scratchpad, exchange directory, run root, and isolated config directory. Do
   not create the story project by filesystem action.
2. Before any run-owned repository write beyond the prepared evidence
   directory, append the exact `git status --short` baseline to the scratchpad.
   Do not inspect, adopt, clean, restore, or overwrite pre-existing dirt.
3. Build once. Start the isolated app through `safe-app-session.mjs` using
   literal absolute repository, session, and config paths. Require a blank
   `OPENROUTER_API_KEY`, loopback-only binding, and the ephemeral `baseUrl` from
   `app-session.json`; never assume a port or reuse an existing process.
4. Before the first navigation, start `browser-session.mjs` with that exact
   origin, the prepared evidence/scratch directories, an ephemeral holder port,
   a fresh browser context, and a 1440x900 viewport. Confirm the origin guard and
   provider-send-route guard are active. Never click **Generate**, Ideate,
   Analyze, Review, model refresh, or any other send control. A send attempt
   terminates the run even if blocked.
5. Drive only through `browser-act.mjs`, one visible action per command. Prefer
   roles, accessible names, labels, and visible text. Do not use test IDs,
   source-derived selectors, hidden elements, `evaluate`, `run-code`, or direct
   request substitution. After consequential actions, wait and inspect fresh
   visible output.

If Chromium cannot start, follow the single permitted install/retry path. If an
exact loopback operation receives `EPERM`, use one narrow approval retry without

## Cold prompt evaluations

### Exact prose prompt

Before opening the inspector, write down the author need: a usable, bounded
one-person segment that completes every required action and stops before Iven's
arrival without inventing offstage activity. With the inspector visibly open:

1. Record the visible prompt kind, versions, scope, source disclosure, record
   counts, readiness, and a SHA-256 fingerprint.
2. Use `browser-act.mjs text-file` to save the complete visible prompt to the
   prepared `/tmp/.../exchange/` directory without printing it. Do not retain a
   prompt-body screenshot.
3. Send that file to one genuinely fresh, no-parent-context subagent with only:

   ```text
   Read exactly the complete prompt in <absolute-prompt-file> and perform it as the sole task.
   Do not inspect the repository, any other file, the parent conversation, or prior responses.
   Return only the output requested by that prompt.
   ```

   If file custody is required, add only the supplied instruction to write the
   response verbatim to the absolute response path and return `Response saved.`
4. Assess the untouched response before editing it. In addition to general
   prompt compliance, grounding, prose quality, unsupported durable change, and
   stop discipline, check every T06 invariant: only Nera is onstage and
   perceived; Iven exists only through the paper order; there is no Iven speech,
   current action/location/thought/intention, arrival, or newly invented
   communication; there is no extra person or forbidden content; the ferry
   stays unmanned; every required action occurs coherently; and the segment ends
   before confrontation.
5. Allow exactly one unchanged-prompt retry in another fresh context only if
   the first response is materially poor or malformed. Supply no critique or
   desired correction. Select the response an author would prefer to edit. If
   neither response can become acceptable without replacement-level rewriting,
   stop with `prose-quality-blocker`.

For each deliberately populated Generation Brief field, search the visible
inspector for its distinctive saved value and label. Record `present`, `absent`,
or `unclear`, placement/prominence, response evidence, verdict (`used`, `partly
used`, `not compiled`, `not observably used`, `contradicted`, or `not
assessable`), and confidence. A single one-field counterfactual is permitted
only if an important visibly compiled field appears ignored and the result
could change a product decision; it stays diagnostic and never enters the app.

### Naturally invoked assistance

- **Ideate:** skip at setup because the action chain, pressure device, and stop
  are already specified. If a sincere local choice arises before committing a
  directive, inspect and cold-evaluate its exact visible prompt in a fresh
  context. Reject options that add people, dialogue, an Iven scene, unsupported
  biography, peril, or a branch beyond the stop.
- **Record Hygiene:** skip unless visible authoring creates plausible overlap,
  staleness, or uncertainty among the minimal records. If invoked, separately
  score actual change proposals and no-change reassurance; never merge Iven into
  Nera or inflate his entity into a dossier merely for completeness.
- **Segment Reconciliation:** invoke after acceptance because fracture
  verification, the photograph/tag, drive lockout, refusal, and evidence
  preservation plausibly change durable state. Cold-evaluate the exact visible
  prompt in a fresh context. Require real deltas contrasted with current state,
  paraphrased evidence, valid targets/shapes, and clear manual work. Reject any
  proposal that invents Iven's state, motives, location, response, arrival, or
  new canon.

Each cold attempt gets a different fresh context and only its exact visible
prompt. Assistance suggestions are never injected into provider-result UI and
never become authority automatically. Log contract compliance, actionable
output, low-value/no-change output, adoption, rejection, repair burden, likely
layer, verdict, and bounded confidence. State in the final report that this path
does not exercise provider response parsing or result cards.

## Candidate, acceptance gate, and authority boundary

1. Preserve the raw prose assessment. Click **Write or paste candidate** and
   fill from the temporary response file. Make only ordinary visible author
   edits. Record intervention as `none`, `light`, `substantial`, or `rewrite`;
   do not disguise replacement-level rewriting as editing.
2. Before accepting, re-read the visible candidate and require all charter
   bounds. In particular, an incidental Iven line, approach, call, attributed
   present action, or offstage cutaway fails the gate; so does any third person,
   occupied ferry, omitted safety/evidence action, or continuation beyond the
   pre-arrival stop.
3. Accept once, through the visible acceptance control. Navigate to **Accepted
   Segments** and visibly verify a positive new sequence. Do not infer success
   from a toast, editor state, or candidate disappearance.
4. Inspect the durable-change reminder. Treat the accepted prose only as bounded
   evidence, never as canon or prose-prompt authority. Review the reconciliation
   response, then manually author only chosen and verified deltas into canonical
   visible records or Generation Brief fields: fracture verified, defect
   photographed/tagged, drive locked out, unsafe launch refused, evidence
   preserved. Keep Iven's representation minimal and unchanged unless the
   segment itself truthfully requires a canonical update (it should not).
5. Reopen the relevant visible canonical surfaces and confirm that every claimed
   post-acceptance update is actually present. Do not claim reconciliation from
   the assistance output alone. Stop; do not compile or accept another segment.

## Evidence and bounded diagnostics

Maintain timestamped scratch observations that separate visible fact from
interpretation. For the report, retain only privacy-safe evidence that materially
supports a finding or success condition:

- a tightly cropped accepted-archive view showing the verified sequence without
  accepted prose;
- a tightly scoped view showing the post-acceptance durable-change reminder and,
  if needed, a separate canonical-state confirmation without full payloads;
- a narrowly cropped visible working-context view only if needed to prove Nera
  is onstage and Iven remains offstage/minimal; and
- safety counts and browser/app metadata showing the guarded origin, blank
  credential posture, and zero send clicks, provider attempts, and blocks for a
  completed run.

Keep routine screenshots, full prompts, raw responses, candidate/accepted prose,
project files, session files, traces, app logs, and uncited evidence under
`/tmp` only, then remove them at close. A prompt fingerprint, field-influence
result, verdict, count, burden, or shortest necessary excerpt is sufficient for
the report.

For a probable defect or blocker, first record visible state, intended action,
expectation, and last successful action; then capture privacy-safe evidence.
Inspect existing guard/console/network streams and only then a relevant visible
accessibility tree, geometry result, or DOM fragment. Try one safe visible
recovery, reload once if persistence should be safe, and finally make one fresh
guarded-browser attempt against the same app and project. Never restart the app,
replace the project, randomly click, weaken validation, or inspect source,
stores, request bodies, SQLite, hidden prompt data, or project files.

Any send-control click or provider-block entry ends the run immediately as
`status: blocked` / `completion_reason: provider-request-attempt`. Other terminal
conditions include unavailable cold isolation, two unusable prose attempts, no
visible route to readiness/prompt/candidate/acceptance/sequence proof, repeated
crash or state loss, or a UI requirement that makes the one-person/offstage-only
contract impossible to preserve after bounded recovery.
