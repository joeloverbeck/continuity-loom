# T05 Operator Packet — Continuation Author-Invention Boundary

## Trial status

This packet is an instruction-level simulation. It specifies the operator's
future run but does not claim that the app was launched, the historical project
was opened or changed, prompts were extracted, cold responses were produced, or
a segment was accepted. No provider request is part of this trial.

## 1. Run charter

### Continuation identity

- Run mode: `continuation`.
- Sole prior report: `reports/playtest-the-winter-letter-2026-07-20T023325Z.md`.
- At execution time, read that report completely and no unrelated report. Its
  `project_path` is the sole project locator.
- The project path must be present, under `/tmp`, and still exist. If any of
  those conditions fails, do not reconstruct the story from the corpus or
  prose; produce a complete blocked report with completion reason
  `continuation-project-missing`.
- Reopen the supplied project only through the visible **Open** form. Before
  changing anything, visit **Accepted Segments** and visibly confirm that the
  latest segment is sequence 2. If the archive does not establish a safe
  continuation point, stop rather than guessing.

### Sealed local story intent

Continue in Clara's sanatorium room on Christmas Eve 1972 for exactly one local
segment. Clara must design a test that does not depend on Tomás's testimony,
Daniel's testimony, or her desire for either account to be true. She must be
able to set up and check it herself using ordinary objects available in 1972.

The author must decide the test's actual mechanics during this run. The system,
a cold response, and candidate prose may propose possibilities but may not
silently supply continuity authority. Before prose compilation, the author must
choose the test, re-author its exact terms in an appropriate visible canonical
record, and express the segment-local directive in the Generation Brief.

The segment ends after the terms are fixed and Clara begins sealing her
question. At that point all of the following remain unresolved:

- the test's result;
- the content of the sealed question;
- whether time travel is real;
- any commitment by Clara beyond the already bounded breakfast horizon.

Hard exclusions: do not leave the room, wake Daniel, give staff a speaking
part, reveal the sealed question, deliver the result, let the test prove time
travel, or extend Clara's commitment. Preserve any tighter POV, voice,
knowledge, presence, physical-state, or content constraints found in the prior
report; do not infer missing constraints from source or documentation.

### Pre-use author model and non-goals

Before launching the app, record in the temporary scratchpad:

- Clara as the intended focal character, subject to confirmation from the prior
  report;
- every intended speaker, silent/onstage presence, and offstage pressure taken
  from the prior report and this charter;
- the exact local start, stop, physical bounds, knowledge/reveal locks, voice,
  prose mode, maturity envelope, and content constraints;
- the need to author one falsifiable-but-non-proving 1972 test and the observable
  purpose of each record and Generation Brief field used to express it;
- the expected use of Ideate, if the author sincerely needs alternatives or
  questions; and naturalistic reasons to skip Record Hygiene or Segment
  Reconciliation if no such need arises;
- explicit non-goals: story-wide planning, resolving the test, establishing
  time travel, adding a second segment, or turning prose into canon.

The first-time-author mental model is: use visible help and labels, create only
the continuity needed for the next segment, resolve readiness blockers through
visible canonical controls, and accept at most one segment.

## 2. Guarded setup and custody

1. From the repository root, run the supplied `prepare-run.mjs` helper with the
   working story title and the exact prior-report path. Use every path in its
   receipt unchanged. Do not substitute a repository project path or reuse an
   earlier run directory.
2. Before any run-owned repository write beyond the helper-prepared evidence
   directory, append exact `git status --short` output to the allocated
   scratchpad. Treat all pre-existing changes as user-owned; do not inspect,
   adopt, clean, or restore them.
3. Build the app, then start a fresh isolated app holder with literal absolute
   paths, the allocated `/tmp` session/config directories, blank
   `OPENROUTER_API_KEY`, loopback-only binding, and an ephemeral port. Read the
   base URL from `app-session.json`; never assume port 4173 or reuse an existing
   app.
4. Before first navigation, start the guarded fresh Chromium context at
   1440×900 using the allocated evidence and scratch directories and the exact
   base URL. Wait for `session.json`. The guard must reject provider-send routes
   and requests outside the app origin.
5. Drive only the visible browser through the supplied browser helper, one
   visible action per command. Prefer roles, accessible names, labels, and
   visible text. Do not use hidden elements, source-derived selectors, runtime
   evaluation, direct APIs, saved profiles, project files, databases, or app
   internals.
6. Never click **Generate** or any Ideate, Analyze, Review, or model-refresh send
   control. A click or any provider-request guard entry ends the run immediately
   as blocked with `provider-request-attempt`, even if the guard prevented the
   request.

Routine screenshots, the scratchpad, config, exact prompts, raw responses, and
candidate prose remain under the allocated `/tmp` run root. Only tightly scoped,
privacy-safe evidence may be promoted to the run's repository evidence
directory.

## 3. Visible author journey

### Reopen and establish the continuation point

1. Open the report's `/tmp` project path through the visible **Open** form.
2. Visit **Accepted Segments** before editing. Confirm sequence 2 and read its
   endpoint only as review evidence. Do not copy accepted prose into a record,
   Generation Brief, or prompt, and do not treat it as canon.
3. Use the visible archive and current canonical surfaces to identify durable
   changes still needing author review. Do not fill gaps by invention.
4. Record the visible continuation state, what the UI makes clear, and what
   remains unclear before changing anything.

### Make the test an author-owned fact

Treat the missing test as a required author decision, `A-DECISION-01`:

1. Inspect Story Configuration and the minimum relevant Records through visible
   UI. Preserve the inherited story contract and constraints.
2. If the author sincerely needs help designing the test, open Ideate locally,
   state the need as alternatives or diagnostic questions, inspect its exact
   visible prompt, and use the cold path in section 4. Do not click its provider
   send control.
3. Assess any returned alternatives only as suggestions. Reject options that
   depend on testimony or wishful belief, require objects unavailable to an
   ordinary person in 1972, cannot be set up and checked by Clara herself,
   disclose the result or hidden question, claim to prove time travel, require
   leaving the room or waking Daniel, introduce staff dialogue, or expand the
   commitment beyond breakfast.
4. The author chooses one option or authors a different one. Rephrase it in the
   author's own words and enter the exact setup, checking method, and evidentiary
   limit in the appropriate visible canonical record surface. The record must
   explicitly preserve that no result occurs in this segment and that any
   eventual result cannot prove time travel.
5. Select only records needed for this local scene in the Active Working Set.
   Include every intended POV, speaker, active silent presence, physically
   active person, present-minor participant, and offstage pressure in the
   appropriate visible function. Do not add irrelevant records for coverage.
6. In the Generation Brief, independently author the local directive, physical
   and temporal bounds, knowledge/reveal locks, desired stop, and the already
   chosen test terms. Record the author need and intended observable influence
   for every populated prompt-facing field before compilation.
7. Save, distinguish blockers from warnings, and resolve blockers through
   visible canonical UI. Validation-focus controls are readiness controls unless
   the UI visibly says otherwise. Never bypass readiness.

Private Notes are permitted only for sincere inert scratch. Any adopted detail
must be independently entered into the canonical record or appropriate brief
field before it can guide generation.

### Naturalistic assistance decisions

- **Ideate:** invoke only if alternatives or clarifying questions would help the
  author decide `A-DECISION-01`; otherwise record that the author could make the
  decision directly.
- **Record Hygiene:** invoke only if visible records create plausible overlap,
  staleness, or uncertainty about what should change. Otherwise record that no
  hygiene need arose.
- **Segment Reconciliation:** after acceptance, invoke only if the latest
  segment plausibly changed durable state or inherited reconciliation remains.
  Otherwise explain the skip. It may propose manual work but cannot make prose
  canonical.

## 4. Cold prompt evaluations

### Exact-prompt procedure

For the prose prompt and every naturally invoked assistance prompt:

1. Before opening the inspector, record the author need and expected useful
   result.
2. With the inspector visibly open, record visible prompt kind, fingerprint,
   versions, scope, counts, readiness, and source disclosure.
3. Use the guarded browser helper's `text-file` operation to save the complete
   exact visible prompt under `/tmp/.../exchange/` without printing it.
4. Give that one prompt to one fresh no-parent-context subagent. Its only task is
   to read the exact prompt file and perform it; it receives no story brief,
   desired answer, suspected defect, prior report, repository context, or prior
   response. If file custody is needed, it writes verbatim to the allocated
   response file and returns only `Response saved.`
5. If genuine cold isolation is unavailable, stop with
   `cold-subagent-unavailable`. Do not simulate a cold response in the parent.
6. Never inject the response into a provider-result UI, intercept a provider
   request, call OpenRouter, or manipulate the DOM.

### Assessment boundary

Assess each untouched response before any manual re-authoring. For prose, test:

- local room and Christmas Eve 1972 scope;
- Clara's POV/knowledge and inherited voice constraints;
- use of the author-chosen canonical test without changing its mechanics;
- exclusion of testimony and wish fulfillment as proof;
- use only of ordinary 1972 objects that Clara can set up and check herself;
- no result, revealed sealed question, proof of time travel, room exit, awakened
  Daniel, staff speech, or commitment beyond breakfast;
- the exact stop after terms are fixed and Clara begins sealing the question;
- unsupported durable changes, genericity, coherence, and prose quality.

Record contract compliance, grounding, unsupported invention, omissions,
overreach, contradictions, repair burden, likely layer, verdict, and bounded
confidence. Allow one fresh unchanged-prompt retry only if the first prose
response is materially poor or malformed. Do not give the second executor a
critique. Choose the response an author would prefer to edit.

For Ideate, value distinct, grounded, locally usable non-prose alternatives or
questions. A suggestion is not adopted until the author independently chooses,
re-authors, and saves it before prose compilation. For Record Hygiene, separate
real changes from no-change reassurance and reject unsafe merges. For Segment
Reconciliation, require current-state contrasts, paraphrased evidence, valid
targets, no invented canon, and clear manual work.

### Generation Brief influence ledger

Before compilation create one row for each deliberately populated
prompt-facing field:

| Field | Author need | Intended observable influence | Visible prompt evidence | Response evidence | Verdict | Confidence |
| ----- | ----------- | ----------------------------- | ----------------------- | ----------------- | ------- | ---------- |

Search the visible prompt inspector for each field's distinctive saved value and
label. Mark prompt evidence `present`, `absent`, or `unclear`, including placement
and prominence. After the cold response, use only `used`, `partly used`,
`not compiled`, `not observably used`, `contradicted`, or `not assessable`.
Do not score IDs, blanks, validation-only controls, or author-only metadata as
ignored prompt content.

Use at most one one-field counterfactual only if an important visibly compiled
field appears ignored and the result could change a product decision. Keep it
diagnostic, give it to a fresh agent, and never use its response in the app or
continuity.

## 5. Acceptance gate

The run may proceed from cold response to the candidate editor only when all of
these are true:

- archive sequence 2 was visibly verified;
- `A-DECISION-01` was made by the author and saved into visible continuity
  authority before prose compilation;
- the visible compiled prompt contains the intended test and stopping bounds,
  or any absence has been assessed and safely resolved through visible author
  controls;
- the preferred cold prose response can become acceptable without
  replacement-level rewriting;
- the response does not decide or alter the test, supply its result, reveal the
  sealed question, prove time travel, violate presence/location bounds, or
  broaden Clara's commitment.

Then click **Write or paste candidate**, paste the preferred local response from
the temporary file, and make only ordinary visible author edits. Record
intervention as `none`, `light`, `substantial`, or `rewrite`. If both permitted
prose attempts require replacement-level rewriting, stop with
`prose-quality-blocker`.

Accept coherent prose exactly once. Immediately verify **Accepted Segments**.
Because the prior latest sequence is expected to be 2, the expected new sequence
is 3, but report 3 only after the UI visibly proves it. Inspect the durable-change
reminder and perform only deliberate post-acceptance record or Generation Brief
updates through visible controls. Accepted prose itself remains non-canonical;
do not claim reconciliation until canonical surfaces reflect every adopted
durable change. Do not generate or accept another segment.

## 6. Evidence plan

Keep observation separate from interpretation. The scratchpad should record
time-ordered visible facts, intended actions, expectations, last successful
actions, recovery, and assessments. Candidate evidence targets are:

- the archive view establishing the prior latest sequence and continuation
  endpoint without retaining accepted prose;
- the visible canonical record proving the author entered the test before
  compilation, redacted to the minimum necessary if promoted;
- prompt metadata and SHA-256 fingerprints, never full prompt text;
- cold-attempt counts, verdicts, adoption/rejection counts, and repair burden,
  never raw responses;
- the accepted-segment archive proving the actual new sequence without retaining
  candidate or accepted prose;
- the durable-change reminder and visible post-acceptance canonical work when
  they support a finding;
- privacy-safe console, network, or guard excerpts only when tied to an observed
  problem.

Routine screenshots, traces, session files, app logs, complete prompts, payloads,
raw responses, prose, API keys, and project files are forbidden durable evidence.
Every retained artifact must be listed under the report's **Evidence Index**
with its purpose. If none is necessary, write `No retained evidence.`

On a probable blocker, record and capture the visible state first, inspect only
existing safety streams and relevant visible diagnostics, try one safe visible
recovery, reload once if state should persist, then permit one fresh guarded
browser attempt against the same app and project. Never restart the app, replace
the project, randomly click, or silently omit a recovered failure.

## 7. Report and terminal handling

### Cumulative report

Always write the unique prepared `reports/<report-stem>.md`, including for a
blocked run. Use the supplied schema-version-2 scalar frontmatter and all 16
required sections in their exact order. This continuation report must name the
exact repository-relative prior report, preserve the `/tmp` project path, and
carry forward every prior cumulative-ledger row with its stable ID and an honest
status. Mark a prior finding resolved only after explicit visible retest.

The report must include:

- the sealed story intent and all T05 knowledge, presence, location, temporal,
  authorship, and stopping boundaries;
- a condensed visible author journey;
- strengths and prioritized findings using a single stable `F###` namespace,
  allowed classifications, bounded confidence, and only evidence tags actually
  earned;
- exact validator-owned tables for prioritized findings, prompt usefulness,
  Generation Brief influence, assistance, and the cumulative ledger;
- naturalistic reasons for every skipped assistance surface and a disclosure
  that cold evaluation did not exercise provider response parsing/result cards;
- candidate intervention and acceptance proof;
- a continuation handoff containing the exact project path/existence, actual
  latest sequence, next unresolved response point, intended POV/cast/current
  pressure, chosen canonical work, outstanding author decisions, useful retests,
  and current report path, but no prose or full payloads;
- diagnostics, evidence index, and honest coverage limitations.

For a completed run, frontmatter must say `status: completed`,
`completion_reason: accepted-one-segment`, the visibly verified positive
sequence, one or two cold prose attempts, the actual assistance-attempt count,
the actual intervention level, and zero OpenRouter send clicks, provider-request
attempts, and blocked provider requests. The three method-evidence compatibility
counters remain zero. For a blocker, use `status: blocked`, a specific kebab-case
reason, and `Not reached — <specific blocker>` in every unreached section.

### Safe closeout

1. Copy safety counts and browser/app metadata into the report material.
2. Request browser-holder shutdown and confirm exit 0, then request app-holder
   shutdown and confirm it. Stop only run-owned processes and record unexpected
   termination.
3. Remove session files, empty streams, routine screenshots, and uncited or
   forbidden evidence. Preserve a continuation project even after a blocker.
4. Validate the report with the supplied validator and fix it until passing.
5. After the first passing validation, delete the temporary exchange and
   scratchpad, validate again, and compare final `git status --short` with the
   baseline. The only new repository delta may be the report and its cited
   evidence directory.
6. Remove a successful run root only after verifying it is a specific child of
   `/tmp/continuity-loom-playtest/` and is not the separate project path. If
   validation never passes, retain and return the exact run root instead of
   copying private material into a recovery report.

The terminal result is therefore exactly one of:

- **Completed:** one visibly verified new accepted segment (expected sequence 3
  only if sequence 2 was verified), deliberate post-acceptance continuity work,
  zero provider-request attempts, a passing cumulative report, stopped run-owned
  processes, preserved project, and no unrelated repository delta.
- **Blocked:** a specific honest terminal blocker after bounded recovery, no
  manufactured acceptance, a complete passing blocked report where possible,
  preserved project and diagnostic custody, and no unrelated repository delta.
