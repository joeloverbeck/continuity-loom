# T05 Candidate Operator Packet — Continuation Author-Invention Boundary

## Trial disposition

This is an instruction-level simulation of the supplied playtest procedure. It
does not launch Continuity Loom, open or mutate the historical project, execute
prompts, make provider requests, or claim observations from a real browser run.
The instructions below are the complete run packet for a later operator.
That operator remains source-and-doc blind throughout the author journey and
learns the product only from the visible browser UI and its field help.

## Run charter

- **Mode:** continuation.
- **Only prior report permitted:**
  `reports/playtest-the-winter-letter-2026-07-20T023325Z.md`. Read it completely
  at the start of the real run, and read no unrelated playtest report. Use only
  its `project_path` to locate the project.
- **Continuation precondition:** the reported project path must be an existing
  child of `/tmp`. If it is absent, outside `/tmp`, missing, or cannot be opened
  visibly, do not reconstruct it from the report's prose. Produce a blocked
  cumulative report with `completion_reason: continuation-project-missing`.
- **Starting proof:** before changing story state, open **Accepted Segments** and
  visibly confirm that the latest accepted sequence is 2 and that its endpoint
  is Clara demanding a test independent of Tomás's testimony, Daniel's
  testimony, and her wish for either account to be true. If the visible archive
  and supplied continuation contract cannot be reconciled safely, stop with a
  specific blocker rather than guessing.
- **Local segment intent:** Clara designs a test she can set up and check herself
  with ordinary objects available in 1972. The segment fixes the test's terms
  and takes Clara only as far as beginning to seal her question.
- **Exact stop:** end while the result is still pending, the sealed question is
  still undisclosed, and Clara's commitment extends no further than breakfast.
- **Place and time:** Clara's sanatorium room, Christmas Eve 1972, throughout.
- **Participation:** Clara is the POV and active speaker. Preserve Daniel as a
  silent, sleeping onstage presence if the visible continuation confirms that
  state. Tomás's and Daniel's testimony remain pressures, not proof. Staff have
  no speaking part. Do not add another active participant merely to make the
  scene move.
- **Knowledge and reveal locks:** do not reveal the sealed question, supply the
  test's result, let any character know a result not yet obtained, or make the
  setup prove time travel.
- **Physical and continuity locks:** do not leave the room, wake Daniel, rely on
  an anachronistic object, or make the test depend on either man's testimony.
- **Non-goals:** no second segment; no answer to the test; no larger promise from
  Clara; no automatic canonization of prose or assistance; no systematic use of
  optional fields or assistance merely for coverage.
- **Quantitative tracking:** inactive. T05 does not request action or field
  counts.

### Sealed author-decision checkpoint

The missing test is an author decision, not a blank the system may fill. The
operator must complete this checkpoint before compiling the prose prompt:

1. Review the prior report and visible canonical state without copying accepted
   prose into a record or Generation Brief.
2. State the author need: devise a fair, materially checkable test whose setup
   Clara controls now and whose result arrives later.
3. If Ideate is sincerely useful, use its local visible prompt only to obtain
   alternatives or clarifying questions. Treat every suggestion as a proposal,
   never as story truth.
4. Make an explicit author choice. Record in the temporary scratchpad which
   test was chosen, why it does not depend on Tomás or Daniel, how Clara can set
   it up and later check it herself, which ordinary 1972 objects it uses, why
   the result is deferred, and why even a positive result would not prove time
   travel.
5. Re-author the chosen test in the operator's own words into the appropriate
   visible canonical record or Generation Brief field. Private Notes alone do
   not satisfy this gate. Save through the UI and visibly verify the saved
   value.
6. Reject any option that embeds the result, discloses the hidden question,
   relies on testimony, requires leaving the room or waking Daniel, extends the
   commitment past breakfast, or establishes time travel as fact.

If the author cannot make and canonically save a choice, do not ask the prose
model to invent one and do not infer one from later candidate prose. Stop before
compilation with a specific `author-decision-unresolved` blocker.

## Guarded setup for the real run

1. From the repository root, run the supplied `prepare-run.mjs` with the story
   title and the exact prior-report path. Use every returned report, evidence,
   `/tmp` run-root, scratchpad, exchange, config, and session path literally.
2. Before any repository write beyond the allocated evidence directory, append
   the exact `git status --short` baseline to the scratchpad. Existing dirt is
   user-owned and must not be inspected, adopted, cleaned, or restored.
3. Build, then start a fresh run-owned app through `safe-app-session.mjs` with
   its isolated `/tmp` session and config directories. Require a blank
   `OPENROUTER_API_KEY`, `127.0.0.1` binding, and the ephemeral `baseUrl` recorded
   in `app-session.json`; never reuse another app or assume port 4173.
4. Before first navigation, start `browser-session.mjs` against that exact
   origin. Require a fresh Chromium context at 1440×900, provider-route blocking,
   exact-origin enforcement, and privacy-safe console/page/network capture.
5. Drive only through `browser-act.mjs`, one visible action at a time. Prefer
   accessible roles, names, labels, and visible text. Do not use internal
   selectors, hidden elements, runtime evaluation, direct APIs, saved profiles,
   project files, or hidden state.
6. Never click **Generate**, any Ideate/Analyze/Review/model-refresh send control,
   or any other OpenRouter control. A click attempt or guard event is terminal,
   even if blocked. Prompt execution is local and cold, outside the app.

## Visible author journey

1. Use the visible **Open** form with the exact `project_path`, then perform the
   starting archive proof above before any edit.
2. Review the visible story configuration, records, current working set, and
   Generation Brief only as needed to regain returning-author context. Accepted
   prose may be read for review but must not be copied into prompt authority or
   treated as canon.
3. Preserve or visibly correct the state required for this segment: Clara as
   POV and speaker; Daniel in the proper silent/onstage function; relevant
   testimony as offstage pressure; no speaking staff. Select only records needed
   for the local segment.
4. Invoke Ideate only if the unresolved-test design creates a sincere need for
   alternatives or questions. Cold-evaluate it, then complete the author-choice
   and canonical-save checkpoint. Useful suggestions must be manually
   re-authored; rejected suggestions change nothing.
5. Fill only naturally necessary prompt-facing fields. For each populated
   Generation Brief field, record the author need and intended observable
   influence before compilation. At minimum the saved brief must make the chosen
   test terms, exact stop, deferred result, hidden-question lock, Clara's bounded
   breakfast commitment, room/time, POV/participation, and no-proof boundary
   available to the prompt through appropriate visible fields.
6. Save, distinguish warnings from blockers, and resolve blockers through
   canonical visible controls. Do not bypass readiness.
7. Open the locally compiled prose prompt in the visible inspector. Do not send
   it from the app. Custody its exact visible bytes to the allocated exchange
   path for cold evaluation.
8. After one accepted segment, verify the new archive sequence, inspect the
   durable-change reminder, and perform deliberate post-acceptance continuity
   work. Segment Reconciliation is natural only if the accepted segment appears
   to change durable state; its output remains advisory. Record Hygiene is
   skipped unless a real overlap, staleness, or uncertainty appears. Explain
   each skip in the report.

At safe points, make light visible checks for focus, accessible naming,
label/help association, color-only state, clipping, scrolling, long-form
readability, and the 1440×900 layout without turning this into a full audit.

## Cold prompt evaluations

For each naturally invoked assistance prompt and for the prose prompt:

1. Before opening its inspector, record the author need and expected useful
   result. With it visibly open, record prompt kind, fingerprint, versions,
   scope, counts, readiness, and source disclosure.
2. Use `text-file` to save the complete exact prompt under the run's exchange
   directory without printing it. Give that file to one fresh executor with no
   parent context, story brief, prior report, expected answer, defect hint,
   earlier response, or repository context. Its sole instruction is to read and
   perform that prompt; if file custody is needed, it writes the response
   verbatim to the designated exchange file and returns only `Response saved.`
3. Use a different fresh context for every attempt and surface. If genuine cold
   isolation is unavailable, stop with `cold-subagent-unavailable`; the parent
   must not simulate the response. No output is injected into app result cards.
4. Assess every untouched response before re-authoring anything: contract
   compliance, grounding, unsupported invention, relevance, omissions,
   contradiction, overreach, adoption, rejection, repair burden, likely layer,
   verdict, and bounded confidence.

### Ideate-specific gate

Useful output is a set of distinct, feasible local alternatives or questions.
Restatements, disguised branches, anachronistic mechanisms, testimonial tests,
pre-decided outcomes, disclosed question text, time-travel proof, and expanded
commitments are rejected. Even a useful alternative becomes story authority
only after the explicit author choice, manual re-authoring, canonical save, and
visible verification described above.

### Prose-specific gate

Assess the untouched prose for local scope, selected test fidelity, physical
state, Clara's POV and knowledge, reveal locks, participant roles, voice,
content policy, stop discipline, unsupported durable change, genericity, and
prose quality. In particular, verify that it:

- uses the exact author-chosen, pre-saved test rather than inventing or replacing
  one;
- keeps Clara in the sanatorium room on Christmas Eve 1972;
- leaves Daniel asleep and silent and gives staff no dialogue;
- fixes the test terms and lets Clara begin sealing her question;
- withholds both the question and the result;
- makes no claim that the test can prove time travel; and
- ends without carrying Clara's commitment beyond breakfast.

Allow one unchanged-prompt retry in another fresh context only when the first
response is materially poor or malformed. Choose the response an author would
prefer to edit. If neither of the two allowed responses can satisfy the gate
without replacement-level rewriting, stop with `prose-quality-blocker`.

For every deliberately populated Generation Brief field, search the visible
prompt for its distinctive saved value and label, record `present`, `absent`, or
`unclear` plus placement, then assess response influence as `used`, `partly
used`, `not compiled`, `not observably used`, `contradicted`, or `not
assessable`. A counterfactual is permitted only if one important visibly
compiled field appears ignored and a one-field comparison could change a
product decision; it uses a temporary prompt copy and fresh executor and never
enters the app or continuity.

## Candidate and acceptance gate

1. Preserve the raw prose assessment before opening **Write or paste
   candidate**.
2. Paste the preferred response through that visible control, then make only
   ordinary author edits. Re-check every prose-specific condition after editing
   and record burden as `none`, `light`, `substantial`, or `rewrite`.
3. Accepted prose cannot establish the test or any other durable fact. The test
   must already exist in canonical state from the author-decision checkpoint.
   Any other durable change suggested by the candidate requires a separate,
   deliberate author decision and visible canonical edit.
4. Accept exactly once only when the prose is coherent and all task boundaries
   hold. Return to **Accepted Segments** and visibly verify sequence 3 (the
   sequence following the confirmed sequence 2), its discoverable endpoint, and
   the one-segment boundary. If sequence proof is absent, do not claim
   completion.
5. Inspect the durable-change reminder. Cold-evaluate Segment Reconciliation if
   sincerely warranted, assess its proposed deltas against current canonical
   state, and manually apply only author-approved changes through visible
   canonical controls. Never copy accepted prose into canon, silently accept a
   reconciliation proposal, reveal the sealed question, or record a result that
   has not occurred.
6. End the run after this post-acceptance continuity review. Do not compile or
   accept another segment.

## Blockers, evidence, and bounded recovery

For a probable defect or blocker, first record the visible state, intended
action, expectation, and last successful action; then capture tightly scoped,
privacy-safe evidence. Inspect existing console/network/guard streams and only
then the relevant visible accessibility tree, geometry, or DOM fragment. Try
one safe visible recovery, reload once if persistence should be safe, and make
at most one fresh guarded-browser attempt against the same app and project. Do
not restart the app, replace the project, randomly click, repeatedly retry,
weaken validation, or mutate files/API state. On recurrence or unsafe ambiguity,
stop and report the specific blocker.

A provider-send click or a `provider-request-blocks.jsonl` entry ends the run
immediately with `status: blocked` and
`completion_reason: provider-request-attempt`. Confirm the guard abort and
blank credential; do not retry.

Evidence is based on visible observation, with interpretation labeled
separately. Keep routine screenshots and all sensitive material in `/tmp`.
Promote only minimal privacy-safe artifacts that directly support a finding or
important strength. Durable evidence must never include full prompts, payloads,
raw responses, candidate or accepted prose, project files, traces, session
files, app logs, credentials, or uncited material. Useful closeout evidence
includes the visible accepted-sequence proof, tightly scoped boundary or
workflow evidence, prompt fingerprints and field-presence results, response
verdicts/counts/burden, guard safety counts, and app/browser metadata.

## Cumulative report and terminal handling

Write a new `reports/<prepared-report-stem>.md`; never overwrite the prior
report. Use the supplied schema-2 frontmatter and all 16 required sections in
their exact order. A successful run reports `run_mode: continuation`, the exact
repository-relative prior report, the preserved `/tmp` project path,
`status: completed`, `completion_reason: accepted-one-segment`, accepted
sequence 3 after visible proof, one or two cold prose attempts, the actual
assistance count, the actual intervention burden, and zero send clicks,
provider attempts, and provider blocks. A blocked run uses truthful blocker
values and `Not reached — <specific blocker>` for unreached phases.

Carry every prior finding-ledger row forward with its stable ID. Mark an old
item resolved only after explicit visible retest; otherwise use the appropriate
`open`, `repeated`, `not-retested`, or `preserve-strength` status. Give every new
strength or problem a stable `F###` ID and separate direct visible facts from
inference. Include all validator-owned tables, the reason each assistance
surface was used or naturally skipped, a disclosure that cold execution did not
exercise provider response parsing/result cards, field-influence findings, raw
response verdicts, acceptance proof, and an evidence index. The continuation
handoff records the exact project path/existence, latest sequence, next
unresolved response point, intended POV/cast/current pressure, chosen canonical
post-acceptance work, outstanding author decisions, useful retests, and current
report path—without accepted prose, full prompt payloads, or the hidden
question.

Before validation, copy safety counts and metadata. Request browser shutdown and
wait for exit 0, then request app shutdown and wait. Stop only run-owned
processes. Remove session plumbing, routine shots, empty streams, and uncited or
forbidden evidence. Validate the report with the supplied validator until it
passes; delete the exchange directory and scratchpad after that first pass,
validate again, and compare final `git status --short` with the baseline. The
only new repository delta may be the report and its cited evidence directory.
Preserve the continuation project. Remove the successful run root only after
confirming it is a specific child of `/tmp/continuity-loom-playtest/` and is not
the separate project path. If report validation cannot pass, preserve and return
the exact run root rather than copying private material into the repository.
