# T07 Operator Packet — Clipboard Cast drafting and promotion

## Trial disposition

This is an instruction-level packet for a future browser run. Creating this
packet does not launch Continuity Loom, allocate a project, invoke a cold
executor, or mutate story data. The run described below is a **new-story** run
whose only successful terminal state is one visibly accepted local prose
segment followed by deliberate continuity review. A specific blocker is the
only honest alternative.

## Run charter

Before opening the app, seal this author contract in the run scratchpad.

- **Story intent:** Mara Chen, alone in a storm-isolated observatory, receives a
  signal that uses a private calibration sequence associated with her estranged
  former mentor, Elias Venn. The scene explores recognition, uncertainty, and
  pressure without resolving the signal's provenance.
- **Next local segment:** Mara detects the signal, recognizes the calibration
  sequence, verifies only what the available observation supports, and reaches
  the immediate pressure of deciding whether to respond.
- **Exact stop:** stop before Mara decides whether to reply, how to reply, or
  whether to take any other consequential response action.
- **Unresolved facts:** sender, sender location, motive, safety, and Elias's
  physical presence are unknown. The sequence is associated with Elias; it is
  not proof that he transmitted the signal or is alive, nearby, safe, hostile,
  or asking for help.
- **POV and voice:** close third person through Mara, past tense, restrained and
  technically concrete rather than expository. Mara is the only onstage body
  and the only speaker. Elias is an offstage pressure and may be recalled, but
  he cannot speak, act in the room, or acquire a known present location.
- **Physical and knowledge boundaries:** remain inside the observatory. Preserve
  storm isolation. Mara may know the calibration sequence and her history with
  Elias; she may not know who sent the signal or why. No external rescue,
  visitor, verified two-way contact, or newly established technology should
  appear.
- **Content and prose bounds:** suspense and estrangement are welcome; avoid
  graphic harm. Do not resolve the mystery, turn the segment into a backstory
  summary, or cross the response-decision stop.
- **Expected records:** one reviewed Cast dossier for Mara, one reviewed Cast
  dossier for Elias, and only the minimum atomic facts/location/object or signal
  records visibly needed for this segment. Imported Cast drafts are proposals,
  never authority.
- **Working Set intent:** Mara is active. Elias is selected once as the same
  Cast member throughout, initially in the offstage/compressed participation
  band. His later promotion changes only that visible participation band; it
  must not create a second membership, edit either dossier, change canon, or
  imply physical presence.
- **Generation Brief intent:** deliberately populate prompt-facing setting,
  knowledge, continuity, and stopping-point fields. Each value must be tied to
  an author need and an observable expected effect before compilation.
- **Assistance intent:** use the visible static Cast drafting prompt for each
  dossier because the task sincerely calls for clipboard drafting. Use Segment
  Reconciliation after acceptance because receiving and recognizing the signal
  plausibly creates durable state to review. Skip Ideate unless an actual local
  possibility gap appears; the premise already supplies the scene direction.
  Skip Record Hygiene unless visible records produce genuine overlap,
  staleness, or update uncertainty. Record the natural reason for every skip.
- **First-time-author model:** visible prompt copying and response import are
  drafting aids; Import is a review boundary; Save is the explicit authority
  action. Participation bands control prompt relevance, not story truth.
- **Non-goals:** no provider generation, no hidden state inspection, no second
  accepted segment, no exhaustive feature tour, no manufactured assistance
  coverage, and no attempt to decide the signal's truth.

   project.
2. Append the exact pre-run `git status --short` to the allocated scratchpad.
   Treat all existing changes as user-owned and do not inspect or alter them.
3. Build the app, then start the isolated app holder with the prepared absolute
   session/config paths. Require blank `OPENROUTER_API_KEY`, `127.0.0.1`, and the
   ephemeral `baseUrl` reported by `app-session.json`; never assume port 4173.
4. Before first navigation, start the guarded fresh Chromium context against
   that exact origin at 1440×900. The guard must reject off-origin requests and
   provider-send routes. A blocked provider route is not permission to click a
   send control.
5. Drive only visible UI with one semantic action at a time, waiting and
   inspecting after consequential actions. No API calls, source-derived
   selectors, hidden elements, runtime evaluation, DOM mutation, saved profile,
   project-file inspection, or trace collection is allowed. Keep routine shots,
   exact prompts, raw responses, and prose under the prepared `/tmp` run root.
6. If the exact loopback operation fails with `EPERM`, use the one narrow
   approval retry. If browser engines are unavailable, request permission for
   the prescribed Chromium install and retry once. Neither recovery permits a
   different origin, credential, framework, project, or app restart.

## Visible author journey

### 1. Enter and create

Inspect the untouched initial screen before typing. Record its apparent purpose,
suggested next action and result, and unclear terminology. Create the project
through the visible Create form using the prepared parent path, folder, title,
and a description that preserves the sealed uncertainty. Confirm the displayed
project path and status.

Set Story Configuration through visible controls: the POV/prose contract,
restrained suspense tone, maturity envelope, physical isolation, knowledge
limits, and exact pre-reply stopping boundary. Do not add facts merely to make a
form look complete.

### 2. Draft and review both Cast records locally

Perform the following separately for Mara and Elias. Do not reuse an executor
context or let one draft become context for the other.

1. Open the visible static Cast drafting prompt for the record and record the
   author need, visible prompt kind/version/scope/disclosure, and expected useful
   result before extraction.
2. With the prompt visibly open, save its complete exact text to the prepared
   exchange directory using the guarded browser text-file path. Do not print it
   in the terminal or supplement it with the story premise.
3. Give a fresh no-parent-context executor only: `Read exactly the complete
   prompt in <absolute-prompt-file> and perform it as the sole task. Do not
   inspect the repository, any other file, the parent conversation, or prior
   responses. Write the requested output verbatim to
   <absolute-response-file> instead of returning its contents. Return only:
   Response saved.`
4. Assess the untouched response for its visible contract, useful specificity,
   omissions, unsupported invention, contradictions, and repair burden before
   import. The raw response remains temporary and non-authoritative.
5. Use only the visible local paste/import path to load the response. Do not
   click OpenRouter, Generate, refresh/send, or any equivalent model control.
6. Treat the visible Import review as mandatory. Correct every field to exact
   story truth. For Mara, keep her alone, observatory-bound, able to recognize
   the private sequence, and uncertain about provenance. For Elias, retain only
   `estranged former mentor`, the calibration-sequence association, and offstage
   relevance; remove invented sender status, motive, location, safety,
   availability, current actions, or physical presence. Leave genuinely unknown
   fields unknown rather than smoothing them over.
7. Explicitly save the corrected record and visibly confirm the saved result.
   Only that reviewed save creates record authority. Preserve a privacy-safe
   observation that Import review occurred before Save; do not retain the raw
   response or a screenshot containing its full contents.

Create only other visibly necessary canonical records. Private Notes may hold
genuine scratch, but any adopted constraint must be independently authored in a
canonical record or brief field.

### 3. Establish the base participation state

In Active Working Set, select Mara for active participation and Elias once for
offstage/compressed relevance. Record the visible membership and band. Save the
brief with the four deliberate prompt-facing needs:

| Field purpose | Author need | Intended observable influence |
| --- | --- | --- |
| Setting | Keep the entire segment in the storm-isolated observatory | Physical description and action remain local; no scene move or visitor |
| Knowledge | Mara recognizes an Elias-associated calibration sequence but cannot identify its sender | Recognition is clear while provenance, motive, location, presence, and safety remain unknown |
| Continuity | Mara is physically alone; Elias is an estranged offstage pressure | Elias may affect thought and stakes but cannot speak or act onstage |
| Stopping point | End before Mara decides whether or how to reply | No response decision, transmission, plan, or mystery resolution |

Resolve readiness blockers only through canonical visible controls. Distinguish
warnings from blockers and never bypass validation.

### 4. Inspect the base prompt, then promote Elias

Compile and visibly open the base prose prompt while Elias is still in the
offstage/compressed band. Record its visible fingerprint, version, scope,
readiness, record counts, and source disclosure. Save the exact prompt under
`/tmp` without printing it. Search the visible inspector for each deliberately
saved brief value and label, recording `present`, `absent`, or `unclear` plus
placement/prominence.

Return through visible controls and promote the existing Elias membership from
offstage/compressed to the visible active participation band. Do not edit or
resave either Cast dossier as part of promotion. Confirm visibly that:

- the same Elias membership still exists exactly once;
- Mara's membership and band did not change;
- Elias's dossier facts and unknowns are unchanged;
- no canon/record mutation or physical-presence claim appeared; and
- the band change itself saved successfully.

Compile and visibly inspect the promoted prompt. Save its exact text separately,
record a second fingerprint and the same metadata, and compare the two temporary
prompt copies without exposing their contents. The comparison must identify the
bounded visible participation difference: whether Elias receives fuller or more
prominent prompt treatment, whether counts/disclosures change, and whether any
new text incorrectly promotes him into physical presence or known senderhood.
Unrelated changes make the comparison inconclusive and must be reported; do not
silently repair them in hidden state. This is an actual visible-state comparison,
not a fabricated temporary-prompt counterfactual.

## Cold prompt evaluations

For the final promoted prose prompt, dispatch a fresh no-parent-context executor
with only the exact-file instruction above and a temporary response path. Assess
the untouched prose for local scope, directive, observatory physical state,
Mara's POV and knowledge, reveal locks, Elias's non-presence, voice/content,
unsupported durable change, genericity, and exact stop discipline. Record likely
failure layer, verdict, confidence, and intervention burden.

One retry is allowed only if the first prose response is materially poor or
malformed. It must use the unchanged exact prompt in a different fresh context
with no critique or prior output. Choose the response an author would prefer to
edit. If neither response can meet every T07 boundary with **no more than light**
ordinary editing, stop with `prose-quality-blocker`; do not accept a
substantial-edit or rewrite result merely to complete the run.

If Segment Reconciliation is sincerely invoked after acceptance, extract its
exact visibly open prompt and use another fresh executor with the same isolation
contract. Evaluate proposed durable deltas against visible current state,
paraphrased evidence, valid targets/shapes, invented canon, and manual-work
clarity before adopting anything. Never inject an assistance response into a
provider result card. Record that clipboard cold evaluation does not exercise
provider response parsing/result cards.

For each populated brief field, complete the skill's field-influence row with
visible prompt evidence and raw-response evidence. A field is not `used` merely
because it appears in the prompt. In particular, Elias becoming more prominent
must not be scored as successful if the response turns him into a present person
or confirmed sender.

## Acceptance and continuity gate

Only after preserving the raw prose assessment may the operator click **Write
or paste candidate**, fill from the selected temporary response file, and make
ordinary visible edits. Accept exactly once only when all of these are true:

- intervention is `none` or `light`;
- Mara remains the sole physically present character and POV;
- the sequence association is recognized without identifying sender, location,
  motive, safety, or Elias's presence;
- the scene remains in the observatory and stops before any reply decision;
- the prose is coherent and no unsupported durable change slipped through; and
- readiness is visibly satisfied without a bypass.

After acceptance, visit **Accepted Segments** and verify the new positive
sequence and endpoint visibly. Do not infer acceptance from a toast or editor
state. Stop prose generation immediately; a second segment is forbidden.

Inspect the durable-change reminder. Review the signal's receipt/recognition and
Mara's still-pending response as possible canonical updates. Adopt only chosen,
accurate deltas through visible record or Generation Brief edits; accepted prose
and reconciliation output never become canon automatically. Reconfirm that
Elias's promoted participation band did not mutate his dossier, establish him as
sender/present, or change Working Set membership. Record the band's closing state
explicitly for the next author rather than pretending it is a story fact.

## Evidence and observation discipline

For every notable event, keep `visible fact`, `expected result`,
`interpretation/impact`, and `confidence/limit` separate in the scratchpad.
Retain only tightly scoped, privacy-safe evidence needed to support findings or
important strengths, for example:

- a cropped visible saved-status view proving each reviewed Cast import reached
  explicit Save, without raw draft contents;
- before/after participation views proving one Elias membership changed band
  while membership and canonical record identity remained stable;
- prompt fingerprints and bounded count/disclosure observations supporting the
  base/promoted comparison, never the prompt bodies;
- an Accepted Segments sequence/status view without accepted prose; and
- guard safety counts plus browser/app metadata.

Keep routine screenshots, prompt bodies, raw Cast/prose/assistance responses,
candidate and accepted prose, project files, session plumbing, and scratch data
under `/tmp`; none belong in durable evidence. Use short privacy-safe
console/network excerpts only when they directly explain a visible finding.
Never retain a trace.

On a probable blocker, first record visible state, intended action, expectation,
and last successful action; capture safe evidence; then inspect only existing
guard/console/network streams and, if needed after the visible observation,
scoped accessibility/geometry/DOM diagnostics. Allow one visible recovery, one
reload if persistence should be safe, and one fresh guarded browser against the
same app/project. Never restart the app or replace the project. Recurrence or
unsafe ambiguity terminates the run.

Any click on a provider send control or any provider-request guard entry ends the
run immediately as blocked with `completion_reason:
provider-request-attempt`, even if the guard prevented transmission. Other
terminal blockers include unavailable exact Cast/prose prompt extraction,
unavailable fresh cold execution, inability to review/import/save locally,
missing readiness/acceptance/sequence proof, unsafe state ambiguity, or two
unusable prose attempts.
