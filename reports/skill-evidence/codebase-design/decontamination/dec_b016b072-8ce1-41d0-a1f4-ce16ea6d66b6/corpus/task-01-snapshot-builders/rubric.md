# Observable success criteria

- Maps the actual responsibilities and dependencies before proposing a seam.
- Defines an implementable interface with a clear owner and no vague `utils` layer.
- Preserves assistance-profile-specific errors, projections, and source boundaries.
- Identifies the interface as the test surface and names focused tests.
- Avoids abstraction whose only benefit is fewer lines or superficial similarity.
- Gives a bounded migration sequence and identifies uncertainty explicitly.
