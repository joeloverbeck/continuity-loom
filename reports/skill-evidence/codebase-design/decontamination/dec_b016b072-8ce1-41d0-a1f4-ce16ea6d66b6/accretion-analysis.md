## Audit-induced risk patterns

| Pattern | Justification |
|---|---|
| Several rules solving one hazard in different places | The dependency taxonomy, adapter threshold, internal/external seam distinction, and test-surface rule appear in both `SKILL.md` and `DEEPENING.md`. The detailed reference can own these while the entrypoint provides an explicit load trigger. |
| Duplicated definitions | Deep/shallow figures, the relationship list, glossary definitions, and the testability examples restate the same depth/leverage/interface concepts. Collapsing them leaves one canonical definition per concept. |
| Rare/detail-heavy material crowding the common path | Four detailed dependency summaries load even for a pure interface-design question. Conditional routing retains the knowledge for I/O, adapters, migrations, and versioned contracts. |

## Candidate hypothesis

Keep all domain vocabulary, authority routing, core principles, dependency knowledge,
version/state-integrity guidance, and design-it-twice behavior. Simplify only the
entrypoint by removing diagrams, generic examples, relationship restatement, and
the duplicate dependency taxonomy, with explicit reference-loading triggers so the
detailed behavior remains available when relevant.
