Problem: An author opening a clean existing Private Note sees `Retry Save` beside `Saved`, so the action implies failure while the status asserts success. A successful edit preserves the same contradiction and forces a close-and-preview check even though persistence succeeded.
Product rule: The manual note-save action must use normal save language in non-failed states and reserve `Retry Save` for a known failed attempt; the visible status and accessible action name must describe one coherent state without changing autosave, identity, or local-buffer behavior.

Acceptance:

- A clean existing note and a successfully saved note show `Saved` without any visible or accessible action named `Retry Save`.
- A normal manual save opportunity uses `Save note` for a new titled note or `Save changes` for an existing note, and clicking it preserves the current no-duplicate and latest-draft semantics.
- A failed create or update visibly reports failure, retains every local field, and exposes exactly one `Retry Save` action that retries through the existing persistence path.
