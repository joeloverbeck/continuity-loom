## Reassessment Verdict

The audit's central diagnosis is valid: Continuity Loom has demonstrated strong constrained rendering, but its post-acceptance accounting loop remains clerical, difficult to trust, and insufficiently exercised through the real provider path. Repeated false no-ops are meaningful instance evidence, although they are not a measured population failure rate and were not produced through the configured in-app provider flow.

The audit's complete staged Consequence Review should not be adopted as written. Its natural-language-first idea is the right redesign seam, but automatic dependency retrieval, persisted before-state copies, structured mutation mapping, editor prefill or application, and future-possibility generation either exceed the evidence or conflict with current authority.

Published PRD [#132](https://github.com/joeloverbeck/continuity-loom/issues/132), **Accepted-Segment Change Review — Readable, Scoped Post-Acceptance Accounting**, owns the selected product direction. It replaces the formal-mutation-first response with a shallow typed envelope of plain-English change statements, complete coverage accounting, and explicit epistemic and retention classifications. It retains the latest-segment boundary, the existing nineteen Current Authoritative State and Immediate Handoff fields, and explicit Active Working Set or Whole Project scope.

Independent bug [#131](https://github.com/joeloverbeck/continuity-loom/issues/131) now owns the defect that must close before the PRD's real-provider comparison: the app currently discards actionable provider failure details. The UI's request catch replaces thrown errors with a generic sentence, and server-side OpenRouter normalization also collapses provider detail into broad categories. The actual provider rejection remains unknown until a safe underlying error is surfaced or captured.

Supporting skill result: domain model unchanged. **Accepted-Segment Change Review** is a workflow name, not a glossary concept, and PRD #132 plus the replacement prompt authorities own the architectural decision rather than duplicating it in an ADR.

External research: a narrow delegated diagnosis checked the [official OpenRouter model endpoint](https://openrouter.ai/api/v1/model/anthropic/claude-sonnet-5) and [Structured Outputs documentation](https://openrouter.ai/docs/guides/features/structured-outputs). ## Published First PRD Contract

### Accepted-Segment Change Review — Readable, Scoped Post-Acceptance Accounting

**Purpose:** Make the transfer from accepted prose back to author-owned story state readable, reviewable, and materially less clerical without allowing assistance output to become canon.

**Sources:** The complete latest accepted segment; saved-draft Current Authoritative State and Immediate Handoff fields; every complete non-archived record in one explicit Active Working Set or Whole Project scope; deterministic evidence keys, record and field contrast keys, source fingerprint, and version metadata.

**Problem:** The current feature asks one model response to comprehend accepted prose, choose retention, and express exact schema-valid record mutations. It can return structurally valid but substantively empty arrays, imposes mutation-oriented schema burden even though the UI cannot apply proposals, and makes provider or parser failures unnecessarily difficult to diagnose. The author must still scan and re-author changes manually.

**Recommended product rule:** Replace formal operation proposals with one shallow typed envelope whose content is plain English. Each change-review item contains:

- a concise English change statement;
- accepted-segment evidence keys and current record or field contrast keys;
- epistemic status distinguishing established change from interpretation requiring author judgment;
- retention horizon: durable record candidate, next-brief-only, no storage, or author decision required;
- affected-record or Generation Brief hints when identifiable;
- uncertainty or a rival reading when material.

The response also contains complete coverage for six human-facing dimensions:

1. spatial, material, and bodily state;
2. time, clocks, and ongoing processes;
3. facts, knowledge, beliefs, and secrets;
4. intentions, plans, commitments, promises, and open pressures;
5. emotions and relationships;
6. immediate next-segment handoff.

Each dimension is marked `changes found`, `checked - no relevant change`, or `uncertain`, with a short reason. A zero-item result is structurally complete only when every dimension is accounted for, but it remains unverified advisory assistance rather than proof that canon is current.

**Epistemic boundary:** A strongly caused but unstated present implication may appear only as an interpretation requiring author judgment. A future action or development is not a change-review item. For example, a segment-established continuing bell sound may be a next-brief environmental change; a character's likely distress may be an interpretive `EMOTION` candidate; a future confrontation is outside this workflow.

**Scope:**

- Keep only Active Working Set and Whole Project record scopes.
- Make Active Working Set the default.
- Show record counts, type counts, accepted-segment identity, and SECRET inclusion before sending.
- Include every complete non-archived record in the chosen scope.
- Fail visibly on oversize input; never retrieve, rank, summarize, trim, or expand scope automatically.
- Keep LLM review targets to the existing nineteen Current Authoritative State and Immediate Handoff fields.
- Provide a separate deterministic review-and-reset control for consumed segment-specific guidance. Its fixed allowlist is nonblank `manual_moment_directive.must_render[]`, `manual_moment_directive.may_render_if_naturally_caused[]`, `manual_moment_directive.do_not_force[]`, populated `current_cast_voice_pressure[]` and `cast_voice_overrides[]` entries, `generation_validation_focus.validation_focus_tags.expected_local_modes[]`, `generation_validation_focus.validation_focus_tags.possible_durable_changes[]`, and `stop_guidance.soft_unit_guidance`.
- The reset control never lists `generation_context`, `active_working_set` or its selected POV, `current_authoritative_state`, or `immediate_handoff`. It preselects nothing, infers no field was consumed from accepted prose, and performs no write. The author chooses exact fields or entries, reviews the resulting Generation Brief draft, and uses the existing explicit Save action as the sole canonical mutation gate.

**Comparison semantics:** The operation asks what remains unreconciled now. It compares the latest accepted segment with current saved state and current chosen record scope. It does not persist a second pre-generation snapshot. Any source change invalidates the ephemeral result and requires a fresh review.

**Interaction:** Render readable cards. Cards may be kept for the browser session, copied, marked reviewed, or used to navigate to cited records or the Generation Brief. Review state is session-only, performs no project-store write, and never acknowledges the durable-change reminder. There is no inline rewriting, split/combine workflow, editor prefill, apply, patch, create, deactivate, remove, merge, batch acceptance, or prose-generation action.

**Failure and recovery:** The server attaches trusted request and fingerprint metadata rather than asking the model to echo it. A malformed response remains visible as quarantined raw text with a precise reason. Transport, provider rejection, structured-response/schema rejection, parser quarantine, and source-freshness failure are distinguishable. Safe actionable detail must survive from server to UI; secrets and full prompt or record payloads must not enter logs or error text.

**Retention guidance:** Display static help that distinguishes meaningful historical occurrences (`EVENT`), independently reusable propositions (`FACT`), typed current truth or pressure, immediate next-segment handoff, and incidental texture requiring no storage. The model suggests; the author decides.

**Acceptance:**

- Identical declared inputs compile to identical prompt bytes and fingerprints.
- Source inspection proves exactly one latest accepted segment, the nineteen declared brief fields, and every complete qualifying record are present without hidden omission.
- Change-review output contains only the shallow English-item and coverage contract; no record-creation schema catalog, JSON Patch, full proposed record payload, lifecycle operation, or model-echoed fingerprint remains.
- The new authority, UI, routes, source profile, contract identity, and implementation symbols use Accepted-Segment Change Review terminology; the superseded active identities and routes are absent.
- Established changes, interpretations, and future possibilities are visibly and mechanically distinct.
- No invented or merely inferred item can be classified as an established change.
- Every response accounts for all six coverage dimensions.
- All-empty output remains visibly unverified and causes no automatic retry or write.
- Malformed, stale, and provider-failed responses show distinct safe reasons and recovery actions.
- Session keep/review state, clear, and navigation perform no project-store or prompt-context write.
- Consumed-guidance reset requires explicit field selection and explicit save.
- Gold fixtures cover high-impact object, location, bodily-status, clock, epistemic, commitment, emotion, relationship, lifecycle, and brief-only changes, plus a genuine no-change segment.
- Seeded death, injury, location, custody, clock-threshold, commitment, and secret-disclosure changes are either found or explicitly classified as uncertain; none may disappear behind an unreasoned no-change result.
- A bounded real-provider comparison records recall, precision, retention classification, invented changes, empty/malformed frequency, category coverage, review time, fields touched, prompt size, latency, and cost for the old and proposed contracts.
- Removal requires an explicit steward go/no-go receipt based on that comparison and the deterministic hard floors. The audit's provisional percentages are not pass/fail requirements.
- Until that receipt exists, old and proposed implementations may coexist only in a non-user-facing comparison harness. A passing receipt removes the old workflow rather than maintaining a public toggle or permanent compatibility fork.
- Root lint, typecheck, tests, and build pass, together with focused core, server, web, browser, and privacy/source-boundary checks.

**Potential implementation slices after PRD triage:**

1. Ratify and update the constitutional and prompt-contract authority family.
2. Replace the core, server, and API identity with the readable prompt, shallow response contract, trusted metadata wrapper, and deterministic quarantine/freshness behavior.
3. Render accessible cards, coverage, session checklist controls, navigation, and the deterministic consumed-guidance reset.
4. Replay gold fixtures and the bounded real-provider comparison, record the steward decision, and remove the superseded formal workflow if the gate passes.

## Independent Issue And Routed Or Deferred Candidates

### Preserve Sanitized OpenRouter Failure Detail Across Every Consumer — published as #131

**Purpose:** Preserve safe, actionable OpenRouter failure information through the shared server/client boundary and present consistent recovery guidance everywhere the app uses that boundary.

**Problem:** Server normalization reduces upstream failures to a category and generic message, then individual web consumers repeat partial category mappings or replace thrown request detail with generic text. The confirmed information-loss defect affects model refresh, Generate, Ideate, Story Record Hygiene, and the current Segment Reconciliation surface; the underlying real rejection that triggered this investigation remains unknown.

**Recommended rule:** Extend the shared transport failure with a stable category, provider HTTP status when present, bounded sanitized reason when safely extractable, retry timing when present, and category-specific recovery guidance. Sanitize through an explicit bounded policy rather than forwarding arbitrary upstream JSON or text. Distinguish transport, authentication, credits, rate limit, timeout, model/provider rejection, request or structured-response rejection, moderation, malformed response, and unknown failure. Workflow-local freshness and parser quarantine remain distinct local failures. Never expose API keys, authorization material, full prompts, candidate or accepted prose, record payloads, or unsanitized upstream text in UI messages or logs.

**Acceptance:** Server tests prove category/status/retry preservation, bounded reason sanitization, secret and payload redaction, and safe fallback when no reason can be retained. Web tests drive every existing consumer through the shared typed failures and assert a consistent accessible alert plus appropriate recovery action; generic request catches must preserve the safe contract. No consumer maintains an incompatible private category map. Logs remain free of credentials, prompts, prose, and record payloads. These deterministic criteria are sufficient to close the bug; a guarded live retry is a later Change Review comparison prerequisite, not a required external-network test for this issue.
