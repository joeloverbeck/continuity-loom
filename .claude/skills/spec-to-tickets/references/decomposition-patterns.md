# Decomposition Patterns (Step 3)

Analyze the spec and identify discrete work units:

- Each ticket is a **reviewable diff** — small enough for comfortable manual review.
- Map **dependencies** between tickets and name them in each ticket's `Deps`.
- Determine **priority ordering** from the dependency graph and criticality.
- Ensure **every spec deliverable is covered** — no silent skipping. If a deliverable seems wrong, flag it with the 1-problem / 3-options / 1-recommendation format rather than omitting it.
- Use the spec's **Out of Scope** section to populate each ticket's Out of Scope — these are pre-validated non-goals.
- **Anchor the enumeration** — when the spec has no numbered `§Deliverables`, seed the work units from a *per-deliverable* `§Implementation sequencing guidance` / `§Sequencing` ordered list if present (its steps are pre-ordered tickets) — but when that list is organized by *layer* (schema → migration → validation → compiler → UI → docs) across several deliverables rather than by deliverable, do NOT seed one ticket per phase (a per-phase ticket leaves a broken intermediate; see §Schema-field-removal vertical below): anchor on the deliverable axis and treat the phase plan as ordering input only. Else seed from `§Scope` in-scope items or distinct named sections; cross-check coverage against `§Acceptance` / `§Coverage` / `§Verification` so no testable rule goes unticketed (see SKILL.md Step 1 §Non-numbered deliverables for the precedence order).

## Deliverable-coverage categories

Most deliverables get their own ticket. These categories are exempt from per-deliverable ticket production — document them in the Step 4 summary and Step 6 coverage mapping instead:

- **No-change deliverable** — the spec explicitly states no change is needed ("No new skill"). Note it if non-obvious; produce no ticket.
- **Cross-spec consumer-side deliverable** — the work is consumed by a sibling spec's tickets rather than producing a discrete output here. The change is needed, but it happens in another spec's decomposition. Note the ownership as `cross-spec: <sibling-spec-id>` in Step 4 and `not ticketed (cross-spec: …)` in Step 6.
- **Cross-spec producer-side deliverable** — work *produced* here (a schema, normalizer, or type) but *consumed* by a sibling spec's tickets, with no consumer in this spec. Unlike the consumer-side category, this DOES get its own ticket + unit tests here; it is **excluded from the in-spec capstone leaf set** (its end-to-end consumer lives downstream, so there is nothing in this spec to exercise it against). Note it `produced here; consumed by <sibling-spec-id>` in Step 4 and Step 6. When an external ordering/sequencing doc marks a deliverable as shared across specs, the earlier-decomposed spec owns the ticket.
- **Distributed deliverable** — one conceptual deliverable encoded as a contract distributed across multiple tickets (e.g. a per-mode applicability rule carried as a predicate on each affected module, with no single owner). Note it as `distributed across <ticket-list>`; produce no separate "container" ticket. This also covers a **co-located supporting asset** — a deliverable that is not a contract but a shared resource each feature ticket extends in place, such as a styling deliverable whose CSS lands with the UI tickets it serves, or a shared test-helper added by each test file that needs it. Note it the same way (`distributed across <ticket-list>`), produce no separate asset ticket, and surface the shared file under Step 6 §Shared-file overlaps when ≥2 mutually-independent tickets each touch it. **Parallel-branch exception**: when the consumers are mutually-independent tickets on *parallel branches with no natural ordering* (e.g. per-pillar test tickets, each `Deps` only a common foundation ticket), prefer **per-ticket self-contained helper files** over one shared file. A single shared helper would force every consumer that extends it to `Deps` the ticket that creates it (per §Intra-batch create-then-modify chains), coupling otherwise-independent branches and serializing work that should run in parallel — a *conceptually false* dependency (the consumer needs the helper file, not the creator's feature logic). Reserve the single-shared-file + §Shared-file-overlaps route for consumers that already share an ordering (the aggregation-point registry case below, where the `Deps` on the creator is real).
- **Informational-pointer deliverable** — a section whose body only points to sibling deliverables ("handled in §X above and §Y below"). The actual work is in the pointed-to siblings, each ticketed. Note the routing; produce no ticket.
- **Already-landed deliverable** — the intent was satisfied by an edit landed during a prior in-session skill invocation (e.g. `/reassess-spec`'s user-authorized in-session edit). Produce no ticket; record `not ticketed (already landed during <skill> session <date>)` in Step 6.
- **Inferred deliverable** — named or implied by §Approach / §Verification / §Definition of Done but absent from §Deliverables. This DOES get a ticket; document the inference in the ticket's Assumption Reassessment with a citation to the motivating section, and flag it in Step 4 Notes as `inferred from <section>`. When the motivating reference is ambiguous or would expand scope, route through a Step 2 Issue instead. If the inferred deliverable touches files or surfaces another spec in the set owns (per Step 2 spot-check (c) / `references/codebase-validation.md` §Sibling-spec surface ownership), treat it as `cross-spec: <sibling>` and do not ticket it here.
- **Completion-bookkeeping deliverable** — a deliverable that performs end-of-phase bookkeeping rather than producing product code: flipping a status line in a sequencing/ordering doc, adding a "Phase-N implementation note" to governing docs, and archiving the spec (`docs/archival-workflow.md` → `archive/specs/`). Route it into the trailing capstone / cross-cutting-docs ticket rather than producing its own ticket; its acceptance is grep-proofs on the status/note lines plus `test -f` / `test ! -f` on the archival boundary. Recurs in Continuity Loom as the spec's "governing-doc updates on completion" deliverable.

## Sizing rules

- **Merge** — when multiple deliverables share the same file set and cannot be implemented independently, merge into one ticket. Note merged deliverables in Step 4 Notes. This also covers the **removal-breaks-consumer indivisible case**: when removing a field/symbol breaks an existing consumer whose only shim-free fix is a coupled feature or validation change (e.g. removing a schema field a validation rule reads, where the rule's correct rewrite needs a paired delivery fix), the removal + consumer fix + coupled feature are one indivisible ticket even when large — "split when in doubt" is overridden because no compliant intermediate exists (a split would leave a typecheck-failing or regressed state, and CLAUDE.md forbids the shim that would bridge it). Cite the indivisibility rationale in the ticket's Assumption Reassessment.
- **Split** — when a single deliverable is too large for one reviewable diff (it spans, say, a contract section + a schema + a validation rule + a skill emitter), split by logical sub-concern (e.g. write path vs. read/verify path). Note the split in Step 4 and show `D<n>→00x+00y` in Step 6.
- **Schema-field-removal vertical** — removing a stored schema field is one *vertical* ticket (or a strict same-revision set), never a per-layer split: the strict-schema tightening co-lands its server migration that strips the key *before* strict parse (else opening a legacy project breaks — the SPEC-024 failure mode the spec's risk section will cite), AND all typecheck consumers across packages (`@loom/core` + `@loom/server` + `@loom/web`) co-land in the same diff, because removing a type member in one package breaks consumers in the others under strict TS (`npm run typecheck`, per CLAUDE.md). A core-only removal ticket, or a schema-tighten ticket without its migration, leaves a broken intermediate; CLAUDE.md forbids the shim that would bridge it. Three same-storage-domain removals (e.g. several generation-session fields) are therefore *sequenced* (each `Deps:` the prior) to serialize the shared `*.ts` + migration edits rather than split by layer.
- **Same file / same new directory** — when all deliverables modify the same file or land under one new directory, decompose by logical section/module, not by file boundary. Each ticket targeting a different section is a valid reviewable diff. Note the shared file once in Step 4.
- **Aggregation-point fan-out** — when several contributor tickets each register into ONE shared aggregation file (a catalog, a validator registry, a barrel/index — e.g. N guidance-content tickets each adding entries to one registry, or N validators each inserted into one framework array), do NOT have each ticket write the bulk of its content into the shared file. Instead, give each contributor its **own area module file** (`(new)`) and have it touch the shared registry **only at a single import/spread (or array-insert) line**, each declaring `Deps:` on the registry-creating ticket (per §Intra-batch create-then-modify chains). This shrinks the unavoidable conflict among mutually-independent contributors to one line apiece; surface that registry import/insert block under §Shared-file overlaps (Step 6 item 5) as the coordinated mechanical merge. Prefer this over routing all content through the shared file directly, which forces an N-way conflict over the whole body.
- **Intra-batch create-then-modify chains** — when ticket B's Files to Touch contains a `(modify)` entry for a file ticket A creates `(new)` in the same batch, declare `Deps: A` on B explicitly. The file-creation is a load-bearing structural dependency (B fails at implementation time if A hasn't landed). This is distinct from §Shared-file overlaps (Step 6 item 5), which covers parallel adds by mutually-independent tickets to a shared file — whether pre-existing or created in-batch by a common foundation ticket they each `Deps:` on.
- **Same-revision co-landing coupling (non-merge)** — when a deliverable must ship in the **same revision (commit/PR)** as a dependent ticket but warrants its own reviewable diff, keep it a separate ticket rather than merging. The governing case in Continuity Loom is a `docs/FOUNDATIONS.md` amendment: FOUNDATIONS §1.1 requires an amendment coupled to dependent code to *land in the same revision as that change, never standalone ahead of it*. Model it by (a) keeping the amendment its own ticket — its constitutional wording is a distinct sign-off/review surface; (b) declaring the dependent code ticket's `Deps:` on it; and (c) recording an explicit `same revision; never merge standalone (§1.1)` constraint in both tickets' Architecture Check and Acceptance. Distinguish from **Merge** (one diff, used when the pieces cannot be reviewed apart) and from ordinary **Deps** (before/after ordering where separate commits are fine): co-landing forbids the standalone-ahead merge an ordinary `Deps` would permit. **Multi-ticket feature**: when the amendment is depended on by a whole feature (many tickets) rather than one, do NOT repeat `Deps:` on the amendment in every ticket — declare it only on the feature's **root ticket(s)** (the earliest feature code, e.g. the core-schema and storage tickets), so the rest of the feature DAG inherits the amendment-before-feature ordering transitively. The `same revision; never merge standalone (§1.1)` constraint line still belongs in the amendment ticket and in each root that declares the `Deps:`.
- **Greenfield foundation batch** — when the spec is the repo's first code (no pre-existing code tree, e.g. a Phase-1 repository/runtime foundation), every ticket's Files to Touch is legitimately all-`(new)`; do not treat that as a suspicious under-statement. The structural ordering is carried by intra-batch create-then-modify chains (above), and the deliverables' consumers are the sequenced later phases, not present code — mirroring the greenfield note in `reassess-spec`'s `references/codebase-validation.md`.

## Ticket shapes

### Capstone integration ticket

A single trailing ticket whose scope IS the spec's §Verification section — or its equivalently-named carrier when the spec has no literal §Verification: §Testing, §Acceptance / §Acceptance Criteria, §Coverage rules, or §Definition of Done (reassess-prepared specs routinely name it one of these) — exercising every prior implementation ticket end-to-end. It introduces no new production logic; it exercises the pipeline the earlier tickets composed.

- Its acceptance criteria enumerate the spec's §Verification bullets (or the equivalent carrier's bullets, per the section above) as test sub-cases.
- Re-enumerate expected counts from a fixture at test start rather than hardcoding (hardcoded counts go stale).
- **Files to Touch** — `_TEMPLATE.md` makes the section mandatory, but a capstone introduces no production logic, so it is either the new e2e/smoke test file the ticket adds, or `None — verification-only` when the gate is exercised purely by running existing scripts plus a manual runbook. Do NOT list the upstream tickets' files — the capstone exercises them, it does not modify them.
- **Manual-runbook variant** (formerly "skill dry-run variant") — when a §Verification bullet cannot run in the project's test infra — either it requires invoking an LLM-driven skill (not runnable from test code), **or** it is a UI/manual smoke with no browser-automation harness in the project — structure the ticket as a hybrid: a *manual runbook* (in the What to Change section) the implementer follows by hand — for an LLM skill: fixture copy strategy, skill invocations against the copy, expected post-skill state with verification commands; for a UI smoke: the click-path with each step's expected observable result — plus an automated-test subsection for any test-runnable portion (grep negative tests, schema assertions, server-side e2e). The acceptance criteria distinguish CI-runnable assertions from implementer-checklist runbook steps.
- **`Deps`** — prefer a single `Deps: <transitive-head-ticket>` when the upstream DAG is a linear chain (the head's own `Deps` reach every prior ticket). When the upstream DAG has parallel branches (e.g. a docs ticket running parallel to implementation tickets), enumerate the **leaf set** — every leaf whose transitive `Deps` collectively cover the full chain. A **cross-spec producer-side** ticket (produced here, consumed downstream — see §Deliverable-coverage categories) is legitimately excluded from this leaf set: it has no in-spec consumer to exercise, so it is unit-tested in its own ticket rather than gated by the capstone. A parallel **docs-only leaf** (a cross-cutting docs ticket running alongside implementation) is likewise excluded — the capstone gates *behavior*, which the docs ticket does not carry; let the docs ticket land in parallel and gate the capstone on the implementation leaf set only. **Hybrid capstone (manual-runbook variant)**: when the capstone's automated assertions are implementable against an *earlier* ticket (e.g. a server-side non-leakage sentinel needing only the routes) while its manual runbook needs the *full* feature (e.g. the UI), gate the whole capstone on the leaf set covering **both** — i.e. the full feature, since the capstone proves the complete feature is correct. Do NOT split the capstone, or pull its `Deps` back to the earlier ticket, merely because the CI-runnable portion is ready sooner.

### Cross-cutting docs ticket

A single trailing ticket whose Files to Touch are markdown docs (READMEs, architectural docs, skill reference files) that must land atomically once all upstream implementation tickets ship.

- One Files-to-Touch entry per docs surface the deliverable names.
- Acceptance criteria as grep-proofs against the post-implementation tree (exact-string matches for new/renamed symbols, count assertions for status-line updates).
- Out of Scope names the implementation surfaces this ticket does NOT touch, so reviewers know docs-only is the contract.
- **`Deps`** — list every implementation ticket whose surface the docs reference (docs reference each surface independently, so transitive-head is insufficient here). **Exception** — when the docs ticket only annotates *aggregate completion* (a status line / "verified by SPEC-NNN" note) rather than citing individual symbols, and a capstone ticket already gates every referenced surface, `Deps: <capstone>` is sufficient and more legible than re-enumerating the surface set. Enumerate per-surface only when the docs cite individual symbols/APIs that could go stale independently.
- **When NOT to use** — if each docs change can co-locate with its implementing ticket without a staleness window, per-ticket docs are cleaner. Use this shape when ≥1 docs surface requires all upstream surfaces to exist coherently before it can land (status-line counts, cross-tool tables, multi-surface skill guidance).
- **§8 same-change docs are mandatory co-locations, not preferences** — a contract/authority doc that a FOUNDATIONS same-change rule binds to its code MUST co-land in the implementing ticket's Files to Touch, never the trailing docs ticket: `docs/compiler-contract.md` for any compiler / placeholder / section-order change (§8 — drift between template, schema, and contract is a continuity bug), and `docs/story-record-schema.md` for a record-schema change. Route it into the code ticket even when the spec lists it under a docs deliverable; only the *non*-§8-bound docs surfaces stay in the trailing docs ticket: `docs/ACTIVE-DOCS.md` registry/version notes, the user guide, and a *non-template* new authority doc (a process/registry/guide doc). **A new *prompt-template* / compiler-contract authority doc — one the compiler renders (e.g. `docs/story-record-hygiene-prompt-template.md`, a sibling of `docs/prompt-template.md` / `docs/ideation-prompt-template.md`) — is itself §8-bound**: it is the "template" whose drift from the compiler is a continuity bug, so co-land it with its compiler ticket, together with its `docs/ACTIVE-DOCS.md` registry row, so the registry never points at a doc authored in a later trailing ticket.

### Merging the trailing shapes

The capstone and the cross-cutting docs ticket MAY merge into a single trailing ticket when a phase's §Verification is a manual smoke (no automation harness — the capstone is the manual-runbook variant) AND its completion-bookkeeping / docs updates are gated on that smoke passing. Both depend on the same upstream leaf set and the docs/status flip is gated on the smoke, so splitting them only adds a needless inter-ticket dependency; the merged ticket's Files to Touch are the docs surfaces (plus any archival move) and its What to Change carries the manual runbook. Keep them separate when the docs cite individually-staleable symbols that each need their own per-surface `Deps`, or when there is substantial CI-runnable verification distinct from the docs change (then the capstone owns a real test file and the docs ticket stays docs-only).
