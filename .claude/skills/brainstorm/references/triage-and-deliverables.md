# Triage & Deliverables

Detailed rules for two SKILL.md branches: the triage recommendation (Step 3 §Triage brainstorms) and deliverable classification (Step 5). Also collects the Step 4 design-presentation carve-outs.

---

## Triage recommendation structure

Used when the brainstorm evaluates a report, finding-set, or diagnostic question and produces work items, instead of proposing approaches.

### Per-item structure

Each triage item has:

1. **verdict** — one of the types below.
2. **rationale** — 1-2 sentences naming the FOUNDATIONS / codebase / contract grounds.
3. a **conditional sub-field** — `modification scope` (for accept-with-modification) / `alternative path` (for reject) / `deferred_to` (for defer) / `verification source` (for refuted-by-verification). Absent for accept / already-resolved.

### Verdict types

| Verdict | Sub-field | Use when |
|---|---|---|
| `accept` | none | item warrants action as recommended |
| `accept-with-modification` | `modification scope` | item warrants action with refinements (scope-narrowed, severity-shifted, technique-substituted) |
| `reject` | `alternative path` | item declined with no positive scheduling intent; pair with what to do instead (or "none") |
| `defer` | `deferred_to` | item judged sound but routed to a follow-up deliverable; names the follow-up shape + the trigger condition for re-evaluation |
| `already-resolved` | none | re-triage case: the item was actioned between the original pass and this one; cite the resolving artifact + date |
| `refuted-by-verification` | `verification source` | the item's claimed gap or premise is disproved by codebase/contract verification at triage time; quote the file:line evidence |

The seven-bucket vocabulary is closed — don't coin new verdicts. A user-elected skip ("skip the polish for now") is a `defer` whose trigger is the user's batch-scoping choice. An item whose premise is refuted but carries a valid residual best folded into another finding uses the dominant verdict plus a rationale cross-reference to the absorbing item's ID.

### Per-item identifiers

Derive from the source report's own numbering when present (`P1`, `R10`, `F-01`). When unnumbered, use `R<N>` for source-report items. Use `O<N>` for **out-of-report** findings (auditor-discovered, no presence in the source report) — always the literal `O<N>` prefix so they're unambiguous in cross-references. IDs must be stable so the user can reference them by number. When a prior triage on the same surface already used `O<N>` keys, continue the numbering (start at the next free integer, e.g. `O5`) or round-prefix (`R3-O1`) so cross-round cross-references stay unambiguous.

### Grouping & out-of-report findings

Group items by verdict bucket so the user can scan by shape (all accept together, all reject together, etc.). In cross-references and the verdict field itself, use the canonical lowercase-hyphenated form (`accept-with-modification`).

Findings discovered during exploration that are NOT in the source report (adjacent pre-existing bugs, schema drift) go in a separate **out-of-report findings** sub-section AFTER the verdict buckets, keyed `O<N>` — not a new verdict bucket. But a correction that refutes a *source-report item's* premise HAS presence in the report, so it's a `refuted-by-verification` (or `already-resolved`) verdict keyed `R<N>` in the buckets. Corrections that reframe the whole triage (tied to no single item) go in the triage lead or a verification headline before the buckets.

### No-source-report diagnostic case

When the request is a diagnostic question or exploration prompt rather than a formal report ("figure out why X", "what's happening with Z"), there's no source report to evaluate verdicts against. **Before structuring the presentation, read the §Worked skeleton (no-source-report) below and reuse its headings verbatim — `## Findings (out-of-report)`, `## Recommended fix`, `## Deliverable-shape recommendation`, `## FOUNDATIONS alignment`, `## Named assumptions`; the bullets below fill those sections, they don't replace the skeleton.** The handling:

- **Check for a prior triage of this surface first.** A no-source-report conformance/diagnostic *re-audit* is the common case where an earlier `triage/` record already exists; glob/grep `triage/` (and existing `tickets/`/`specs/`) for the same spec or surface **before** forming findings, and frame the pass as a delta — don't re-propose a deliberately-deferred or already-rejected item as fresh. This discovery is also the precondition for the §User-pre-authorization reversal-check (surface a prior-decision reversal before the scope `AskUserQuestion`). **User-excluded-triage carve-out:** when the user explicitly scopes out the triage folder (e.g. "find solutions aligned with `docs/**` without the triage folder"), this mandatory prior-triage discovery yields to that directive — state in one line that the `triage/` discovery was skipped per the user's exclusion and proceed; do not silently skip it, and do not override the user to run the glob anyway. (The exclusion governs the *discovery* step only — a `triage/` companion record is still written at the deliverable threshold per §Deliverable classification, since that's an output, not a read of the excluded folder.)
- **For a runtime/UI-defect diagnostic, reproduce before diagnosing.** **Pick the branch by where the defect lives (the layer that must change), not where the symptom surfaces:** a UI-presented symptom whose root cause is a pure `@loom/core` function — a descriptor, the compiler, a validator, a normalizer — follows the deterministic-compiler / core-logic branch below (including its static-root-cause skip), not this browser-repro branch. Drive the defect with the available browser-automation / CLI tooling (Puppeteer/Playwright MCP, `curl`, devtools resources such as `console://logs`) and capture concrete evidence — console output, the failing request(s) + status code, repro steps — *before* forming `O<N>` findings. Map each symptom to a specific request/root cause rather than relaying raw console text (e.g. when the console omits URLs, correlate via resource timing and probe each endpoint's status). When the user names a specific repro tool or path (e.g. "open the project, then Settings" via Puppeteer/Playwright), confirm the symptom at that layer at least once before substituting a cheaper proxy (`curl`, a code trace) for the remaining investigation — a proxy can diverge from what the user actually sees. For a UI-*rendering* symptom specifically, in-browser API calls (`puppeteer_evaluate` running `fetch`) are still a backend proxy — same-origin against the real server, but they prove backend state, not what renders; capture one DOM/visual confirmation (the rendered element, or a `screenshot`) before substituting API/code evidence for the rest. When several routes share **one confirmed backend root cause** (e.g. several pages all failing on the same 500), one backend confirmation plus **one** DOM/visual confirmation across the set suffices — you needn't render-confirm every page — but capture at least one rendered-state confirmation, never zero, so a client-side-only error specific to one page isn't missed behind the shared backend failure. **Reproduce read-only by default:** prefer non-mutating probes (GET endpoints, read-only DB opens, code traces). Before any reproduction step that could mutate user data — especially against a *production* project the user named — confirm the action is read-only, or operate on a disposable copy. Beware endpoints whose mere invocation has side effects (e.g. a project-open path that runs migrations, as Continuity Loom's does). **Reconcile with the named-path rule:** when the user names a *production* project as the repro path *and* opening it mutates (Continuity Loom's open runs migrations), copy the project to a disposable location and drive the copy — that still confirms the symptom "at that layer" — or, when a copy is impractical, surface in one line that the repro will mutate the named project and proceed only on that basis. **For a client-routed SPA:** drive in-app navigation (or set app state) rather than deep-linking to a route, which may render an empty DOM; a view downstream of a workflow (e.g. a prompt preview that needs a configured generation) may need its preconditions set before it shows evidence. **Regression-framed symptom:** when the user attributes the defect to recent work ("recent PRs broke X"), treat the named suspect as a hypothesis — run `git log`/`git blame -L <start>,<end>:<file>` on the symptom's render/data path to date the introducing change; the true cause may predate the suspect (a recent change may only have *surfaced* a long-standing defect, e.g. a field reclassification that made an old data-starvation bug visible), and `git status` catches only uncommitted confounds. Don't let the user's most-recent-PR framing anchor a wrong-target ticket. **Harness dies mid-flow:** if the live repro environment becomes unavailable partway through (the dev server crashes, HMR breaks, the browser session drops), capture whatever was already observed, complete the diagnosis via static/read-only means (read-only DB open, code trace, git history), and mark the visual confirmation *partial* in the post-exploration confidence anchor rather than blocking on a dead harness.
- **Mutation guard — apply before the first open of a user-named production project.** If the user names a *production* project as the repro path and opening it mutates on disk — Continuity Loom's project-open runs migrations (`packages/server/src/project-store.ts` `openProject` calls `migrateGlobalConfigRecords` / `migrateGenerationSessionDraft`, which write the database) — drive a disposable copy, **or** state in one line that the repro will mutate the named project and proceed only on that basis; never open it silently. **Tie-breaker:** when the user explicitly directs the exact UI flow against the named project (names the project *and* the open steps/tool, e.g. "open `red-bunny`, click Open Project, do this with Puppeteer"), prefer the surface-the-mutation-and-proceed branch — substituting a disposable copy would diverge from the instructed repro. Reserve the copy branch for when the repro path is not literally pinned; note that a copy preserves preconditions, so it still reproduces data-dependent symptoms (don't justify skipping the copy by claiming it wouldn't reproduce the precondition — it would). Once evidence is captured, delete the disposable copy and any scratch reproduction script you wrote into the repo (a one-off `tsx`/Node replay, a fixture) — create → use → delete, the same scratch-artifact hygiene the client-side bullet below applies to scratch tests; an in-repo scratch script left behind will surface in `git status`. (Broken out as its own bullet because the named-path reconciliation folded into the read-only sentence above is easy to miss inside that dense bullet.)
- **For a client-side-only component bug, the reproduction harness is a scratch test, not a server probe.** When the defect lives in client-side logic, state, or rendering (e.g. a React component) with no server in play, the browser/CLI tooling above does not fit — author a **scratch unit/component test (Vitest/RTL)** as the reproduction harness, run it to capture evidence, then delete it once the evidence is captured (scratch-artifact hygiene: create → use → delete, since it mutates the working tree). **Precedence when the app is already running:** if the symptom is DOM-, keystroke-, or focus-native and a real browser is available (Puppeteer/Playwright), drive the real browser instead — it is higher-fidelity than the scratch test and sidesteps the jsdom false-pass described next. Reserve the scratch test for pure client-side logic/state bugs, or when no running app/browser is in play. **Mind jsdom's fidelity gaps:** jsdom does not emulate native DOM behaviors such as radio-group exclusivity, focus, or layout, so a jsdom repro can confirm the *root cause* yet false-pass on the *native symptom* (e.g. a controlled radio's `.checked` survives in jsdom even when colliding `name`s break it in a real browser). A second false-pass class is *event granularity*: RTL's `fireEvent.change` delivers the whole value in a single synthetic event, so it cannot reproduce an *incremental-keystroke* symptom — to trip a per-keystroke normalization round-trip (e.g. an `onChange` that trims/filters each edit), assert on a **trailing-whitespace** value (`"door closing "`), not an internal-space one (`"door closing"`), which false-passes because `trim()` leaves it unchanged. When the symptom is DOM-native, either drive a real browser (Puppeteer/Playwright) or assert on the observable proxy (the offending attribute itself) plus a code trace — and carry that proxy-vs-native distinction into the deliverable's regression test so it asserts on the proxy rather than the symptom jsdom can't reproduce.
- **For a CI/build/lint/typecheck/test-failure diagnostic, reproduce by running the failing command locally.** When the trigger is a failing CI job (e.g. "CI / lint is failing") rather than a runtime defect, the reproduction harness is the project's own script: run the exact failing step locally (`npm run lint`, `npm run typecheck`, `npm test`, `tsc -p …`) and capture the failing output **before** forming `O<N>` findings. Prefer the project's real scripts over an approximate filter so the local run matches CI's gate, and diff the local result against the CI log to confirm you reproduced the same failure rather than a neighbouring one. This is distinct from the non-reproducible data/schema-shape case further below — a CI command failure *is* reproducible, so don't diagnose it from the CI log alone.
- **For a deterministic-compiler / core-logic output defect, the reproduction harness is a `@loom/core` unit or golden test.** When the defect is wrong *output* from pure deterministic logic (the prompt compiler, a validation rule, a normalizer) rather than a runtime/UI symptom or a CI-command failure, it IS reproducible: run the pure function (`compilePrompt`, `runValidation`, the normalizer) in a `@loom/core` unit test, or point at the existing golden/snapshot fixture — a frozen golden baseline that renders the wrong value (e.g. a POV UUID where a character name belongs) is itself the reproduction. Capture that evidence **before** forming `O<N>` findings; it is higher-fidelity than a browser repro and needs no server. **When the question is whether a real stored artifact satisfies a schema or pure function**, source the input from the real persisted data (a read-only DB open) and replay it through the exported function — higher-fidelity than a synthetic fixture for data-shape defects, since it tests the actual on-disk state, not a constructed approximation; delete the scratch script afterward (create → use → delete). Reserve the "non-reproducible" treatment in the next bullet for genuinely undrivable data/schema-shape questions where there is no function to run.
  - **Static-root-cause skip (parallels SKILL.md sub-step 6's research skip):** when static reading pins an unambiguous `file:line` root cause (e.g. the resolver that emits the raw id vs. the sibling resolver that resolves it to a display label), the confirmatory unit/golden repro is optional. This holds in **two** cases: (a) the user supplied the verbatim wrong output; or (b) the root cause is pure deterministic `@loom/core` code (a descriptor/compiler/validator/normalizer) that static reading pins directly from the schema + code, **even when the user only *described* the symptom** — there the schema + code are the evidence, so verify the symptom premise against them in lieu of pasted output. **Note the skip explicitly** — e.g. in the post-exploration confidence anchor ("root cause pinned statically from the descriptor schema; confirmatory golden/unit repro skipped"). The reproduce-before-diagnosing rule exists to correct *mis-stated* symptom premises; a user-pasted verbatim output — or a symptom premise checked directly against the pure-logic code — has already captured the symptom, so a separate repro is redundant. Still write/update the golden or unit test when *implementing* the fix (the ticket's test plan), not necessarily at diagnosis time.
- **Distrust a sub-agent map that contradicts the user's first-hand symptom.** When an exploration sub-agent's structural claim about the code (a field is excluded, a code path is dead, X does not exist) contradicts the user's reported first-hand symptom, the symptom wins as the hypothesis — confirm it by direct Read/grep (or reproduction) before letting the sub-agent's map anchor the root cause. Sub-agent code-maps can be confidently worded yet wrong; the reproduce-before-diagnosing rule above is what catches this for UI defects, but the same distrust applies to non-reproducible (data/schema-shape) diagnostics where there is no function to run.
- **Omit the verdict-bucket section entirely** — the verdicts are defined against source-report items.
- **Route all findings to the out-of-report sub-section**, keyed `O<N>`. The answers to the user's questions ARE the findings — emit them as `O1`, `O2`, …, not as synthetic `R<N>` items restating the questions. When the diagnosis is a *single* root cause backed by an evidence chain, key it as one `O<N>` finding (the root cause) with nested evidence bullets — reserve multiple `O<N>` for genuinely distinct findings, so `O<N>` always means "finding," not "evidence line." **A finding may resolve to no deliverable.** When the request bundles fixes with a standalone analysis question (the common "fix X and Y, and tell me whether Z" shape), the answer to that question is a first-class `O<N>` whose disposition is *no action* — an answer that refutes a user concern or confirms safe behavior. Key it `O<N>` like any finding and carry it in the finding→deliverable map with an explicit "no ticket — analysis answer" outcome, rather than dropping it or forcing a fix; the close shapes above are fix-centric, but a no-deliverable answer-finding is legitimate alongside them.
- **The closing structure still applies** — except that when the diagnostic is a single root cause resolving to a single pre-authorized inline action with no deliverable left to cross-reference, a condensed prose close (root-cause headline + recommended action + verification) may stand in for the full `O<N>` + worked-skeleton form. Reserve the full form for diagnostics that produce a persisted deliverable or whose findings get referenced later.
- **Companion `triage/` record shape (when ≥3 tickets / ≥2 specs trigger it).** The §Deliverable classification companion rule is written for source-report triage ("accepted items / dismissed items"); that framing does not fit a no-source-report diagnostic, which has no verdict buckets. Instead structure the companion as: the reproduction/evidence summary (repro steps, failing requests + status codes, captured console output); the `O<N>` findings, each with its recommended fix and the ticket/spec it landed in; the decisions taken (including any `AskUserQuestion` choices that shaped the deliverables); and the finding→deliverable map. Omit the "dismissed items" section unless a candidate fix was genuinely considered and rejected. The ≤80-line companion budget still applies. **Reuse the `O<N>` keys exactly as presented in the triage recommendation** — if a presented finding was folded or split during ticket creation, preserve its original key (note the fold inline, e.g. "`O2` covers both sections") rather than renumbering. The companion is a durable artifact the user references by number, so its keys must match the conversation they approved (per the stable-ID rule above); re-deriving the list a turn later is where keys silently drift.
- **Close-shape selector** (full logic in the bullets below):

  | Diagnosis shape | Close shape | Deliverable-count decision |
  |---|---|---|
  | One course of action weighed vs. alternatives | Recommended-action upfront + rejected alternatives | usually 1 (or inline action) |
  | Several genuinely distinct findings, each its own fix | Per-`O<N>` fix + finding→deliverable map | N, settled by the map — but confirm 1↔N via `AskUserQuestion` when the fixes share a surface (bundling viable) |
  | One root cause, one fix, remediation split across deliverables (e.g. fix-code + *separate* docs) | Single-action close | 1↔multi via `AskUserQuestion` |
  | Pure observations, no per-finding fix to weigh | Bare flat `O<N>` list | none / as needed |
  | Any of the above + an adjacent separable gap | Matching shape per group; one `AskUserQuestion` for the total | composes across groups |
- **When the diagnostic resolves to a recommended *action* (not a set of independent work items)** — i.e. the question is "what should we do about X?" and the answer is one course of action weighed against alternatives — the `O<N>` findings carry the *answer* (the diagnosis), and the close borrows Step 3's recommendation shape in place of (or alongside) the deliverable-shape recommendation: name the recommended action upfront, then the rejected alternatives with their grounds, then any optional add-ons. This is the sanctioned blend for action-shaped diagnostics; don't force an action choice into the flat `O<N>` finding list. **When the diagnostic instead resolves to a *set* of independent fixes** (several `O<N>` findings, each warranting its own work item), each `O<N>` may carry its own recommended fix + rejected alternatives, and the close is the deliverable-shape recommendation with a finding→deliverable map — distinct from both the single-action blend and the bare flat list (reserve the bare flat list for findings that are pure observations with no per-finding fix to weigh). **Don't auto-default the count to N when the independent fixes share a code surface/file** (same module, same golden, same test file) — bundling them into one ticket is then a real alternative, so the 1↔N count is a material-deliverable-shape decision: confirm it via `AskUserQuestion` (lead with the recommended option, marked "(Recommended)") per §Closing structure and SKILL.md §User pre-authorization, rather than silently writing N. **A third shape sits between these: a single root cause with one fix whose remediation spans multiple deliverables** (the common code-fix-plus-*separate*-docs case — note that a fix and its **co-located regression test** are normally one reviewable diff = one ticket, so the split refers to fix-code + separate docs, or fix + cross-cutting test-infra, not a fix and the unit test that guards it). Use the single-action recommendation shape for the diagnosis/close, and treat the deliverable split as a material-deliverable-shape decision settled via `AskUserQuestion` (the count crosses 1↔multi) per §Closing structure and Guardrails §User pre-authorization — don't mistake it for a set of independent fixes (the root cause and fix are one; only the deliverables divide). **When the candidate remedies are themselves materially-different deliverable *classes*** (e.g. docs-only vs. a FOUNDATIONS amendment vs. a code+UI change), pose the remedy choice itself via `AskUserQuestion` rather than relegating the rejected alternatives to prose only — the choice is material, not rhetorical, so the user picks it rather than reading why you picked it (still lead with your recommended remedy as the first option, marked "(Recommended)" — letting the user pick doesn't relieve you of recommending one). And when a downstream count/shape question only becomes meaningful *after* the remedy is chosen (under one remedy there may be no tickets at all — e.g. a FOUNDATIONS-amendment remedy yields no tickets), ask it in a **separate later round contingent on that answer**, not batched with the remedy choice: `AskUserQuestion` batching is for mutually independent questions, never for a question whose options or existence depend on a prior answer. Keep the single-root-cause `O<N>` granularity from the no-source-report bullet above even when the evidence is rich and the remediation splits: the root cause is **one** `O<N>` with nested evidence bullets, and additional `O<N>` are reserved for genuinely distinct secondary defects — not for evidence lines, the reproduced impact, the FOUNDATIONS basis, or corroboration, which all nest under the root-cause `O<N>` or fold into the recommendation. **These shapes compose.** A real diagnostic often has a multi-deliverable root cause *plus* one or more independent side findings (e.g. a benign-but-noisy symptom worth its own fix); key each independently as its own `O<N>`, apply the matching close shape per group, and let a single `AskUserQuestion` settle the total deliverable count across all groups. **Asymmetric-authorization exception:** when the primary finding is pre-authorized for immediate inline action ("if easy, fix it now") but exploration surfaces an *adjacent, separable* gap the pre-authorization does not cover, complete the authorized fix first and offer the adjacent deliverable separately — a prose offer or a fresh follow-up cycle — rather than bundling a pre-authorized "do-now" action and an optional "offer-later" gap into one `AskUserQuestion`. The single-`AskUserQuestion` rule above assumes all deliverables are decided at one point; it does not fit a do-now/offer-later split.
- **When the request is a conformance audit against an authority doc/contract** ("ensure the implementation matches `story-record-schema.md`", "verify X conforms to `compiler-contract.md`") — a frequent no-source-report shape — group the findings by *remediation route* (fix-code / amend-authority-doc / no-action-affirm) rather than by the report-oriented verdict types, since the natural disposition of each discrepancy is which side moves (code or doc). Keep the `O<N>` keying (the closed identifier convention still holds — don't coin an `F<N>` prefix), and include a brief affirming "what already conforms" note so the audit reads as coverage, not just a defect list. The remediation route is the per-item disposition; a "judgment-call" discrepancy (code-or-doc unclear) is surfaced as its own item with the recommendation and the rejected alternative. **Composes with the defect-diagnosis case (hybrid requests):** a request is often *both* a specific defect diagnosis and a conformance check ("fix the noise in X, and ensure X aligns with `docs/**` / is used as intended in all cases"). Pick the organizing axis: apply this bullet's remediation-route grouping only when conformance *is* the deliverable's organizing axis; when the deliverable is a specific bug fix, keep the `O<N>` defect findings and the set-of-fixes close above — but still add the affirming "what already conforms" coverage note (e.g. a one-line statement of the sibling fields/sections that already satisfy the contract) so the conformance dimension is answered as structured coverage, not just inline prose. The two cases compose; conformance is "a frequent no-source-report shape," not a mutually-exclusive branch.

#### Worked skeleton (no-source-report)

Parallels the source-report §Worked skeleton below. Use the `## Recommended fix` block for the action-shaped and single-fix-multiple-deliverable closes; for a set of independent fixes, drop that block and give each `O<N>` its own fix + rejected alternatives.

```markdown
## <Why X — root-cause headline>

## Findings (out-of-report)
- **O1** — <root cause>. _Evidence_: <file:line / repro output>.
  - <evidence bullet>            ← single root cause: nest evidence under one O<N>
  - <evidence bullet>
- **O2** — <distinct finding, only if genuinely separate>.   ← set-of-fixes: one O<N> per finding
  _Recommended fix_: <fix>. _Rejected_: <alt — grounds>.    ← per-finding fix (set-of-fixes shape only)

## Recommended fix    ← action-shaped & single-fix-multiple-deliverable shapes only
<recommended action upfront>
_Rejected alternatives_: <alt — grounds>; <alt — grounds>.

## Deliverable-shape recommendation
<1 ticket / N tickets / spec — per §Deliverable classification; for single-fix-multiple-deliverable, confirm the count via AskUserQuestion; finding→deliverable map for ≥2 deliverables>

## FOUNDATIONS alignment (product-behavior only)
<principle @ surface — aligns / tensions>

## Named assumptions
(1) <unknown> — assuming <assumption>; (2) ...
```

### Closing structure

Close every triage recommendation with:

1. **Deliverable-shape recommendation** — one spec / N tickets / mixed batch / in-place edits, per §Deliverable classification.
2. **Named assumptions** — remaining gaps in the format `(N) X — assuming Y`.

For a multi-deliverable triage (≥2 specs or ≥3 tickets), make the finding→deliverable mapping explicit in the recommendation (either inline `R3 — <summary> → SPEC-002`, or a `deliverable → findings` map) so the user can see which accepted finding lands where at approval time.

**`AskUserQuestion` vs named-assumptions at close-out:** if a remaining gap is material-deliverable-shape (changes deliverable type / scope / count), prefer `AskUserQuestion` to settle it before proceeding — even under auto mode or pre-authorization — because a shape mismatch requires rewriting rather than refining. (A coverage-widening *within* one already-approved deliverable that follows directly from an approach the user approved is not a shape change — surface it with a redirect offer, don't re-ask; see SKILL.md §User pre-authorization §Scope-change classes.) For content-level gaps within a stable shape, prefer `AskUserQuestion` outside auto mode; under auto mode or pre-authorization, default to named-assumptions plus the design-approval gate. But when the deliverable is written directly with no design-approval round-trip — a template-structured ticket/spec under pre-authorization or auto mode, where the only post-write gate is the Step 6 menu — that fallback gate does not fire, so a wrong named assumption would bake straight into the deliverable's content and acceptance criteria. In that case escalate any *load-bearing* content gap (one that materially changes the deliverable's behavior, not just its wording) to `AskUserQuestion` before the write, as if it were a shape gap. Symmetrically, an `AskUserQuestion` answer may *accept* the recommendation yet carry a rider concern in the same response (a new adjacent question, a usability doubt about the chosen option); if that rider is load-bearing for the deliverable, verify and resolve it — reproduce or Read/grep when the answer turns on a fact — before the pre-authorized write, not after.

## Deliverable classification

The full per-type rules behind SKILL.md Step 5's quick-triage table.

**Pre-authorized class vs. project change-intake routing.** When the user pre-authorizes a specific deliverable class ("create tickets", "write a spec"), check the project's documented change-intake / spec-trigger rules (e.g. `docs/ACTIVE-DOCS.md`'s "when a change needs a spec, ticket, ADR, or doc correction" list) *before* writing. If those rules route the change to a **different** class than the one pre-authorized — e.g. the user asked for a ticket but the change extends a compiler / generation-brief-schema / empty-state-rendering surface the project routes to a spec — treat it as a deliverable-class gap and surface it once via `AskUserQuestion` (recommended option first), per SKILL.md §User pre-authorization. Pre-authorization of one class does not silence a documented routing conflict; honoring the stated class is fine once the user reaffirms it against the routing rule.

- **Inline ops/setup task or mechanical-fix batch** — small tooling/ops work or a bounded mechanical-fix batch executed inline with no persisted design artifact (repo setup, local config, a short pre-approved sequence). Skip both the `docs/plans/` design doc and the Step 6 menu; the deliverable is the in-conversation design plus a post-execution summary. The HARD-GATE still requires explicit approval of the consolidated design before executing. The file edits/new files ARE persisted — only the design doc is elided.

- **Approved code fix / behavior change implemented inline** — a substantive fix to application source (e.g. `packages/*/src`) that emerges from a diagnostic triage or an approved design and is implemented in-session, optionally test-driven (red → green → lint/typecheck/test gate). Distinct from a *mechanical*-fix batch: it may introduce new types, functions, or tests, but it stays bounded and code-only (no spec/ticket/design-doc artifact). Skip both the `docs/plans/` design doc and the Step 6 menu; the deliverable is the code change plus a post-execution verification summary (what was reproduced, the gate results). The HARD-GATE still requires the design/triage to be approved — or pre-authorized — before editing; file edits ARE persisted, only the design doc is elided. Do not commit — leave for user review. When the work spans several cycles in one session, each new cycle re-enters at Step 1 and emits its own post-exploration confidence anchor (SKILL.md Step 6) — and if a follow-up cycle's deliverable is a spec/ticket/triage, that deliverable's Step 6 menu is mandatory; this inline-fix menu-skip does not carry forward to it.

- **New skill design** — the deliverable is the skill file at `.claude/skills/<name>/SKILL.md`; the skill file IS the design, so skip the `docs/plans/` doc. Adjust the Step 6 menu (omit "create a spec"). In plan mode, write the plan file first; write the full SKILL.md as the first post-approval implementation step.

- **Modify existing skill file(s)** — the edits ARE the design; skip the design doc. For a merge, include the new unified file, deletion of superseded directories, and updates to any cross-references.

- **Project documentation & root instruction/config files** — edits to (or creation of) `README.md`, `docs/*.md`, or root agent-instruction/config files (`CLAUDE.md`, `AGENTS.md`) where the doc IS the deliverable; the content IS the design. The Step 6 menu may be omitted when it completes inline in the same turn. (A root instruction file matches neither the `README.md` nor `docs/*.md` glob literally but is handled identically — inline-completion, menu-skip.)

- **New dev-tooling/CI/config file** — a created tooling/CI/config file (`.github/workflows/*`, lint/format/build config) where the file IS the deliverable; the file content IS the design. Created in place; the Step 6 menu may be omitted when it completes inline in the same turn. Verify the commands/actions the file invokes (build/test/lint scripts, action versions) before writing, as for any operator-introduced premise.

- **Amend `docs/FOUNDATIONS.md`** — a constitutional change. Per FOUNDATIONS §1, the constitution is amended deliberately and on purpose. Present the amendment explicitly as a constitution change, name the principle(s) affected and the downstream specs/features the change unblocks or invalidates, and require explicit user sign-off before writing. Skip the design doc — the amendment IS the deliverable. **When the amendment is inseparable from a code/implementation deliverable** (e.g. a triage ticket whose fix cannot ship without it), it remains a constitutional change: prefer authoring it as a named first deliverable with its wording signed off before the dependent ticket; if genuinely bundled into one ticket, fence it as a labeled sign-off-gated section showing the proposed wording — do not defer the wording to implementation time. **Post-sign-off:** when the user signs off a bundled amendment (e.g. via a "sign off the amendment" Step 6 menu option) but the dependent ticket is not yet implemented, record the approval on the ticket — clear the gate and mark the wording APPROVED — and apply the `docs/FOUNDATIONS.md` edit **atomically in the same revision** as its inseparable code change. Never land the constitutional edit standalone: that leaves the constitution declaring the new behavior while the code still does the old thing, a doc-ahead-of-code contradiction the §29 checklist itself would flag.

- **Port external skill** — deliverable is (a) the new skill file, (b) deletion of the reference source once verified, and (c) a transformations table enumerating per-element strip/replace/preserve decisions (one row per substitution site, not per source line). The approach focuses on identifying extraneous source-repo elements and their repo-appropriate replacements. A substitution not itemized in the table is out of scope until itemized. **Co-ported dependency files**: a ported skill often depends on co-ported files from the source repo (templates, READMEs, referenced docs the skill hard-depends on); itemize each as its own strip/replace/preserve row — they typically carry the same source-repo residue as the skill itself, and a dependency file not itemized is out of scope until itemized.

- **Replaces an existing artifact** — include (a) confirmed deletion of the old artifact, (b) a check for cross-references to it (in other skills, `README.md`, memory), (c) a note of the replacement in the deliverable.

- **System spec** — deliverable is the spec in `specs/`; the spec IS the design, so skip the design doc. **Section structure**: if a drafting-rules file or an existing spec in `specs/` establishes a convention, follow it; otherwise use the canonical section set that the `reassess-spec` and `spec-to-tickets` skills parse — Problem Statement, Approach, Deliverables, FOUNDATIONS Alignment, Verification, Out of Scope, Risks & Open Questions — and add a top-of-file note that the default structure was used. An existing spec under `archive/specs/` does **not** set the convention, even if it was successfully decomposed; the canonical section set is authoritative for new `specs/` files regardless of archived house style. This rule governs only the *tooling-parsed section set* — non-parsed preamble embellishments the established SPEC series uses (e.g. a `Primary authority docs:` / `Supporting authorities:` split, a `> Section style note:` blockquote) MAY mirror the series for consistency. To make that mirroring actionable rather than passive: while scanning `archive/specs/` for the highest ID (see Spec-ID assignment below), open the most-recent archived spec and copy its preamble embellishment style — the `Depends on:` enumeration form, the `Primary/Supporting authority` split, the section-style-note blockquote wording — so a new spec is not visibly inconsistent with its own tightly-uniform series. Do **not** adopt the `docs/requirements-version-1/*` requirements-doc house style (Purpose / Scope / Non-goals / … / Done Means) for a `specs/` file — those are requirements documents, not specs, and the spec tooling keys off the canonical section names. The Problem Statement should capture the motivation, evidence, and key "considered X, chose Y because Z" decisions from the brainstorm; a Brainstorm-Context / provenance header MAY additionally prefix the spec. For a lightweight spec that header is brief (original request, references, final confidence + assumptions). For a **codebase-grounded or roadmap/phase spec** the preamble may legitimately be substantial and SHOULD additionally carry (a) a *premise-verification* block citing `file:line` evidence that each operator-introduced design premise holds — the Step 4 "verify operator-introduced premises" rule, recorded inline so the downstream `reassess-spec` / `spec-to-tickets` skills inherit the verification (because they trust it, each `file:line` placed here must be operator-verified by Read/grep, not relayed from a sub-agent report whose line ranges may be off) — and (b) a *scope-decisions* block recording the confirmed scope-edge choices and their rationale. These two blocks are what make such a spec self-contained for decomposition; do not compress them away in the name of brevity. A roadmap/phase-driven spec MAY also carry a lightweight status header block (`Status:` plus, when relevant, `Phase`, `Depends on`, and `Governing authority`); a newly drafted spec's initial status is `DRAFT` (flipped to `COMPLETED` at archival). **Spec-ID assignment**: scan `specs/` **and `archive/specs/`** (plus any other spec home this repo uses) for the highest existing `SPEC-NNN` and claim the next integer (first spec is `SPEC-001`). Scanning `specs/` alone is wrong under the `specs/` → `tickets/` → `archive/` workflow, where a decomposed spec is moved to `archive/specs/` and `specs/` is routinely empty — a `specs/`-only scan would re-claim `SPEC-001` and collide with archived history. The Step 6 menu is mandatory (post-deliverable phase): offer reassess / decompose into tickets / implement / done.

- **Implementation tickets** — deliverable is the ticket file(s) in `tickets/`; the tickets ARE the design. If a ticket template exists, follow it; otherwise use a minimal Title / Context / Acceptance Criteria / Verification shape and note the template's absence. **Namespace**: tickets use per-initiative prefixes (e.g. `COMPILER-NNN`, `VALIDATE-NNN`), not one global sequence. Scan `tickets/` **and `archive/tickets/`** for existing prefixes and reuse an established one; when reusing a prefix, take the highest existing number under it across both locations — the `tickets/` → `archive/tickets/` workflow routinely empties `tickets/`, so a `tickets/`-only scan can re-claim `-001` and collide with archived history (mirroring the System-spec ID rule above). For a fresh namespace, derive a short uppercase initialism from the deliverable's primary subject and start at `<PREFIX>-001`; still glance at `archive/tickets/` to avoid a confusing collision with an archived prefix. **When one triage yields multiple tickets**, default to a **single family prefix with sequential numbers** (`<PREFIX>-001..-00N`) when the tickets share a surface or initiative; reserve **distinct prefixes** for tickets that are genuinely independent initiatives. Cite the namespace choice — and, for a multi-ticket triage, whether you used one family or distinct prefixes — in the deliverable lead so the user can redirect, collapse, or split before implementation. When referencing not-yet-created tickets earlier — in the triage/approach discussion, before the namespace scan — either defer naming them or mark the IDs explicitly provisional, so the post-scan prefix isn't a surprise. Each `file:line` citation written into a ticket's Assumption-Reassessment block must be operator-verified by Read/grep, not relayed from a sub-agent report whose line ranges may be off — a downstream implementer trusts it (mirroring the System-spec premise-verification rule above). When the template's Assumption-Reassessment block uses a numbered menu (items 4+ are an opt-in menu in `tickets/_TEMPLATE.md`), renumber the *included* items sequentially from 4 — do not carry the template's original menu numbers verbatim (a list like `1, 2, 3, 4, 8` is the malformed output the template explicitly forbids); verify the final list is gapless before the write. Ticket *creation* presents the Step 6 menu; ticket *update in place* is inline-completed — skip the menu and summarize the delta.

- **Triage producing ≥2 specs or ≥3 tickets** — additionally write `triage/YYYY-MM-DD-<topic>-triage.md` summarizing the source report, accepted items (with the full path to each spec/ticket + a one-line rationale), dismissed items (one-line reason each), and identified-but-unactioned follow-ups. Keep it under ~80 lines; reference deliverables by path rather than duplicating their content. This makes the brainstorm's decisions durable without re-running it. For a single spec or fewer than 3 tickets, skip this file by default — the deliverables are sufficient history. This summary shape assumes a source report; for a no-source-report diagnostic that crosses the threshold, use the companion shape in §Triage recommendation structure → No-source-report diagnostic case instead (reproduction/evidence + `O<N>` findings + decisions + finding→deliverable map).

- **Re-audit delta against an existing triage record** — when a re-audit changes the disposition of items in an existing `triage/YYYY-MM-DD-<topic>-triage.md` (a deferral now actioned, a "not ticketed" item now ticketed, a prior finding now refuted), update that record in place with a dated delta/addendum — even when the ≥2-spec/≥3-ticket new-companion threshold isn't met — so the durable decision trail stays truthful. Cite the new ticket/spec paths; don't silently leave a stale "deferred, not ticketed" note.

- **Triage analysis, all deliverables deferred** — when the brainstorm emits verdicts but produces no spec/ticket now (everything deferred) yet the user wants the verdicts persisted, write the decision record to `triage/YYYY-MM-DD-<topic>-triage.md` with the full triage (source, per-item verdicts + rationale, recommended shape, named assumptions). The file IS the deliverable, so it carries full verdict content (the ≤80-line companion budget does not apply). Step 6 offers: re-invoke `brainstorm` on this file to produce the deferred deliverables / adjust a named assumption / done.

- **Design doc (default)** — when none of the above fit, write `docs/plans/YYYY-MM-DD-<topic>-design.md`, where `<topic>` is a kebab-case short name. Consolidate all approved sections into a clean document with a "Brainstorm Context" header (original request, reference file, load-bearing decisions, final confidence + assumptions). In plan mode, write to the plan file instead.

- **Research brief** — a self-contained markdown report at `reports/<topic>-research-brief.md` targeted at an external researcher/LLM whose findings feed a later design. Inline all schemas, evidence, terminology, hard constraints, and explicit research questions, since the audience has no repo access. For product-behavior topics, include a non-negotiable-constraint section naming the FOUNDATIONS principles the topic engages and any rejection criteria future recommendations must satisfy. Optimize for completeness over brevity. Step 6: feed the brief to the researcher / wait for findings.

**Deliverable pivot.** If the user redirects the deliverable type mid-brainstorm ("actually, make this a spec"), reclassify and adjust the flow; don't re-confirm — they told you what they want. When the request pre-authorizes a choice among types ("ticket or spec, whichever fits"), the operator may select based on scope evidence from exploration without re-prompting — cite the scope basis in the deliverable lead.

When persisting ≥3 files, track one task per file so progress is visible. Do NOT commit any deliverable — leave it for user review.

## Design-presentation carve-outs

Detail for the Step 4 carve-outs. Each keeps the HARD-GATE — explicit approval of the consolidated artifact (or the per-tier unit) is required before any write.

- **Small-deliverable carve-out.** When the design comprises ≤4 distinct *decisions* (user-approveable choice points where the user could meaningfully redirect — atomic facts following from a parent decision count with the parent) AND confidence is ≥85%, present the design as a single structured artifact (a transformations table, a bullet list of decisions, a short enumerated summary) approved in one turn. Permitted by default under auto mode; outside auto mode, announce the consolidation. When the count is borderline, prefer consolidating — the gate that matters is "can the user review this in one turn".

- **Template-structured-deliverable carve-out.** When the deliverable has its own canonical template (a ticket template, the default spec sections, a skill file), the template provides the section navigation; present the full draft as one artifact, and one approval covers it. Covers the common case of a single template-structured deliverable with many small atomic line-items that don't decompose into the Step 4 section list. Announce the consolidation the way the small-deliverable carve-out does: present a delta summary (recommendation plus any scope confirmations) in the same turn as the write, so a same-turn write under §User pre-authorization (or the Non-plan-mode fast-track below) is never silent.

- **Multi-deliverable triage navigation.** For triage brainstorms producing ≥2 deliverables, apply the template-structured carve-out per deliverable — each spec/ticket is approved as a single consolidated draft. The triage approval covers all N deliverables together; no per-deliverable gate fires. (A companion `triage/` file is a companion to the set, not a member of it, and doesn't count toward the ≥2 threshold.)

- **Non-plan-mode fast-track.** When confidence is ≥85%, a single approach is approved with named assumptions covering remaining gaps, and the deliverable is template-structured, the consolidated-draft approval may be collapsed with the approach/triage approval: present a delta summary in the same turn as the file write. For triage brainstorms the triage-recommendation approval transitively covers the write in any mode; for non-triage approach-selection the same-turn collapse needs auto mode *or* §User pre-authorization (the consolidated draft is the user's first look at the full surface, but pre-authorization independently satisfies the gate via the recommendation presentation — auto mode is not separately required when the request pre-authorized the deliverable). The material-deliverable-shape check (Guardrails §User pre-authorization) still fires before a same-turn write — if the shape shifts, confirm with `AskUserQuestion` first.

- **Re-emergent interview during design.** If the user asks a discovery-style question or requests enumeration of open decisions during Step 3/4 ("ask me the questions that need settling to amend SPEC-X"), conduct a constrained interview applying the Step 2 rules (one question per message, prefer multiple-choice, name uncertainty, recommend when delegated). Label questions (A, B, C) so the design presentation can cite them. The settled decisions feed the design; the HARD-GATE still holds.

- **Mid-design term-clarification.** When the user asks what a term you introduced means, answer inline with one self-contained explanation (diagram, worked example, or short prose), then continue the section flow — not a labeled-question sequence.

- **Mid-design investigation re-entry.** When a user's mid-brainstorm question turns on a fact you have *not* yet established (e.g. "how is field X rendered at compile time?", "what does the compiler do with this reference?") — as opposed to the meaning of a term you introduced (term-clarification above) — loop back to targeted exploration *before* answering: announce the probe hypothesis-first, run it (sub-agent / grep / Read), then answer with `file:line` or captured-output evidence and re-pose the pending decision. Do not answer a factual question from assumption or memory — Step 2 rule 9 (verify load-bearing technical premises) and Step 4 (verify premises before persisting) apply equally to a mid-approach factual question. Distinct from **Re-emergent interview** (which poses questions *to* the user) and from **term-clarification** (which explains from knowledge you already have).

- **Mid-design scope-narrowing.** When the user requests reduced surface area after sections are approved (the architecture stays valid), recompute under the narrowed scope and announce the deltas before re-presenting; name the dropped elements with a concrete re-evaluation trigger each. Update the design doc in place unless the user asks for a fresh one. This is not a register shift and not a term clarification — don't mis-route it as either.

- **Plan-mode interaction.** Per-section approval is replaced by whole-plan approval via `ExitPlanMode`. Present key decisions inline (1-2 messages, grouping related sections) before writing the plan file, pausing after the first message for course corrections. The goal is conversation-level checkpoints, not per-section gates. When the approach is architecturally constrained (single viable option), the confidence announcement, approach proposal, and design presentation may fold into one message.
