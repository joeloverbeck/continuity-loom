# Brainstorm

Confidence-driven collaborative brainstorming. Interviews you until it understands what you **actually want** — not what you think you should want — then proposes approaches, builds a design, and lets you choose what happens next.

<HARD-GATE>
Do NOT write any code, scaffold any project, invoke any implementation skill, or take any implementation action until you have presented a design (or a triage recommendation) and the user has explicitly approved it. This applies to EVERY topic regardless of perceived simplicity.

Carve-out: when the user's initial request pre-authorizes a deliverable contingent on the brainstorm's verdict (e.g. "write a spec if changes are warranted"), presenting the design or triage recommendation IS the approval that satisfies this gate — see Guardrails §User pre-authorization. The Step 6 menu still fires afterward.
</HARD-GATE>

## Process flow

```
Read context (reference file + classify topic)
        |
        v
Confidence-driven interview loop (target: 95%)
        |
        v
Propose 2-3 approaches with tradeoffs
        |
        v
Present design section by section, get approval per section
        |
        v
[If product-behavior] Validate against docs/FOUNDATIONS.md
        |
        v
Persist the deliverable (design doc / spec / tickets / ...)
        |
        v
Next-steps menu (user chooses)
```

**In plan mode**: the design writes to the plan file instead of `docs/plans/`, and `ExitPlanMode` replaces the next-steps menu.

**Triage/analysis brainstorms** (evaluate a report or finding-set, or answer a diagnostic question, and produce work items): the interview loop and the propose-approaches step are replaced by a single triage recommendation — see Step 3 §Triage brainstorms and `references/triage-and-deliverables.md`.

## Step 1: Read context

1. **Reference material.**
   - **`reference_path` argument**: read the entire file. Extract its key claims, proposals, and open questions.
   - **Inline file references**: if the user names files inline in the request, read them with the same treatment. Read all of them.
   - **Inline-pasted prose**: if the user pastes reference content (external research output, a report excerpt, a finding-set) directly into the request body, that prose IS the reference — there's no file to read.
   - **Not reference material**: the user's own first-hand problem description (bug report, symptoms, repro steps, pasted error/program output as evidence of the defect) is the request itself, not externally-authored reference content. A file named only as the alignment authority (`docs/FOUNDATIONS.md`) is also not reference material — it's governed by sub-step 3. Inline-named **internal codebase files read for orientation** (`README.md`, CI/config, source files the user points at to ground the design) are likewise not reference material — they're covered by exploration (sub-steps 6–7) and get no reference summary; reserve the 2-3 sentence summary for inline-named **authored artifacts** (a report, research doc, external spec, or a skill/file being ported from another repo). The discriminator is authored-reference-content vs. repo-state-for-context, not merely whether a filename was mentioned — a port source physically resident in this tree (e.g. a staging copy under `.agents/skills/`) is still externally-authored content and warrants the summary.
   - **Oversized files**: if a reference exceeds the Read token limit, read in sequential offset/limit chunks until fully covered, and say so in the summary. Run a quick `wc -l <path>` first when you suspect a file is large.
   - **Path resolution**: if an inline filename doesn't resolve, glob the parent directory for the closest match (pluralization, hyphen drift, capitalization) before asking the user to re-specify; announce any correction inline. When the *named parent itself* yields no close match — the prefix is wrong, not just the leaf — broaden the search before asking: a repo-wide `find`/glob on the leaf name, and a `git status` scan for an untracked staging copy (a port source is commonly staged under a different tree, e.g. `.agents/skills/` when the request names `.claude/skills/`).

   **Emit a 2-3 sentence summary** of what the reference material contains as user-facing text before any further exploration — this is the user's audit-trail anchor for verifying you understood the reference. For a multi-item report, summarize its *shape* (item count, structure, headline position) rather than enumerating each item; per-item detail belongs in the Step 3 triage. When the reference makes claims about the codebase or product state, treat them as hypotheses to verify during exploration (sub-step 6), and flag prominently any that exploration contradicts. When there is no reference material at all (pure analysis from the request text), no summary fires — proceed to exploration; a single confirming line ("No reference material — the request itself is the spec; proceeding to exploration") is acceptable as the audit-trail anchor and is distinct from the multi-sentence reference summary.

2. **Topic classification.** Announce the classification on its own line as soon as it's determinable — often right after reading the primary reference's frontmatter/intro, and in all cases before the broader project-context exploration in sub-step 7. For a no-reference diagnostic (the request itself is the spec), classification is determinable from the request alone — announce it before exploration begins, not after. Don't let a large multi-file reference read push the announcement arbitrarily late; classify from the minimum evidence that settles the category, then keep reading. Pick the category the brainstorm falls into:
   - **product-behavior**: anything that changes how the app behaves at runtime or what it stores — story records and the active working set, the generation-time brief, deterministic prompt compilation and the universal prompt contract, validation rules, POV/knowledge/secrets handling, physical continuity, cast dossiers and voice, causal-pressure records, the accepted-segment archive, prompt inspection, OpenRouter/secrets, local-first data ownership, the mature-fiction envelope, LLM-assistance surfaces, or UI/workflow over those surfaces. These are governed by `docs/FOUNDATIONS.md`.
   - **dev-process / docs / tooling**: work that does not change product behavior — build/test scripts, dev ergonomics, repo tooling, skill design, or documentation edits that don't change product semantics.

   When genuinely ambiguous, give a one-to-two-sentence justification. Example: `Classification: product-behavior (the design adds a validation rule to the prompt-compilation path).` A **hardening / regression / docs spec that *armors* a product surface without changing runtime behavior** (e.g. golden-output tests, accepted-prose-exclusion regressions, an end-user guide) still classifies **product-behavior** — the governing surfaces (`docs/FOUNDATIONS.md`) decide the bucket, not whether behavior changes: `Classification: product-behavior (a regression/test/docs hardening spec that armors the prompt-compilation and accepted-prose surfaces — product-behavior governs because it engages FOUNDATIONS principles, even though no runtime behavior changes).`

3. **Read `docs/FOUNDATIONS.md` for product-behavior topics.** You'll need it in Steps 3 and 4 to validate approaches against the constitution and its §29 hard-fail checklist. For dev-process/tooling/docs topics the read is optional — read it only if the topic directly engages a FOUNDATIONS principle. The read is satisfied if FOUNDATIONS content is already in this session's context; re-read only after compaction or when you need a specific principle's wording. If a brainstorm reveals that the *right* design conflicts with FOUNDATIONS, the deliverable may need to be a FOUNDATIONS amendment first (per FOUNDATIONS §1) — flag this prominently and get explicit sign-off; never quietly design against the constitution.

4. **Calibrate initial confidence from the reference and request.** A thorough reference (rationale, decisions, structure) or a richly-specified request (problem analysis, evidence, root cause, code locations, a clear ask) can start confidence at 70-95%, reducing the interview to closing operational gaps. Don't ask motivational questions when the user has already demonstrated deep understanding.

5. **Interview-skip threshold.** If exploration and research bring confidence to 95%+ before the interview starts, skip Step 2 and go to Step 3; announce the confidence level. At 85-94% with a user-framed, fully-constrained directive (enumerated alternatives, or a clear "do X" with minimal open choices — triage brainstorms are the common case), also skip the discovery interview: go to Step 3, announce confidence, and carry remaining gaps as named assumptions for the user to correct during approach/triage selection. **When a remaining gap would *materially shape the deliverable*** (which approach, what gets stripped vs. kept, the deliverable's structure or class) rather than only a detail, prefer posing it via `AskUserQuestion` at approach-selection rather than carrying it as a silent assumption; reserve named-assumptions for detail-only gaps. (Related to §Guardrails §User pre-authorization, but broader: that rule gates *writing* on count/class/new-file changes; this one gates *approach-selection* and also covers internal-structure choices — which, when doc-settled, may proceed without asking even under pre-authorization.)

6. **External research.** If the topic needs domain knowledge beyond the codebase (algorithms, industry practice, competing architectures), launch research agents (via the `Agent` tool's `subagent_type`) BEFORE the interview and summarize findings for the user first. If codebase exploration already yields a clear root cause with concrete evidence, research may be skipped — note the skip. If research was supplied as a reference file or pasted prose, the sub-step 1 summary satisfies the announcement.

7. **Project context.** Explore relevant existing material before the interview — `docs/`, `specs/`, `tickets/`, `reports/`, `triage/`, prior design docs, and the prompt-template / story-record-schema files when the topic touches them. Launch exploration agents for broad surveys via the `Agent` tool's `subagent_type` — `Explore` when that agent type is available, else `general-purpose` (there is no bare `Explore` tool to call directly). For frustration-triggered brainstorms, also explore the history of attempted fixes — repeated remediation or accumulated workarounds signal that Step 3 should include radical options (reset, strip, rebuild) alongside incremental fixes. For triage brainstorms whose input is a report — **or a re-run conformance/diagnostic audit of a surface that may have been triaged before, or ANY brainstorm (a design/UX brainstorm included) whose deliverable touches a named product surface** — check whether that report, surface, or an earlier version was already triaged into an existing spec/ticket/`triage/` file (glob/grep `triage/` for the same spec or surface **before** forming findings — for a design brainstorm one `ls triage/` + grep for the surface is cheap and is what arms the reversal check below), and frame the current pass as a delta so already-accepted or already-rejected items aren't silently re-proposed; for a design brainstorm, treat any same-surface decision or same-surface ticket found there as a reversal-check and coordination input, and cite the overlap in the deliverable's provenance. **If the user has explicitly scoped out the triage folder** (e.g. "aligned with `docs/**` without the triage folder"): this mandatory discovery yields to that directive, but its acknowledgment does not — **emit one explicit line stating the `triage/` prior-triage discovery is skipped per the user's exclusion, then proceed.** That line is required output, not optional; don't silently skip the discovery, and don't override the user to run it anyway (the reference §No-source-report carve-out has the full rule). **The exclusion governs the *discovery/read* step only — it does NOT suppress the Step 6 companion-record *output*: a `triage/` companion is still written at the ≥2-spec/≥3-ticket threshold, because writing that file is an output, not a read of the excluded folder (see Step 6).** This up-front discovery is also a precondition for the §User-pre-authorization reversal-check exception: you can only surface a prior-decision reversal *before* the scope `AskUserQuestion` if you found the prior decision record first.

   **For a runtime/UI-defect diagnostic — or any brainstorm anchored on observed runtime/UI behavior, including a UX-improvement design brainstorm whose output is approaches or a spec rather than triage findings**, exploration is not only static — reproduce the defect (or the observed behavior) before forming findings *or designing approaches*. **Pick the branch by where the defect lives (the layer that must change), not where the symptom surfaces:** a UI-presented symptom whose root cause is a pure `@loom/core` function — a descriptor, the compiler, a validator, a normalizer — follows the deterministic-compiler / core-logic branch (including its static-root-cause skip), not the browser-repro branch.

   - **Reproduce by surface.** Reproduce the defect with the available browser-automation / CLI tooling (Puppeteer/Playwright MCP, `curl`, devtools console resources) **during this step, before forming findings** — capturing concrete evidence (console output, failing request + status code, repro steps) here corrects mis-stated symptom premises early rather than at Step 3 structuring.
   - **Production-project safety.** Prefer non-mutating probes (GET endpoints, read-only DB opens, code traces); before any reproduction step that could mutate user data — especially against a production project the user named — confirm it is read-only or use a disposable copy, and beware endpoints whose mere invocation has side effects (e.g. a project-open path that runs migrations) — **when the user *names* a production project as the repro path and opening it mutates, drive a disposable copy or surface the side-effect in one line before the first open — never open the named project silently** (the reference gives the full reconciliation, including the tie-breaker: when the user pins the exact UI flow, prefer surface-and-proceed over a disposable copy).
   - **Named repro tool/flow.** **When the user names a specific repro tool or flow** (e.g. Puppeteer/Playwright), exercise it at least once to capture a rendered/DOM confirmation of the symptom *before* substituting a cheaper proxy (`curl`, a read-only DB open, a code trace) for the rest of the investigation — a proxy can diverge from what the user actually sees; if you do substitute, say so in one line.
   - **Client-side branch.** For a **client-side-only** component bug (React state/render/DOM) with no server in play, the harness is instead a scratch unit/component test (Vitest/RTL) — **unless the app is already running and the symptom is DOM-/keystroke-/focus-native, in which case drive the real browser (higher-fidelity)** — and jsdom can confirm a root cause yet false-pass on a DOM-native symptom.
   - **Uncommitted-change confound.** For a UI/runtime-perception symptom, also scan the provided `git status` for uncommitted changes on the symptom's render/data path and reconcile them before forming findings — an in-flight edit may be the cause or a confound, and is first-class evidence for a "what I see is wrong" report.
   - **Regression framing.** **When the symptom is framed as a regression** ("recent work/PRs broke X"), the user's named suspect is a hypothesis to verify, not accept: run `git log`/`git blame -L <start>,<end>:<file>` on the symptom's render/data path to date the introducing commit — `git status` catches only uncommitted confounds, whereas the true cause may predate the suspected PR (a recent change may merely have made a long-standing defect *visible*), and a ticket that targets the wrong (most-recent) suspect is the failure this check prevents.
   - **Branch reference + core-logic static-skip.** See `references/triage-and-deliverables.md` §No-source-report diagnostic case for the reproduce-before-diagnosing rule, including the client-side branch, the CI/build/lint/test-command branch (run the failing CI step locally before diagnosing), and the deterministic-compiler / core-logic branch (reproduce via a `@loom/core` unit or golden test — **or skip the confirmatory repro when static reading already pins an unambiguous `file:line` root cause: for a pure deterministic `@loom/core` root cause (a descriptor/compiler/validator/normalizer) the schema + code are themselves the evidence, so the skip applies whether the user pasted verbatim wrong output or only *described* the symptom — verify the symptom premise against the schema/code, and note the skip explicitly in the post-exploration confidence anchor**, per the reference's static-root-cause-skip bullet; still write/update the golden or unit test when *implementing* the fix).

   **Roadmap selection** — when an explicit implementation-order, roadmap, or phased-build doc governs the work (e.g. `docs/requirements-version-1/IMPLEMENTATION-ORDER.md`), the recurring "which X next" brainstorm is roadmap selection, not finding-set triage: locate the next not-yet-implemented unit, verify its phase-gate and one-way dependencies against the codebase, and proceed to the single-approach + scope-confirmation path at high confidence — the order doc, not an interview, settles which item. When the request names a standing list of authority docs (the same near-identical doc list often recurs every phase), treat the phase's specific authority (from the order doc's cross-spec dependency map) as binding and consult the rest *by relevance* — note in the spec's provenance which named docs the selected phase does and does not engage, rather than forcing all of them into the spec. The open work is then verifying the gate and resolving the deliverable's scope-edge decisions (prefer `AskUserQuestion` per sub-step 5). The deliverable is typically a `specs/` spec; an unconditional "create the next spec" directive satisfies §User pre-authorization, so present the single approach (Step 3, fully-constrained case) and write the spec directly via the Step 4 Template-structured carve-out, using the `references/triage-and-deliverables.md` §Deliverable classification (System spec row) preamble convention (initial `Status: DRAFT`, plus `Phase` / `Depends on` / `Governing authority` where relevant). Claim the next `SPEC-NNN` by scanning **both `specs/` and `archive/specs/`** for the highest existing number (per the System spec row), since `specs/` is routinely empty in this workflow and a `specs/`-only scan would re-claim `SPEC-001` and collide with archived history. **While in the archive for the ID scan, open the most-recent archived spec and mirror its non-parsed preamble embellishments** (the `Depends on:` enumeration style, the `Primary authority docs:` / `Supporting authorities:` split, the `> Section style note:` blockquote) so the new spec stays consistent with its series — the canonical parsed section set still governs the body, and subject to the System-spec row's external-generator-scaffolding caveat (do **not** replicate evidence-ledger / `Target commit:` boilerplate even if the most-recent archived spec carries it). Close the roadmap-selection flow with a delta summary plus the mandatory Step 6 menu.

   **Hardening / polish / test / docs phases** — not every roadmap unit is a *feature*; the final phase (and sometimes intermediate ones) is often a **hardening / regression / polish / documentation basket** rather than new behavior. Recognize these by the order doc's gate language (golden-output stabilization, leakage/regression tests, "documented and hardened", user-facing docs) and its "Done Means" forbidding *new* product behavior. Scope each gate item as a **delta over the existing implementation, not a from-scratch build**: before drafting Deliverables, run an **EXISTS-vs-GAP survey** (an exploration agent inventorying what is already tested/documented/implemented vs. what each gate item still needs) so the spec armors behavior rather than rebuilding it. Apply YAGNI per item, make the spec's Out-of-Scope explicitly forbid new product behavior, and prefer `AskUserQuestion` for scope-edge choices that change the deliverable's shape (e.g. one spec vs. split, markdown guide vs. in-app UI, test-first vs. proactive build). Such a spec classifies **product-behavior** when it armors FOUNDATIONS-governed surfaces (per sub-step 2), and its alignment table reads `aligns` across the board — it adds regression armor *for* the hard rules rather than deferring any clearance, so there is typically no interim tension.

   **Phased-build interim FOUNDATIONS tensions** — a single phase often *cannot* clear every §29 hard-fail on its own because the order doc deliberately sequences a clearing feature to a later phase (e.g. the acceptance write lands one phase before the durable-change reminder that clears §29.8's "remind the user" hard-fail). This is an *interim tension settled by the order doc*, not a violation requiring sign-off (Step 3 / Guardrails): do **not** block for sign-off — instead (a) add a minimal in-phase mitigation so the hard-fail is never *silently* tripped (here, an ephemeral post-accept notice short of the deferred full feature), (b) tag the alignment-table row `tensions (interim) — cleared by Phase N`, and (c) cite that the order doc is itself FOUNDATIONS-aligned and defers the full clearance on purpose. Reserve true sign-off for a design whose *correct* form conflicts with FOUNDATIONS irrespective of phasing. The **inverse** also occurs: when the selected phase *clears* a hard-fail an earlier phase deliberately left open (e.g. a write-only archive that trips §29.8's "hide segments from review" until this phase's browser lands), mark that alignment-table row `aligns` and note in the rationale that it **closes the earlier interim trip** — it reads as a clearance, not a fresh alignment.

8. **Confidence checkpoint.** After all context reading, announce post-exploration confidence in one sentence on its own line (e.g. `Post-exploration confidence: ~90% — reference file is exhaustive, scope fully specified by the request`). This drives the Step 2 decision and is the audit-trail anchor before any triage recommendation. For a statically-pinned diagnostic where a confirmatory repro (a `@loom/core` unit/golden test, a browser/CLI reproduction) was skipped because static reading already pins an unambiguous `file:line` root cause — whether from the user's pasted verbatim output or, for a pure deterministic `@loom/core` root cause, from the schema + code with the symptom premise verified against them — say so explicitly in this anchor (e.g. `…root cause pinned statically from the descriptor schema; confirmatory golden/unit repro skipped`) — the full skip rule is in sub-step 7's deterministic-compiler branch; this is the pointer at its emission site.

## Step 2: Confidence-driven interview

The core of the skill. Reach **95% confidence** about what the user actually wants before proposing solutions.

### The protocol

After each user answer, display a confidence block:

```
Confidence: X%
Gaps: [specific list of remaining unknowns]
```

Keep asking until confidence reaches 95%, then announce: "I'm at 95% confidence. Moving to approaches." Under auto mode with a 0-2 question interview, an inline prose alternative is fine (`Confidence: ~90% — gaps listed as named assumptions below`) provided the gaps surface as named assumptions in the very next message.

### Interview rules

1. **One question per message.** Never batch questions in the discovery interview. (This does not restrict `AskUserQuestion`, whose multi-question schema is the sanctioned form for batching approach-selection or section-approval decisions outside the discovery loop.)
2. **Prefer multiple-choice** when the answer space is bounded; open-ended when it isn't. Use `AskUserQuestion` with labeled options when its schema is available; inline numbered options are an acceptable fallback (and are preferred in plan mode for speed).
3. **Probe motivations before solutions.** Ask "What problem does this solve?" and "What happens if we don't?" before "What do you want built?" The first request often describes a solution, not the problem.
4. **Challenge premature specificity.** If the user jumps to implementation details, ask why that specific approach matters. The constraint is often softer than stated.
5. **Detect "should want" vs "actually want".** Watch for buzzword-heavy descriptions, over-scoped requests, vague success criteria ("it should be good" — probe what "good" means), and solutions stated as requirements.
6. **Name your uncertainty specifically.** "I don't know whether this needs to handle edge case X" is useful; "I need more information" is not.
7. **Respect user expertise.** If the user gives a clear, well-reasoned answer, don't re-ask it in different words. Advance.
8. **Handle delegation gracefully.** If the user says "you decide," re-evaluate the options against FOUNDATIONS and project constraints, present your reasoned recommendation, and advance. Don't re-ask.
9. **Verify load-bearing technical premises.** When a user (or reference) premise is load-bearing for the recommendation — library status, version compatibility, feature presence, external-tool behavior — verify it (package metadata, docs, targeted codebase read) before adapting the recommendation to it. The 30-second check prevents recommending against a true premise the user momentarily mis-stated.
10. **Present empirical findings before asking questions.** When exploration produces concrete findings (data, root-cause evidence, confirmed/refuted hypotheses), present a concise summary — hypothesis, evidence, verdict per hypothesis — before the first interview question.

### Confidence scoring

| Range | Meaning | Action |
|-------|---------|--------|
| 0-30% | Don't understand the problem yet | Ask about the problem, not the solution |
| 30-60% | Understand the problem, unclear on constraints | Ask about constraints, success criteria, scope |
| 60-80% | Understand problem + constraints, unclear on priorities | Ask about tradeoffs, what matters most |
| 80-95% | Clear picture, a few edge cases/preferences unknown | Ask targeted questions about specific gaps |
| 95%+ | Ready to propose | Move to Step 3 |

Confidence rises from **both** user answers and research findings; when research closes a gap, note which gaps it closed vs which still need user input.

### Early exit

If the user says "just go" or "that's enough questions," respect it: announce current confidence, list remaining gaps as assumptions you'll make, and proceed. Mark those assumptions explicitly in the design so the user can correct them. Use the format `(N) X — assuming Y` (the unknown as X, the assumption after `— assuming`) so a correction is one line.

### Recovery/reset brainstorms

When triggered by frustration indicators ("huge mistake", "wrong approach", "start over", "strip everything"):

1. **Validate the diagnosis before accepting it.** Present what exploration revealed — both what's broken AND what's working — before agreeing with a scorched-earth instinct. The user may be catastrophizing, or the system may be more salvageable than they think.
2. **Focus confidence on "what's actually broken?"** before "what's the fix?" Is the problem structural (architecture) or tactical (configuration, content, missing inputs)? Is the frustration proportional to the evidence?
3. **Always include a radical option** in Step 3 (strip, delete, rebuild) — even if you recommend a less radical path. Validating the aggressive instinct as a real choice respects the user's judgment.

## Step 3: Propose approaches

**Path router.** The default is the classic approach path below: propose 2-3 approaches, then present the design (Step 4). If the input is instead a **triage / finding-set**, a **no-source-report diagnostic**, or a **roadmap-selected next unit**, use the corresponding special-case flow — see §Triage brainstorms below, `references/triage-and-deliverables.md` §No-source-report diagnostic case, and Step 1 sub-step 7 (roadmap selection) respectively — which replace this propose-approaches step.

Present **2-3 distinct approaches**, each with:

- **Name** — a short descriptive label
- **How it works** — 2-4 sentences
- **Tradeoffs** — what you gain, what you give up

Close with a recommendation: name the chosen option upfront, then justify in 1-3 sentences. The recommendation is one global decision, not a per-approach attribute.

**Sub-variant splitting.** If an option's "How it works" hides two internally distinct paths with materially different cost/risk, split them into sibling options or labeled sub-rows (`C1`, `C2`) rather than bundle — otherwise the user's selection forces an immediate clarification round.

**Comparing concrete artifacts.** When approaches differ in a concrete, comparable artifact (file layout, section set, transformations table, deliverable scope), use `AskUserQuestion` option `preview` fields to render the candidates side by side, so the user compares the actual artifacts rather than labels.

**For product-behavior topics**, annotate each approach with FOUNDATIONS alignment: `Foundations: <principle> @ <surface> (aligns / tensions — reason naming the surface)`. Surfaces include authoring time, prompt-compilation, validation gate, storage, UI. Naming the surface forces a second-order check that the proposal actually touches what the principle governs. If any approach tensions a FOUNDATIONS principle or trips a §29 hard-fail question, flag it explicitly and get sign-off — do not recommend a violating approach silently. **When the approaches are minor variants sharing the same FOUNDATIONS posture** (same principle, same surface, same stance), a single shared `Foundations:` line plus a per-approach *delta* for only the approach(es) whose stance differs (e.g. "C tensions §6.1 @ records-browser") is preferred over near-duplicate per-approach lines; keep one line per approach when they diverge across surfaces.

**If the problem space is fully constrained** (a reference provides a proven design, or requirements eliminate alternatives), present a single approach with a one-sentence rationale naming the constraint that narrows the space. Don't invent artificial alternatives. A roadmap-selected next unit (sub-step 7) is the common fully-constrained case: name the ordering doc plus the dependency chain as the narrowing constraint.

**For a Port external skill deliverable** (adapting an external skill into this repo), the approach IS a transformations table — per-element strip / replace / preserve decisions, one row per substitution site (not per source line). Present that table as the approach/design so the user can redirect any row; this inline guidance covers the common case, so consult `references/triage-and-deliverables.md` §Deliverable classification before presenting when you need the co-ported-dependency rule below. Most rows are passively-redirectable (a sensible strip/replace/preserve default the user can override in passing), but **promote two row classes to an explicit recommendation-led `AskUserQuestion`**: a *destructive* row (deleting the source — which the session typically did not author, so the harness's confirm-before-deleting rule applies) and a *genuine fork* (a row with two defensible outcomes, e.g. keep-vs-rename a domain label), per the Guardrails `AskUserQuestion` recommendation discipline. Per its rule, **a substitution not itemized in the table is out of scope until itemized** — itemize file drops (e.g. a source `examples/` dir), domain re-mappings, and convention removals as rows, not loose prose. **Co-ported dependency files**: a ported skill often depends on co-ported files from the source repo (templates, READMEs, referenced docs the skill hard-depends on); itemize each as its own strip/replace/preserve row — they typically carry the same source-repo residue as the skill itself. This shapes Step 3/Step 4; the Step 5 destination row points back to the same rule.

**If the user challenges a dismissal**, redo the analysis from first principles rather than defending prior reasoning; if it reverses, say where the prior reasoning was incomplete. **If the user clarifies a constraint**, redo the approach proposal under the refined constraint without restarting Step 2 — cite which assumption was sharpened. Brief confirmation suffices when the recommendation still holds; full re-presentation is reserved for clarifications that flip it.

**Wait for the user to choose** (or to ask you to refine/combine) before proceeding.

### Triage brainstorms

When the deliverable is a set of work items evaluated against a report, finding-set, or diagnostic question, replace the approaches step with a **triage recommendation**: which items warrant action, which are dismissed, and why. The user's approval of the triage serves the same gating purpose as choosing an approach. Render the Step 1 sub-step 8 post-exploration confidence sentence on its own line before the triage recommendation as the audit-trail anchor. When the interview was skipped, this is the *same* anchor as the Step 1 sub-step 8 announcement — emit it once, immediately before the recommendation, not twice (for a no-source-report diagnostic, where the `O<N>` findings precede the recommendation, the anchor precedes the **finding presentation** — it is the pre-findings audit-trail anchor, not literally adjacent to the recommendation). Verdict types, per-item structure, verdict-bucket grouping, the out-of-report and no-source-report cases, and the closing structure live in `references/triage-and-deliverables.md` §Triage recommendation structure — open that reference before structuring **any** triage-shaped close (source-report triage, no-source-report diagnostic, conformance audit, inventory audit), rather than working from this section's summaries. **For a no-source-report diagnostic** (the common "figure out why X" / "what's happening with Z" request, with no formal report to score), read that reference's §No-source-report diagnostic case before structuring output **and reuse its §Worked skeleton headings verbatim — `## Findings (out-of-report)`, `## Recommended fix` (action / single-fix-multi-deliverable closes only), `## Deliverable-shape recommendation`, `## FOUNDATIONS alignment`, `## Named assumptions` — rather than approximating them as `## Findings` or inline bold lines**: omit the verdict buckets, key every finding `O<N>`, and close per the diagnosis shape: a *single recommended action* → the Step 3 recommendation shape (action upfront, then rejected alternatives with their grounds); a *set of independent fixes* → a per-finding recommended fix + rejected alternatives **inlined under each `O<N>`** (not a separate `## Recommended fix` block — that block is for the action / single-fix-multi-deliverable closes only), closing with `## Deliverable-shape recommendation` that carries the finding→deliverable map (confirm 1↔N via `AskUserQuestion` when the fixes share a code surface, since bundling them into one ticket is then viable); a *single fix whose remediation spans multiple deliverables* (the common code-fix-plus-*separate*-docs case — a fix and its co-located test are one ticket, not a split) → the single-action shape for the close plus an `AskUserQuestion` to settle the deliverable count; a bare flat finding list only when the findings are pure observations with no per-finding fix to weigh. A finding may also resolve to **no deliverable** — an analysis answer to a question bundled with the fixes (the common "fix X and tell me whether Z" shape) that refutes a concern or confirms safe behavior; key it `O<N>` like any finding and carry it in the finding→deliverable map with an explicit "no ticket — analysis answer" outcome, rather than dropping it or forcing a fix (reference §No-source-report). Whichever close shape applies, a **product-behavior** diagnosis closes with a labeled FOUNDATIONS-alignment line (`<principle> @ <surface> — aligns/tensions`) and a named-assumptions line (omit the latter when there are genuinely no remaining gaps, or when every remaining gap is posed via `AskUserQuestion` in the same turn — asked gaps are settled, not assumed), per the reference worked skeleton — don't fold the FOUNDATIONS basis into prose only. **For a conformance audit against an authority doc/contract** (the common "analyze/verify whether X conforms to `<doc>`" or "audit all fields/endpoints of X against the docs" request), group the findings by *remediation route* (fix-code / amend-authority-doc / no-action-affirm) rather than the report-oriented verdict buckets, keep the `O<N>` keys, and include an affirming "what already conforms" coverage note (e.g. a per-field/per-surface table covering the conformant items too) so the audit reads as coverage, not just a defect list — per `references/triage-and-deliverables.md` §No-source-report (the conformance-audit bullet). **When the audit subjects are themselves a doc/file set with user-supplied dispositions** ("for each doc decide remove / merge / correct / create"), use the reference's adjacent inventory-audit bullet instead: the user's disposition vocabulary is the per-item verdict, `O<N>` keys cover cross-cutting findings, and the affirming keep-as-is rows still apply. **When a request is *both* a specific defect fix and a conformance / "used as intended" / "in alignment with `docs/**`" check (a common hybrid), keep the `O<N>` defect findings and the set-of-fixes close, but still add the affirming "what already conforms" coverage note** — the conformance dimension must be answered as structured coverage (e.g. a one-line statement of the sibling fields/sections that already satisfy the contract), not just inline prose, even when the deliverable's organizing axis is the bug fix (reference §No-source-report, hybrid-composition bullet). The inline summary here is not the whole picture: the reference carries two load-bearing structuring rules it does *not* duplicate — the single-root-cause-vs-multiple-`O<N>` granularity rule (one `O<N>` per genuinely distinct finding; nested bullets for evidence, not extra `O<N>`s) and the worked skeleton — which fixes the *output structure* (the `## Findings (out-of-report)` / `## Recommended fix` headings), not just the keying — so skipping the read risks both mis-keying findings and diverging from the prescribed close format even when the close shape looks settled. The inline prose here can carry you to correct *content*; the reference's worked skeleton is what keeps the *format* aligned, so open it for the no-source-report path rather than reconstructing the structure from this summary.

## Step 4: Present design

**Plan mode**: skip per-section gates. Present key decisions in 1-2 messages with conversation-level checkpoints, then write to the plan file. `ExitPlanMode` carries approval.

**Classification pivot check**: if the design reveals a deliverable type different from the Step 1 classification (e.g. a "docs tweak" now changes validation behavior), re-state the refined classification before presenting Section 1.

Once an approach is chosen, present the design **section by section**, scaling each to its complexity (a sentence for trivial parts, up to ~200 words for nuanced ones). Cover the relevant sections, renaming/combining to fit the topic's natural structure:

1. **Overview** — what this achieves, 1-2 sentences
2. **Architecture / structure** — how the pieces fit together
3. **Key decisions** — important choices and why
4. **Data flow / process** — how information moves
5. **Edge cases** — tricky scenarios and how they're handled
6. **Testing / verification** — how to confirm it works
7. **FOUNDATIONS alignment** — for product-behavior topics, a table with one row per principle the design actively engages or tensions (a focused design doc is typically 3-5 rows; a System spec touching many principles may run 6-9), plus any relevant §29 hard-fail question the design must clear: principle, stance (`aligns` / `tensions` / `tensions (interim) — cleared by Phase N` / `N/A`), and a one-line rationale citing the specific design mechanism and the surface it operates at. The count is descriptive, not a ceiling — never drop an engaged principle to fit a number. Omit for dev-process/tooling/docs topics unless the design touches a FOUNDATIONS principle.

**After each section**, ask "Does this section look right?" and wait for confirmation. If the user pushes back, revise before continuing. After 3 consecutive approvals with no pushback (2 under auto mode), batch remaining sections in groups of 2-3 — announce the batching. Keep a section standalone if it's substantially more complex or higher-risk.

**Carve-outs** (see `references/triage-and-deliverables.md` §Design-presentation carve-outs for detail):
- **Small deliverable** (≤4 decisions, ≥85% confidence): present as a single structured artifact approved in one turn. Announce the consolidation outside auto mode.
- **Template-structured deliverable** (a spec, ticket, or skill file with its own canonical sections): the template provides the navigation; present the full draft as one artifact, one approval covering it — and under §User pre-authorization that approval is the recommendation presentation, so the draft may be written directly in the same turn. Present a delta summary (recommendation plus any scope confirmations) in the same turn as the write, so the consolidation is announced even when no separate approval round-trips. Before such a direct write (no design-approval round-trip), escalate any *load-bearing* content gap to `AskUserQuestion` per `references/triage-and-deliverables.md` §Closing structure (the named-assumptions vs `AskUserQuestion` decision rule).
- **Multi-deliverable triage**: triage approval covers all N deliverables together; each is then approved as a single template-structured draft (no per-deliverable gate).

**Verify operator-introduced design premises before persisting them.** When the design names a specific field, file path, schema shape, API surface, or build/test command from the codebase, verify it exists (a quick Read or grep) before writing it into the deliverable — the 30-second check prevents a premise falsifying during implementation. This applies symmetrically to **absence** claims surfaced by exploration or sub-agent reports — before writing "create a new X" into a deliverable on the strength of a report that X does not exist, confirm the absence with a targeted grep/Read; sub-agent maps can be confidently worded yet incomplete. It applies equally to **presence/location** claims that get persisted into a durable verification block a downstream skill or implementer inherits and trusts — a spec's premise-verification block (consumed by `reassess-spec` / `spec-to-tickets`), a ticket's Assumption-Reassessment block, or a `triage/` record: when such a `file:line` citation is written, verify it yourself with a Read/grep rather than relaying it from a sub-agent report — sub-agent line ranges can be confidently worded yet off. For removals, enumerate ALL references with a broad grep before drafting the files-to-touch and acceptance criteria.

## Step 5: Persist the deliverable

After design approval, do NOT implement until the user picks an implementation option from the Step 6 menu — the design doc is typically the deliverable; implementation is a separate, user-chosen act. Exception: inline ops/setup and in-place edits are executed during the session (the HARD-GATE still requires approval first), and no separate design doc is written.

**Quick triage of deliverable shape → destination** (full per-type rules in `references/triage-and-deliverables.md` §Deliverable classification — open that section before the session's first spec or ticket write rather than working from this table alone; it carries load-bearing rules the table only summarizes):

| Deliverable shape | Destination |
|---|---|
| Inline ops/setup task or mechanical-fix batch | execute inline + summary; no file persisted |
| Approved code fix / behavior change implemented inline (optionally TDD) | execute inline (red→green→lint/typecheck/test gate) + verification summary; no design doc, menu skipped |
| New skill design | `.claude/skills/<name>/SKILL.md` |
| Modify existing skill file(s) | edit in place |
| Project documentation & root instruction/config files (`README.md`, `docs/*.md`, `CLAUDE.md`, `AGENTS.md`) | edit/create in place |
| New dev-tooling/CI/config file (`.github/workflows/*`, lint/format/build config) | create in place |
| Amend `docs/FOUNDATIONS.md` (constitutional change) | edit in place — flag as a constitution amendment, require explicit sign-off |
| Port external skill into repo | new skill file(s) (`SKILL.md` + any `references/`) + delete source + a transformations table itemizing per-element strip/replace/preserve (see the Step 3 Port-external-skill note + `references/triage-and-deliverables.md` §Deliverable classification) |
| Replaces an existing artifact | new file + delete old + update cross-references |
| System spec | `specs/SPEC-NNN-<slug>.md` (initial `Status: DRAFT`; preamble per `references/triage-and-deliverables.md` §Deliverable classification, System spec row) |
| Implementation tickets | `tickets/<PREFIX>-NNN.md` (reuse an existing prefix found across `tickets/`+`archive/tickets/`, else derive a new initialism @ `-001`; cite the namespace choice in the lead — see reference §Implementation tickets) |
| Triage → ≥2 specs / ≥3 tickets | deliverables + `triage/YYYY-MM-DD-<topic>-triage.md` |
| Triage analysis, all deliverables deferred | `triage/YYYY-MM-DD-<topic>-triage.md` (full verdict content) |
| Design doc (default) | `docs/plans/YYYY-MM-DD-<topic>-design.md` |
| Research brief for an external researcher | `reports/<topic>-research-brief.md` |

A default design doc consolidates all approved sections into a clean document with a "Brainstorm Context" header (original request, reference file if any, key decisions that shaped the design, final confidence and assumptions). Specs and tickets skip the design doc — the spec/ticket IS the design (a Brainstorm-Context / provenance preamble MAY still prefix a spec: brief for a lightweight spec, but substantial for a codebase-grounded or roadmap/phase spec — where it SHOULD carry a `file:line` premise-verification block and a scope-decisions block (and, when the brainstorm reproduced runtime/UI behavior live, an observed-live evidence block) so the spec is self-contained for decomposition; see `references/triage-and-deliverables.md` §Deliverable classification, System spec row). If a spec-drafting or ticket template, or an existing spec in `specs/`, establishes a convention, follow it; otherwise use the canonical `specs/` section set that the `reassess-spec` and `spec-to-tickets` skills parse — Problem Statement, Approach, Deliverables, FOUNDATIONS Alignment, Verification, Out of Scope, Risks & Open Questions — and note at the top of the file when you used this default. An existing spec under `archive/specs/` does **not** set the convention, even if it was successfully decomposed; the canonical section set is authoritative for new `specs/` files regardless of archived house style. Do **not** copy the `docs/requirements-version-1/*` house style (Purpose / Scope / Non-goals / … / Done Means) into a `specs/` file: those are requirements documents, not specs, and the spec tooling keys off the canonical section names.

After writing each deliverable file, confirm it actually landed on disk (a quick `ls`/Read) before the Step 6 menu — a write can fail to persist, or an external hook/watcher can remove the file between turns; don't report N deliverables on the strength of N successful Write calls alone. If working in a git worktree, resolve every output path against the **worktree root**, not the main repo root, and confirm the file landed there before the Step 6 menu.

When persisting ≥3 deliverable files, track one task per file so progress stays visible (per `references/triage-and-deliverables.md` §Deliverable classification).

Do NOT commit the file. Leave it for user review.

## Step 6: Next-steps menu

| Deliverable category | Step 6 action |
|---|---|
| Plan mode active (any category) | Call `ExitPlanMode` — skip the menu |
| Spec / ticket / triage / proposal — or any deliverable whose options describe NEXT phases (reassess / decompose / implement / done) | **Menu MANDATORY**, even when written this same turn — writing the file is pre-implementation, and skipping the menu strips the user's named next-phase options |
| Inline-completion (skill design; port external skill; small inline ops/setup; application source-code fixes / behavior changes implemented inline, optionally TDD; new dev-tooling/CI/config files; new or in-place-edited project-doc / root-instruction files — `README.md`, `docs/*.md`, `CLAUDE.md`, `AGENTS.md`; in-place edits to existing spec/ticket/docs where the edit IS the completed work) | Skip the menu; summarize what was done |

Categorize by the middle row first; the inline-completion skip fires only when the deliverable falls outside it. For a mandatory menu, present options like:

```
Design doc written to docs/plans/YYYY-MM-DD-<topic>-design.md

What would you like to do next?
1. Create a spec from this design (write to specs/)
2. Decompose into implementation tickets (write to tickets/)
3. Start implementing directly
4. Done for now — I'll review the design doc later
```

Adjust options to the deliverable: for a spec, offer reassess / decompose into tickets / implement / done; for tickets, offer begin-first-ticket / reassess / done. **When presenting tickets, state the namespace choice in the lead** — whether you reused an existing prefix or derived a new initialism (after scanning `tickets/` + `archive/tickets/`) and the starting number — so the user can redirect it before implementation (per `references/triage-and-deliverables.md` §Implementation tickets). Use `AskUserQuestion` when its schema is available; inline numbered options are an acceptable fallback — and are preferred when the closing turn must carry substantial presentation text in the same message (text preceding a tool call may not render) or under autonomous/auto operation. **When a menu option proposes addressing an adjacent finding whose deliverable shape isn't yet settled** (it could fold into an existing deliverable or become its own), phrase it shape-neutrally ("address the adjacent drift") rather than committing to a count/class ("as a third ticket") — the shape is decided in the follow-up cycle once its resolution direction is chosen, not prejudged in the option label. If the user picks an option that invokes another skill, invoke it — but a triage option that yields tickets with **no live `specs/` source** (e.g. the spec is archived/completed, or the tickets come straight from the triage) authors them directly via the ticket template (`references/triage-and-deliverables.md` §Implementation tickets), not by invoking `spec-to-tickets`, which requires a `specs/` file to read. If they pick "done," end.

**Companion `triage/` record at the menu boundary.** When a triage menu option produces ≥2 specs or ≥3 tickets — *including when the triage (Step 3) and the spec/ticket creation land in separate turns* — also write the `triage/YYYY-MM-DD-<topic>-triage.md` companion before closing (`references/triage-and-deliverables.md` §Deliverable classification). Crossing that threshold via the menu choice is the moment the rule fires; the count may only reach the threshold at execution time, so don't rely on the Step 5 table alone — the turn that runs the menu option may never revisit it. **Write the companion even when the user has explicitly excluded the triage folder** (e.g. "aligned with `docs/**` minus the triage folder"): that exclusion governs reads/*discovery* of `triage/` (Step 1 sub-step 7), not this output write — the companion record is still produced at threshold (`references/triage-and-deliverables.md` §No-source-report diagnostic case, prior-triage bullet). Suppressing it because the user "excluded the triage folder" is a misread that silently drops a mandatory durable artifact. **Reuse the triage presentation's `O<N>` keys verbatim in the companion** — re-deriving the finding list a turn later (after `AskUserQuestion`) is where keys silently drift; if a presented finding was folded or split during ticket creation, keep its original key and note the fold (e.g. "`O2` covers both sections") rather than renumbering downstream findings, since the user references the companion by the numbers they approved.

**If a follow-up brainstorm cycle opens for related scope** — whether a Step 6 option spawns it (a next ticket alongside the just-written one, a related spec the menu surfaced) or the user raises a new design topic conversationally after the prior deliverable completed, distinct from re-triaging the prior verdicts — re-enter at Step 1, not Step 2; emit the Step 1 sub-step 8 post-exploration confidence anchor for the new cycle — **including when the follow-up is a single "file a ticket" / "write the spec" selection from the prior cycle's closing options, which doesn't *feel* like a new cycle but is one** — so emit it even when exploration is trivial; the prior cycle's context typically elevates initial confidence so the interview-skip threshold applies. When the new cycle's deliverable is a spec/ticket/triage, its Step 6 menu is **mandatory** even though the spawning turn was an inline-fix completion that skipped *its* menu — the menu-skip does not carry forward to the follow-up deliverable. **When the follow-up edits a prior deliverable** (e.g. folding an adjacent finding into an existing ticket rather than writing a new one), re-confirm that file is still on disk (a Read) before the Edit — it may have been removed by an external hook/watcher between turns.

## Guardrails

- **YAGNI ruthlessly.** Strip optional extras from every design unless the user explicitly asked for them.
- **No scope inflation.** The design covers what was asked. Resist "while we're at it" improvements.
- **One question at a time** during the Step 2 discovery interview. (Does not restrict `AskUserQuestion` batching outside the loop.) When batching `AskUserQuestion` questions outside the loop, confirm the options are mutually independent — never batch a question whose options become incoherent under another question's answer (e.g. a "how many tickets?" question whose "3 tickets" option dies if a sibling question's answer drops a deliverable); split those into contingent rounds (see `references/triage-and-deliverables.md` §No-source-report diagnostic case on contingent rounds). Within a single multiSelect question, every option must be co-selectable with every other — express "none of the above" by making the question single-select or omitting the option entirely (the user can always reject or answer Other), never as a multiSelect option alongside selectable items.
- **`AskUserQuestion` answers that aren't clean selections.** When an answer's free-text note contradicts or qualifies the selected option, the **note governs** — reflect it back in one line and proceed on the note's intent, not the recorded label (e.g. a recorded "Exclude X" paired with a note "if it aligns with the docs we should do it" means *include* X). When an answer is itself a question or expresses genuine uncertainty ("I'm not sure — if X then Y"), answer it inline and **re-pose** the choice rather than acting on the tentative selection. The same inline-answer-then-re-pose handling covers a *rejected* `AskUserQuestion` whose rejection carries a clarify intent (the harness reports the user wants to clarify but returns no selection): ask what they want to clarify, answer it inline, then re-pose the (possibly revised) choice — a rejection-to-clarify and an uncertain answer get the same treatment. This extends the Step 2 "name uncertainty / recommend when delegated" rules to the out-of-loop `AskUserQuestion` path. **Lead shape/remedy questions with a recommendation.** When you pose an `AskUserQuestion` to settle a material deliverable-shape or remedy choice (per §User pre-authorization and `references/triage-and-deliverables.md` §No-source-report), make your recommended option the first option and mark it "(Recommended)" — the Step 3 recommendation discipline (name the chosen option upfront) applies to the `AskUserQuestion` path, not only to prose approach lists. Posing neutral options with no recommendation invites a reject-to-clarify round-trip when the user wanted your call. Omit the marker only when the choice is a genuine no-default preference.
- **No implementation before approval.** The HARD-GATE means what it says, subject only to the §User pre-authorization carve-out.
- **FOUNDATIONS is authoritative.** For product-behavior topics, if an approach violates a FOUNDATIONS principle or trips a §29 hard-fail, flag it immediately and get explicit sign-off. Never design against the constitution silently; a feature that genuinely needs to diverge requires a FOUNDATIONS amendment first. **Exception — phased-build interim tension:** when an order/roadmap doc deliberately sequences a hard-fail-clearing feature to a later phase, the current phase carries an *interim* tension that the order doc settles — do not block for sign-off; mitigate-minimally, tag the row `tensions (interim) — cleared by Phase N`, and proceed (see sub-step 7). **Coupled-amendment rule:** when an implementation deliverable's fix *requires* a FOUNDATIONS amendment that cannot ship separately, the amendment is still a deliberate constitutional change — either author it as a named first deliverable whose wording is presented for explicit sign-off before the dependent ticket, or, if it must be bundled into one ticket because the change is inseparable, fence it as a clearly-labeled sign-off-gated section with the proposed wording shown. Never defer the amendment wording to "review at implementation". **Once a bundled amendment is signed off** (e.g. the user picks a "sign off the amendment" menu option), record the approval on the dependent ticket — clear the gate, mark the wording APPROVED — and apply the constitutional edit in the *same revision* as its inseparable code change; never land the `docs/FOUNDATIONS.md` edit standalone, which would leave the constitution ahead of the code (see `references/triage-and-deliverables.md` §Deliverable classification, Amend `docs/FOUNDATIONS.md`).
- **Respect early exit.** If the user wants to skip ahead, let them — list your assumptions clearly.
- **Worktree discipline.** If working in a worktree, all file paths use the worktree root.
- **Blocker discovery during implementation.** If implementation reveals an architectural issue or a cost/scope re-estimate that blocks the deliverable or invalidates the approach ranking, present the blocker, options, and a recommendation. If the fix is small (< 50 lines, single file, no public-interface change), fix it inline and continue; otherwise create a ticket and either work around it with a documented constraint or implement the fix with user approval. **Pre-write case:** when the blocker is surfaced not during implementation but during a Step 4/5 verification *before* any deliverable is written, and its fix would add a deliverable of a new class or cross the deliverable count, the §User-pre-authorization shape-confirmation rule takes precedence over this fix-inline allowance — pose a one-shot `AskUserQuestion` confirming the expanded shape before writing, even under pre-authorization, however small the fix.
- **Auto mode** applies when a system reminder instructs continuous operation, the user issues an explicit "just go" directive, or a harness `auto_mode` flag is set. Under auto mode: clarifying questions are minimized (gaps become named assumptions), the Step 4 batching threshold drops from 3 to 2 consecutive approvals, and small/template-structured deliverables may be written in the same turn as approval. Approach-selection and section-approval gates still hold, and the HARD-GATE is never relaxed. Under pre-authorization, approach-selection is satisfied by a recommendation-led presentation in the same turn as the write (Step 4 Template-structured carve-out); for when a remaining gap must instead escalate to `AskUserQuestion` before a direct write, see `references/triage-and-deliverables.md` §Closing structure (the named-assumptions vs `AskUserQuestion` decision rule).
- **User pre-authorization.** When the request pre-authorizes a deliverable — whether contingent on the verdict ("create a spec if warranted", "implement if changes are needed") or unconditional ("create the next spec", common in roadmap selection per sub-step 7) — the HARD-GATE is satisfied by presenting the design or triage recommendation — no separate approval round-trip. Pre-authorization satisfies the HARD-GATE; it does NOT skip the Step 6 menu. The user can still redirect between presentation and write by issuing a counter-instruction. When a remaining gap would materially change the deliverable shape (deliverable type / scope / count — including a same-class count change the user did not pre-scope, e.g. bundling two related fixes into one ticket vs. splitting them; type changing class; or a new deliverable file becoming required), prefer `AskUserQuestion` to confirm the shape before writing even under pre-authorization — leading with your recommended option marked "(Recommended)", per the Guardrails `AskUserQuestion` bullet. This matches the reference's broader "scope / count" trigger (`references/triage-and-deliverables.md` §Closing structure); a count change need not introduce a new class to warrant the question. **Scope-change classes:** the question is required for changes to *which deliverables exist, their class, or their count*. It is **not** required for a coverage-widening *within a single already-approved deliverable* that is a direct, strictly-corrective consequence of an approach the user already approved — e.g. the user approves "generalize the detection" to fix field A, and the same generalization necessarily also corrects the identical defect in sibling field B. Surface that widening prominently with an explicit redirect offer ("say so if you'd rather keep it to A") — no fresh `AskUserQuestion`, since re-asking would over-ask against YAGNI and the user's already-made choice. This rule takes precedence over the §Guardrails "Blocker discovery" fix-inline allowance when a Step 4/5 pre-write verification surfaces the blocker (see that bullet). **Exception:** if the recommendation would reverse a decision the user previously committed to (a `triage/` decision record, a deliberately-rejected item, or an archived completed ticket/spec that recorded a deliberate stance on the same surface — the sub-step 7 prior-triage discovery globs `tickets/`/`specs/` too, so a stance found there is a reversal-check input, not just orientation context), surface the reversal prominently and pose a direction-confirmation `AskUserQuestion` before writing — pre-authorization does not extend to silently overturning a prior user decision.
